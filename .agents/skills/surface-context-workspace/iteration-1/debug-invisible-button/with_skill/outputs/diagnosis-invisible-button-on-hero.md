# Diagnosis: Invisible Button Text on Hero Section

## Problem

A Button placed inside a hero section with a `var(--Primary-FG-Bold)` purple background has invisible text -- the text color is the same as the background color.

## Root Cause

The Button component resolves its text and fill colors from CSS custom properties that assume a **default surface context** (light page background). Without the `<Surface>` component wrapping it, the token remapping that adapts colors for dark/colored backgrounds never activates.

Here is exactly what happens in the CSS cascade:

1. The hero `<div>` sets `background: var(--Primary-FG-Bold)` -- a bold purple.
2. The Button (assuming default `bold` variant) reads its background from:
   ```css
   --_btn-bg: var(--Button-backgroundColor-bold, var(--_btn-fg-bold));
   ```
   where `--_btn-fg-bold` resolves to `var(--Primary-FG-Bold)` -- the **same purple** as the hero.
3. The Button's text color reads from:
   ```css
   color: var(--Button-textColor-bold, var(--_btn-fg-bold-high));
   ```
   where `--_btn-fg-bold-high` resolves to `var(--Primary-FG-Bold-High)` -- which on a **default** surface is white. This part is actually fine for `bold` variant.

However, for `subtle` or `ghost` variants the situation is worse:
- **Ghost variant** text uses `var(--_btn-default-accent-a11y)` which resolves to `var(--Primary-Default-Accent-A11y)` -- a dark purple accent, invisible on a dark purple background.
- **Subtle variant** text uses the same `var(--_btn-default-accent-a11y)` -- also dark on dark.
- **Bold variant** fill uses `var(--Primary-FG-Bold)` -- identical to the hero background, making the button visually disappear (no contrast between button fill and section background), even though text may technically be white.

The core issue: **no `data-surface` attribute** is present in the DOM, so the `[data-surface="fg-bold"]` CSS blocks (generated by the brand CSS engine) never activate. Those blocks are responsible for remapping tokens like:

```css
[data-surface="fg-bold"] {
  --Primary-Default-High: #ffffff;            /* text flips to white */
  --Primary-Default-Accent-A11y: #c8b6ff;    /* accent lightens */
  --Primary-FG-Bold: #8a7de8;                /* bold fill uses brand inversion */
  --Primary-BG-Subtle: rgba(138,125,232,0.24); /* subtle becomes tinted */
}
```

Without these remappings, every token resolves to its "on a light page" value, producing dark-on-dark collisions.

## The Fix

Replace the raw `<div>` with the `<Surface>` component using `mode="fg-bold"`:

### Before (broken)

```tsx
<div style={{ background: 'var(--Primary-FG-Bold)' }}>
  <h1>Hero Title</h1>
  <Button variant="ghost">Get Started</Button>  {/* invisible text */}
</div>
```

### After (correct)

```tsx
import { Surface } from '@oneui/ui/components/Surface';

<Surface mode="fg-bold">
  <h1>Hero Title</h1>
  <Button variant="ghost">Get Started</Button>  {/* text adapts to white */}
</Surface>
```

The `<Surface>` component does two things:

1. **Sets `data-surface="fg-bold"`** on the wrapper element, which activates the `[data-surface="fg-bold"]` CSS blocks in `@layer brand`. These blocks remap all role tokens (Primary, Secondary, Neutral, etc.) so that text, accents, borders, and fills resolve to values that contrast against the bold surface.

2. **Applies the correct background color** via its own CSS module, so you do not need to manually set `background: var(--Primary-FG-Bold)`.

## What Each Button Variant Looks Like After the Fix

| Variant | On Default Surface | On `fg-bold` Surface (with `<Surface>`) |
|---------|-------------------|----------------------------------------|
| **bold** | Purple fill, white text | Tinted accent fill (brand inversion), white text |
| **subtle** | Light tinted fill, dark accent text | 24% alpha tint fill, white text |
| **ghost** | Transparent, dark accent text + subtle border | Transparent, white text + light border |

The "brand bold inversion" is particularly important for the `bold` variant: on a `fg-bold` surface, `--Primary-FG-Bold` is remapped to a contrasting tinted accent (shifted +8 f-steps on the OkLCH scale) so the button fill is visually distinct from the surface. Without `<Surface>`, both the button and the background resolve to the same `--Primary-FG-Bold` value.

## Key Principle

In OneUI, **never set a background color manually** on a container that holds interactive components. Always use `<Surface mode="...">` so the CSS token cascade adapts child components automatically. This mirrors how Figma variable modes work: placing a component instance on a bold surface auto-switches the variable mode, and `<Surface>` is the code equivalent of that behavior.

## Decision Tree for Choosing the Right Mode

| Scenario | Mode |
|----------|------|
| Hero banner, CTA section, dark/colored background | `mode="fg-bold"` |
| Light tinted card or section | `mode="bg-subtle"` |
| Elevated popover or floating card | `mode="elevated"` |
| Reset back to default inside a surface | `mode="default"` |
| Content on a standard white/light page | No `<Surface>` needed |
