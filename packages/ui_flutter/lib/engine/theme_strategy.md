# Flutter theme engine — recommended approach

The web stack resolves brands via **CSS injection** (`useBrandCSS`). React Native resolves via **`buildNativeTheme`** in TypeScript, producing a **`OneUINativeTheme`** object (`packages/shared/src/engine/buildNativeTheme.ts`).

Flutter should use the **same object shape**, transported as **JSON**.

## Best option: JSON from the TypeScript pipeline (recommended)

1. **Source of truth stays in TS** — no duplicate OkLCH / surface stacking logic in Dart.
2. **Parity with `@oneui/ui-native`** — mobile and Flutter consume the same serialised theme; QA can diff payloads.
3. **Evolves with Convex** — whatever `foundationToNativeTheme` accepts today remains the contract; extend deserialization in Dart when fields grow.

### Practical steps

1. **Convex** — deploy `nativeTheme:getNativeThemeSnapshot` (returns full `buildNativeTheme` JSON). Storybook calls it with `theme`, `density`, and `platform`.
2. **Flutter** — parse with [`NativeThemeSnapshot.tryParse`](native_theme_snapshot.dart); [`OneUiSurfaceBootstrap`](../theme/surface_scope.dart) consumes `themeConfig`, `rootParentStep`, and `darkMode` from the snapshot (nested surfaces still use the Dart `surface_engine` port with that `themeConfig`).
3. **`OneUiScope`** — platform / density + `getBrandOverviewData` for dimension and typography foundation pages; **also** `designSystem` + **`nativeTypography`** from `getNativeThemeSnapshot` when the app loads the snapshot.

Full Convex → Flutter matrix and **React / RN Button pipeline** (for architecting Flutter `OneUiButton`): [`docs/flutter-native-parity-and-button-pipeline.md`](../../../../docs/flutter-native-parity-and-button-pipeline.md).

### Why not a full Dart port first?

- The engine is **large** (`surfaceNew`, palette, typography builders, validation).
- Any Dart rewrite **will drift** unless you run dual golden tests forever.
- Your org already ships **`buildNativeTheme`** for RN; **reuse is strictly less risk**.

### Why not only “hardcoded fixtures”?

- Fine for **CI / offline dev** — ship a checked-in **`assets/fixtures/default_light_theme.json`** generated by the script above.
- **Insufficient alone** for studio parity — real apps need **live brand** payloads the same way `apps/mobile` uses Convex.

## Summary

**Use JSON (or MessagePack) from the existing TS `buildNativeTheme` output.** Port **Surface context** in Dart to **consume** `themeConfig` + `rootParentStep` + `darkMode` exactly like `packages/ui-native/src/theme/SurfaceContext.tsx`. Implement deserialization **incrementally** alongside the first components that need brand colours.
