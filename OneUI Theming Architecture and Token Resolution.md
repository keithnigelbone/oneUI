# **OneUI Theming Architecture and Token Resolution**

# 

# ** Purpose**

This reference documents how OneUI theming flows from Convex into
platform code, where token-related data is stored, and how web, React
Native, Android, and Flutter consume resolved theme values.

Convex stores foundation decisions and token sources. The final usable
tokens are generated by the OneUI engine from those records.

# **Executive Summary**

  -----------------------------------------------------------------------
  **System Area**  **Definition**
  ---------------- ------------------------------------------------------
  Final token      Not as the main runtime model. Convex stores brand
  storage          foundations, appearance mappings, preset selections,
                   and component overrides.

  Token generation The shared OneUI engine in \@oneui/shared/engine and
                   \@oneui/ui.

  Web consumption  Web injects generated CSS custom properties into
                   style#oneui-foundation-tokens, then components read
                   var(\--Token-Name).

  React Native     React Native builds a OneUINativeTheme object and
  consumption      components read values through
                   OneUINativeThemeProvider, Surface, and
                   useSurfaceTokens().

  Android and      Consumes a resolved JSON theme snapshot generated by
  Flutter          the same OneUI engine, instead of querying many Convex
  consumption      tables directly.

# **Main Runtime Query**

The primary query used by web and native samples is:

api.foundations.getBrandOverviewData({ brandId })

It collects the brand\'s foundation records and joins in related
records such as:

-   presetSelection

-   appearanceConfig

-   availablePlatforms

-   customFonts

-   decorations

-   motionConfig

-   elevationConfig

That payload is then passed to the engine.

# **Sub-Brand Data Model**

Product-level sub-themes are represented as **sub-brands** in this
codebase.

Sub-brands are not separate full brands. They are theme variants of a
parent brand. A sub-brand defines exactly 4 color-owned fields, then
inherits typography, spacing, icons, motion, elevation, materials, grid,
and component settings from the parent brand.

# **Trace And Inspection Checklist**

Use this sequence to trace a token value from Convex to runtime output:

1.  **Active brand**: read brands and the current brandId.

2.  **Active sub-brand**: read subBrandConfigs via
    > api.subBrandConfigs.getByParentBrand or listAll.

3.  **Available color scales**: check foundations\[type=\'color\'\] and
    > presetColorScales / brandPresetScaleSelections.

4.  **Appearance role mapping**: check appearanceConfigs.getByBrand.

5.  **Runtime aggregate payload**: check
    > foundations.getBrandOverviewData.

6.  **Sub-brand merge**: check applySubBrandAccentsToFoundation in
    > packages/shared/src/utils/subBrandFoundation.ts.

7.  **Generated CSS/native tokens**: check useBrandCSS for web or
    > buildNativeTheme for native.

8.  **Component-level overrides**: check
    > componentTokenOverrides.getAllBrandComponentData and
    > tokenOverrides.

# **Sub-Brand Examples: JioHome And JioMart**

JioHome and JioMart are examples of product-level sub-brand themes.

A sub-brand is not a second full brands record at runtime. It is a row
in subBrandConfigs under a parent brand, usually the base Jio or One UI
brand. That row overrides only the color appearance roles that make the
sub-brand visually distinct.

  ----------------------------------------------------------------------------

  theme**       location**             stores**          
  ------------- ---------------------- ----------------- ---------------------
  JioHome       subBrandConfigs where  primary,          Typography, spacing,
                name = \"JioHome\" and secondary,        shape, motion,
                parentBrandId = \<Jio  sparkle,          elevation, materials,
                parent brand id\>      background as     grid, neutral colors,
                                       scale names and   semantic colors,
                                       step numbers      component settings

  JioMart       subBrandConfigs where  primary,          Typography, spacing,
                name = \"JioMart\" and secondary,        shape, motion,
                parentBrandId = \<Jio  sparkle,          elevation, materials,
                parent brand id\>      background as     grid, neutral colors,
                                       scale names and   semantic colors,
                                       step numbers      component settings

# **Generated Color Tokens**

Final CSS token names and final hex values are generated by the engine.
Convex stores the source records used to generate those values.

The live source is split like this:

-   subBrandConfigs says which scale each sub-brand role uses.

-   foundations\[type=\"color\"\], presetColorScales, and
    > brandPresetScaleSelections define the actual available scales and
    > steps.

-   applySubBrandAccentsToFoundation patches the parent brand appearance
    > config with the sub-brand\'s four owned roles.

-   The OneUI engine resolves final tokens such as \--Primary-Bold,
    > \--Secondary-Subtle, \--Sparkle-TintedA11y, and \--Brand-Bg-Bold.

# **Sub-Brand Snapshot For Android And Flutter**

Android and Flutter consume the resolved output after the parent brand
and selected sub-brand are merged. The subBrandConfigs record alone is
not sufficient for runtime rendering.

Request shape:

GET
/api/themes/{parentBrandId}?subBrandId={subBrandConfigId}&theme=light&density=default&platform=mobile

Server-side resolution steps:

1.  Load the parent brand via getBrandOverviewData.

2.  Load the selected sub-brand from subBrandConfigs.

3.  Run applySubBrandAccentsToFoundation(parentData, subBrand).

4.  Run the OneUI engine.

5.  Return a resolved JSON theme snapshot to Android / Flutter.

# **Surface Context Model**

OneUI surfaces are not simple static colors. They resolve relative to
the parent surface step.

Surface modes:

  -----------------------------------------------------------------------
  **Surface     **Meaning**
  Mode**        
  ------------- ---------------------------------------------------------
  default       Page/root surface

  ghost         Same step as parent; visually transparent but still
                context-aware

  minimal       Parent step plus one step toward contrast

  subtle        Parent step plus two steps toward contrast

  moderate      Parent step plus three steps toward contrast

  bold          Brand base step or darker base step, with fallback if too
                close to parent

  elevated      Parent step shifted lighter, capped at the top of the
                scale
  -----------------------------------------------------------------------

The engine also resolves:

  -----------------------------------------------------------------------
  **Token Group**    **Purpose**
  ------------------ ----------------------------------------------------
  surfaces           Fill colors such as bold, subtle, elevated

  content            Text/icon/stroke colors against the current parent
                     surface

  onBoldContent      Text/icon colors against a bold fill

  onSubtleContent    Text/icon colors against a subtle fill

  states             Hover and pressed colors
  -----------------------------------------------------------------------

# **Android / Flutter Consumption Model**

The complex token-resolution behavior is implemented in the TypeScript
engine:

-   OkLCH 25-step scale generation

-   Preset/custom scale merging

-   Role mapping and neutral fallback

-   Parent-relative surface resolution

-   WCAG contrast-solving for content tokens

-   Hover/pressed state generation

-   Root-only fill tokens versus context-remapped tokens

-   Surface nesting behavior

-   Typography, motion, elevation, spacing, and shape resolution

Duplicating that logic independently in Kotlin and Dart increases drift
risk. A resolved JSON snapshot allows native implementations to map
stable, resolved values into platform theme primitives.

# **Theme Snapshot API**

A Theme Snapshot API exposes resolved theme data to non-web platforms:

GET /api/themes/{brandId}?theme=light&density=default&platform=mobile

The API is responsible for:

1.  Calling getBrandOverviewData or the internal equivalent.

2.  Running the shared OneUI engine.

3.  Returning a versioned JSON payload.

4.  Including brandHash for caching.

5.  Including both root tokens and enough themeConfig or surface
    > metadata if native clients need nested surface resolution offline.

# **Architecture Summary**

Convex is the system of record for brand foundation data. The OneUI
engine is the system of record for token resolution. Web receives CSS
variables. React Native receives a resolved theme object. Android and
Flutter receive the same resolved theme object as JSON, then expose it
through native theme primitives.
