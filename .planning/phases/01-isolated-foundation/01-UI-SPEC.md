# Phase 1 — UI Design Contract

> Visual and interaction contract for the **isolated Jio Experience Builder Lab** shell. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Brownfield note:** This phase builds INSIDE a fully mature Jio Design System. The Lab UI is composed **exclusively** from `@oneui/ui` components, Jio CSS Modules, and Jio design tokens — **no Tailwind, no shadcn, no external visual kits, zero literals** (CLAUDE.md, LAB-02). This spec therefore does not invent a design language; it prescribes **HOW Phase 1's Lab UI consumes the existing Jio DS**. All token names below are the canonical Jio tokens already shipped by `packages/tokens`.
>
> **shadcn gate:** NOT APPLICABLE. `components.json` absent, no Tailwind config; CLAUDE.md explicitly forbids Tailwind/external kits. Tool = none (Jio DS is the design system). Registry safety gate = not applicable.

## Design System

| Property | Value |
|----------|-------|
| Tool | none — **Jio Design System (`@oneui/ui`)** is the mandated system |
| Preset | not applicable (custom token engine: OkLCH color, f-step modular scale, relational typography) |
| Component library | **Base UI (headless) wrapped by `@oneui/ui`** — never fork primitives (CLAUDE.md) |
| Styling | **CSS Modules + `var(--Token-Name)` only.** No Tailwind, no utility CSS, no inline literal styles. Enforced by `pnpm check:literals` / `check:layers` / `validate:tokens` (scope extended to Lab packages + `(experience-lab)` route — LAB-02). |
| Icon library | `@oneui/ui` `Icon` / `IconButton` components (no raw SVG kits, no external icon packs) |
| Font | **JioType Var** via `var(--Typography-Font-Primary)` on every text element; `var(--Typography-Font-Code)` (JetBrains Mono) for the IR/JSON inspector |
| Surface mandate | Any non-`default` background MUST use `<Surface mode="…">`, never `<div style={{ background }}>` (CLAUDE.md Surface Context rule) |
| Shape system | Tokens derive from f-steps; buttons/chips/avatars = `--Shape-Pill`, inputs/selects = `--Shape-2` (f-4 ≈ 8px), cards/panels = `--Shape-3`–`--Shape-10` |

## Spacing Scale

The Jio f-step modular spacing scale is the source of truth. **Multiples-of-4 rule is satisfied by the Jio scale** (f0 = 16px base; all steps derive from the dimension cascade and adapt per platform/density). New Lab CSS must use these tokens — never raw px.

| Role (template) | Jio token | Mobile value | Usage in Lab UI |
|-----------------|-----------|--------------|-----------------|
| xs (icon gaps) | `var(--Spacing-1)` | 4px | Icon-to-label gap inside Select/Button, badge insets |
| sm (compact) | `var(--Spacing-2)` | 8px | Gap between chips, request-panel field internals |
| md (default) | `var(--Spacing-4)` | 16px | Default element spacing, panel field stacking, card body padding |
| lg (section) | `var(--Spacing-6)` | 24px | Request-panel section padding, run-inspector group padding |
| xl (layout gaps) | `var(--Spacing-8)` | 32px | Gap between panel sections, card outer padding |
| 2xl (major breaks) | `var(--Spacing-12)` | 48px | Empty-canvas hero block vertical rhythm |
| 3xl (page-level) | `var(--Spacing-16)` | 64px | Page-level margins where `--Spacing-Margin` is not used |
| page margins/gutters | `var(--Spacing-Margin)` / `var(--Spacing-Gutter)` | platform-adaptive | Lab page chrome, panel docking offsets |

**Exceptions:**
- **Touch targets:** interactive controls keep the Jio minimum (44px mobile / 24px desktop) — provided automatically by `@oneui/ui` components; do not shrink below this for canvas toolbar buttons.
- **tldraw canvas internals:** tldraw owns its own shape geometry (`Rectangle2d`) in canvas coordinate space; that geometry is NOT Lab CSS and is exempt from the token rule. **The HTML rendered inside each shape's `HTMLContainer` (card chrome, text, badges) is NOT exempt** — it uses Jio tokens like any other Lab UI.
- Negative spacing tokens (`--Spacing-Negative-*`) exist but are not needed for Phase 1.

## Typography

Phase 1 uses **4 distinct type sizes** (28 / 16 / 14 / 12px) across 5 active roles, plus 1 mono role for the inspector. Every text element pairs a size token with its line-height token and sets `font-family: var(--Typography-Font-Primary)` (CLAUDE.md MANDATORY rule).

**Font-weight policy (the 2-weight choice + fixed heading weights):**

- **Lab-authored content text (Body / Label / Code roles) declares exactly TWO active weights:**
  - **Default = Medium (500)** — `--Label-FontWeight-Medium` for all controls/labels/meta; `--Code` Medium for the inspector.
  - **De-emphasis = Low (400)** — `--Body-FontWeight-Low` for multi-line body / IR-summary / gap-report description text.
  - These are the only two weights the Lab chrome *chooses*. No 700/High weight is used anywhere in Phase 1 (no emphasized body copy in the shell).
- **Heading roles (Headline-L, Title-M) carry FIXED Jio DS role-token weights** — Headline-L = 900, Title-M = 800 — baked into the role tokens by the design system. They are **not additional free weight choices for the Lab**: each is used exactly once (page title; section heading) and its weight is non-negotiable system metadata that ships with the role token, not a value the Lab author picks.

| Role | Where used in Lab | Size token | Weight token | Line-height token |
|------|-------------------|------------|--------------|-------------------|
| Page title *(fixed DS weight)* | "Experience Lab" page heading | `--Headline-L-FontSize` (f4 ≈ 28px) | `--Headline-L-FontWeight` (900, fixed by DS) | `--Headline-L-LineHeight` |
| Section heading *(fixed DS weight)* | Request-panel / run-inspector group titles, card titles | `--Title-M-FontSize` (f0 ≈ 16px) | `--Title-M-FontWeight` (800, fixed by DS) | `--Title-M-LineHeight` |
| Body / IR-summary *(Low — content weight)* | Card body text, IR-summary outline, gap-report description | `--Body-S-FontSize` (f-1 ≈ 14px) | `--Body-FontWeight-Low` (400) | `--Body-S-LineHeight` |
| UI label / control text *(Medium — content weight)* | Button labels, Select labels, tab/nav labels, status pills | `--Label-S-FontSize` (f-1 ≈ 14px) | `--Label-FontWeight-Medium` (500) | `--Label-S-LineHeight` |
| Micro / meta *(Low — content weight)* | Run #N frame label, event-timeline timestamps, badge text | `--Label-XS-FontSize` (f-2 ≈ 12px) | `--Label-FontWeight-Low` (400) | `--Label-XS-LineHeight` |
| Code (inspector) *(Medium — content weight)* | IR / JSON inspector toggle content | `--Code-M-FontSize` (f0 ≈ 16px) | `--Code` Medium (500) | `--Code-M-LineHeight` |

**Weight summary:** 2 Lab-chosen content weights (Medium 500 default · Low 400 de-emphasis) + 2 fixed-by-DS heading weights (Headline 900, Title 800, each used once). No other weights appear.

**Size note:** Body-S and Label-S **share f-step f-1 (≈14px)** but are distinct roles — they differ in weight (Low 400 vs Medium 500), line-height token, and reading purpose (multi-line body vs single-line control text). The four distinct pixel sizes in play are **28 / 16 / 14 / 12** (Code-M reuses 16px, the section-heading f0 step).

**Rules (binding):**
- Never use legacy `--Typography-Size-*` / `--Typography-Weight-*` in new Lab code (CLAUDE.md).
- Button / chip / tab text uses the **Label** role, never Display/Headline.
- Display role is **not** used in Phase 1 (no hero/splash headline in the Lab shell).

## Color

Phase 1 obeys the **White Canvas Rule + Attention Pyramid** (design-composition skill): 80–90% `default` surface, brand accent reserved for the single primary action. Color is expressed through **surface tokens + role tokens** that remap per `[data-surface]` context — Lab CSS references generic role tokens, never hard-coded hex.

| Role | Jio token / mechanism | Usage |
|------|----------------------|-------|
| Dominant (≈80%) | `default` surface (`var(--Surface-Main)` / page surface; 2500 light, 100 dark) | Lab page background, tldraw canvas background, all card backgrounds, both docked panels |
| Secondary (≈15%) | `<Surface mode="bg-subtle">` / `var(--Surface-Fill-Subtle)` | Run-group "Run #N" frame fill, grouped sections inside request panel, IR/JSON inspector panel background |
| Accent (≈5%) | `--Primary-*` role (e.g. `--Primary-Bold` fill, `--Primary-Bold-High` text) via `<Button variant="bold" appearance="primary">` | **Reserved for the single "Run generation" CTA and active selection/focus states only** |
| Secondary accent | `--Secondary-*` role | Selected states of Select/Chip/Toggle in the request panel (per design-composition: Checkbox/Radio/Switch/Chip default to `secondary`) |
| Destructive | `--Negative-*` role (e.g. Badge/Button `appearance="negative"`) | **Not triggered by user destructive actions in P1** — used only for the **validation-failure / gap-report status indicator** on cards (problem signalling, not a delete action) |
| Status: warning/gap | `--Warning-*` role | Foundation-profile card **gap-report state** badge ("Foundation gap") and component-reference gap badge |
| Status: success/valid | `--Positive-*` role | Artifact-card "Valid IR" status pill after a successful mock run |
| Status: in-progress | `--Informative-*` or `--Neutral-*` role | Run-inspector "running…" status, neutral chrome (icon buttons, dividers) |

**Accent reserved for (explicit list — never "all interactive elements"):**
1. The single **Run generation** primary button (one per request panel — one focal point per viewport, design-composition rule).
2. Active/selected state of the currently-edited prompt card and its selected Select option.
3. Focus halo on interactive controls (`--Focus-Outline` + `--Surface-Halo-Gap`, per CLAUDE.md Focus Halo pattern).

Everything else (panel chrome, card bodies, nav, dividers, secondary controls) stays `default`/`neutral`. No `bg-bold`/`fg-bold` surfaces in the Phase 1 shell (no hero moment — the Lab is a working tool, not a marketing surface).

## Copywriting Contract

All copy is **clear, lowercase-sentence-case, action-first**, matching Jio's restrained voice. No emojis. Status language must distinguish a **gap report** (system has no profile/component — expected, recoverable) from an **error** (something broke).

| Element | Copy |
|---------|------|
| Primary CTA | **"Run generation"** (verb + noun; the single bold primary button in the request panel) |
| Page heading | **"Experience Lab"** (nav entry + page title; visibly distinct from the existing "Builder") |
| Empty canvas heading | **"Start a new experience"** |
| Empty canvas body | **"Add a prompt card to the canvas, then pick a brand, artifact type, and output profile in the request panel to run a generation."** (states the next step) |
| Prompt-card placeholder | **"Describe the experience you want to generate…"** |
| Request panel — section labels | **"Brand"**, **"Artifact type"**, **"Output profile"**, **"Prompt"** |
| Run-inspector empty state | **"No run yet. Select a prompt card and run a generation to see the live event timeline."** |
| Artifact card — valid status | **"Valid IR"** (Positive role pill) |
| Artifact card — IR summary header | **"Generated IR"** with toggle **"Show IR / Show JSON"** |
| Foundation gap-report (FND-03) | Heading: **"Foundation gap"** · Body: **"No Jio foundation profile is defined for {artifactType} → {outputProfile}. Generation stopped — no dimensions were invented. Pick a covered profile, or file this as a Jio system gap."** |
| Component gap-report (REG-03) | Heading: **"Component not in registry"** · Body: **"\"{componentId}\" isn't a Storybook-approved Jio component. It can't be emitted. Choose a registered component or report the gap."** |
| Validation-failure state (VAL-01) | Heading: **"Validation blocked this artifact"** · Body: **"{N} blocking violation(s) must be resolved before this artifact is ready. {firstViolationSummary}"** |
| Generic error state | Heading: **"Generation failed"** · Body: **"The run couldn't complete. Check the run inspector for details, then try again."** |
| Destructive confirmation | **Not applicable in Phase 1.** No destructive actions exist (canvas layout is ephemeral local state, D-08; runs/IR are append-only persisted records — no delete/clear-history UI ships in P1). If a "remove card" affordance is added, it is a local ephemeral action, not a destructive data operation, and needs no confirmation dialog. |

**Voice notes for the executor:** A gap is never phrased as a failure — it is the system **correctly refusing to invent** (this is the Core Value made visible). "Run #N" labels the run-group frame (D-07). Status pills use Label-XS typography + the matching semantic role.

## Interaction Contract

| Interaction | Contract |
|-------------|----------|
| Place prompt card | User adds a prompt card to the isolated tldraw canvas (CANVAS-02); card holds prompt text + acts as run origin (D-01) |
| Configure request | Docked **request panel** edits the *currently-selected* prompt card; config persists on the card object (D-01/D-04). Output-profile Select is **filtered by artifact type** — invalid pairings are unselectable at the UI, not merely rejected downstream (D-03) |
| Run | "Run generation" produces a **linked artifact card** inside a labeled "Run #N" tldraw frame (D-04/D-07); spatial lineage prompt → artifact is explicit |
| Live status | Docked **run-inspector panel** streams the `ExperienceBuilderEvent` timeline (ORCH-03); artifact card shows a lightweight status pill: running → **Valid IR** / **gap** |
| Success end-state | Artifact card renders a **structured IR summary** (artifact type, output profile, brand, layout-section / component-instance outline) + an **IR/JSON inspector toggle** — NOT a real-DOM preview (that is P3) |
| Gap path | On a foundation/component gap the run **short-circuits to the gap-report card and produces no artifact card** (FND-03/REG-03); the foundation-profile / component-reference card flips to its typed gap state |
| Focus | Every interactive control draws the 2-layer focus halo (`--Surface-Halo-Gap` inner gap + `--Focus-Outline` outer), never `--Surface-Main` directly (CLAUDE.md Focus Halo pattern) |
| Accessibility | WCAG AA mandatory: all icon-only controls carry `aria-label`; status conveyed by text + role, not color alone; 44px/24px touch targets preserved |
