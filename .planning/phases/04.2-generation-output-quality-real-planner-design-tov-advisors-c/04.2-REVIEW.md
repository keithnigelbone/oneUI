# Phase 04.2: Code Review Report

**Reviewed:** 2026-06-03
**Depth:** standard
**Files Reviewed:** 23
**Status:** issues_found

## Summary

Reviewed the phase 04.2 generation-quality slice: the IR layout primitive + `irToAst` compile path, advisor threading into the IR generator, the deterministic structural quality gate (`validateIRStructure` + `validateAst` merge in `workflow.ts`), the multi-agent transparency trace (`AgentTraceT` → Convex → card disclosure), and the per-agent canvas cursor overlay built on the sandboxed-iframe `postMessage` rect channel.

The postMessage channel is genuinely rect-only on the **send** side, but the **receive** side does not authenticate the message source/origin, so a hostile frame can inject arbitrary node-ids and rects into the overlay. More serious: the first-party preview iframe is rendered with `sandbox="allow-scripts allow-same-origin"` while serving **same-origin** content — that combination is, by the HTML spec, equivalent to no sandbox at all, and the content rendered is LLM-derived. The documented "old persisted flat IRs round-trip unchanged" guarantee is also broken by the Stack-remap retirement. The Convex additive `agentTrace` field is correctly modeled (`v.optional(v.any())`, append-only, no migration), and the quality-gate control flow in `workflow.ts` is sound.

## Critical Issues

### CR-01: First-party preview iframe is effectively un-sandboxed (`allow-scripts allow-same-origin` + same-origin content)

**File:** `apps/platform/src/app/(platform)/(experience-lab)/_canvas/shapes/ArtifactCardShape.tsx:220`
**Issue:**
```ts
const sandbox = previewSameOrigin ? 'allow-scripts allow-same-origin' : 'allow-scripts';
```
The first-party AST-render path sets `previewSameOrigin: true` (via `LabAstExecutor`) and the preview URL is `/internal/render-ast`, which is served from the **same origin** as the Lab app (see `RenderASTBranded.tsx` header comment — it deliberately re-creates Convex/Foundation context on the Lab origin). Per the HTML sandbox spec, an iframe whose content is same-origin with the embedder AND carries both `allow-scripts` and `allow-same-origin` can reach `window.frameElement`, read app cookies/`localStorage`/Convex client state in the shared origin, and even rewrite its own `sandbox` attribute — the sandbox provides effectively **no isolation**. The content rendered inside is an LLM-derived AST. While `ASTRenderer` blocks raw elements and forbidden props, it spreads arbitrary string props onto real components and renders LLM copy; treating that as fully trusted same-origin code contradicts the project's own hard constraint ("Sandboxed previews run isolated (separate origin + CSP); generated code cannot access auth/session/Convex tokens").

The code comments assert this is "safe because that iframe runs OUR ASTRenderer over a server-derived tree, no untrusted code" — but the tree is composed from untrusted model output, and same-origin + allow-scripts negates the CSP/sandbox guarantee the project mandates.

**Fix:** Serve `/internal/render-ast` from a distinct preview origin (separate subdomain/sandbox domain) so `allow-same-origin` does not grant access to the Lab origin's storage/cookies, OR drop `allow-same-origin` and load fonts/brand CSS via a mechanism that does not require it (inline the brand CSS into the iframe document; embed fonts as same-iframe `<style>`/data-URLs). If same-origin truly must remain, document the explicit threat model and remove the Lab origin's auth/Convex session from any context the `/internal/render-ast` route can reach.
```ts
// Preferred: a dedicated preview origin so allow-same-origin is scoped to it.
const sandbox = previewSameOrigin
  ? 'allow-scripts allow-same-origin' // served from preview.<host>, NOT the Lab origin
  : 'allow-scripts';
```

## Warnings

### WR-01: Cursor-overlay message listener does not verify `event.origin` / `event.source`

**File:** `apps/platform/src/app/(platform)/(experience-lab)/_canvas/shapes/ArtifactCardShape.tsx:528-537`
**Issue:** `AgentCursorOverlay` adds a `window` `message` listener and only validates the payload **shape** via `parseIrNodeRectsMessage`. It never checks that `event.source === <the preview iframe's contentWindow>` or that `event.origin` matches the expected preview origin. Any embedded frame, opened popup, or extension content script can `postMessage({ type: 'oneui:ir-node-rects', rects: [...] })` and inject arbitrary node-ids and bounding boxes into the overlay. Impact is cosmetic (the overlay only draws outlined boxes with `pointerEvents: 'none'`), so this is not a data/auth breach — but the file-level comment in `agentCursors.ts:15-21` claims the channel is hardened "defence in depth," which this gap contradicts.

**Fix:** Capture the iframe's `contentWindow` and reject messages whose `event.source` is not that window (and/or check `event.origin` against the known preview origin):
```ts
useEffect(() => {
  if (cursors.length === 0) return;
  const onMessage = (event: MessageEvent) => {
    if (event.source !== iframeRef.current?.contentWindow) return; // authenticate source
    const parsed = parseIrNodeRectsMessage(event.data);
    if (parsed) setRects(parsed.rects);
  };
  window.addEventListener('message', onMessage);
  return () => window.removeEventListener('message', onMessage);
}, [cursors.length]);
```

### WR-02: Legacy flat-IR round-trip is broken — `irToAst` emits unregistered `Stack`

**File:** `packages/experience-builder-core/src/ir/irToAst.ts:195-204`
**Issue:** When a section has no `layout` tree, `irToAst` falls back to emitting a node with `type: 'Stack'`. The comment (and the matching comment in `workflow.ts:76-79`) claims "Old persisted flat IRs without a layout tree still round-trip through irToAst's own legacy fallback." But this phase **retired** the `Stack`→`Container` remap, and `Stack` is **not** a member of the experience-builder registry (`getRegistryItem('Stack')` fails) nor of the validator's `STRUCTURAL_LAYOUT_TYPES` (`{'Container','Grid'}` only — `astValidator.ts:216`). Any flat IR therefore now fails `validateAst` with `unregistered-component` for every section, and `irToArtifactAst` synthesizes a bogus `@oneui/ui/components/Stack` import. The current generator always attaches `layout`, so the live path is unaffected — but the documented backward-compatibility guarantee is false, and re-running/iterating a previously-persisted flat IR (`parentIr` in the run route) will spuriously gap.

**Fix:** Map the legacy fallback section directly to `Container` (column stack) instead of `Stack`, matching the retired remap's behavior:
```ts
return {
  id: section.id,
  kind: 'component',
  type: 'Container',                 // was 'Stack' — Container is the registered primitive
  props: withIrNodeId({ direction: 'column', ...props }, section.id),
  children: section.instances.map(instanceToComponentNode),
};
```

### WR-03: `targetOrigin: '*'` on the rect reporter post

**File:** `apps/platform/src/app/internal/render-ast/RenderASTBranded.tsx:89-92`
**Issue:** The reporter posts to `window.parent.postMessage(payload, '*')`. The payload is rect-only (no secrets), so this is not a leak today, but `'*'` lets the message reach any parent origin if the preview document is ever embedded somewhere unexpected. Defence-in-depth and the file's own "separate-origin boundary" framing argue for pinning the target origin.

**Fix:** Pass the embedding canvas origin explicitly instead of `'*'` (e.g. inject the expected parent origin via a query param / `data-` attribute and use it as `targetOrigin`). At minimum, document why `'*'` is acceptable given the same-origin embedding in the first-party path.

### WR-04: `parseIrNodeRectsMessage` accepts unbounded `rects` arrays (overlay DoS)

**File:** `apps/platform/src/app/(platform)/(experience-lab)/_canvas/agentCursors.ts:65-91`
**Issue:** The parser iterates and accepts every well-shaped entry in `msg.rects` with no cap on array length and no numeric sanity bounds (NaN/Infinity pass `typeof === 'number'`). Combined with WR-01 (no source check), a hostile frame — or a pathological preview with thousands of `data-ir-node-id` nodes — can push an arbitrarily large rect set into React state, each rendered as an absolutely-positioned DOM node, degrading the canvas. `Number.isFinite` is also not enforced, so `Infinity`/`NaN` coordinates can reach the style object.

**Fix:** Cap the accepted count and require finite numbers:
```ts
const MAX_RECTS = 500;
for (const r of msg.rects) {
  if (rects.length >= MAX_RECTS) break;
  // ...
  if (typeof rect.id === 'string' &&
      Number.isFinite(rect.left) && Number.isFinite(rect.top) &&
      Number.isFinite(rect.width) && Number.isFinite(rect.height)) {
    rects.push({ ... });
  }
}
```

### WR-05: Scaled iframe content vs. unscaled overlay coordinates may misalign cursors

**File:** `apps/platform/src/app/internal/render-ast/RenderASTBranded.tsx:155-169` and `ArtifactCardShape.tsx:556-572`
**Issue:** `RenderASTBranded` renders the AST inside a child div scaled by `transform: scale(calc(100vw / 1440))`. `getBoundingClientRect()` returns **post-transform (scaled)** coordinates relative to the iframe viewport, which is what the overlay needs IF the overlay div is exactly coincident with the iframe's rendered box. The overlay is `position: absolute; inset: 0` over a `previewRegion`-styled container that wraps the iframe — but the iframe element itself uses `previewIframe` styling, and there is no guarantee the overlay container and the iframe share identical content-box dimensions (border/padding on `previewRegion` would offset every cursor). No tests assert pixel alignment. This is a correctness risk for the feature's whole point (cursors land on the wrong region), not a crash.

**Fix:** Position the overlay against the **iframe element's** measured rect rather than a sibling container's `inset: 0`, or assert in a test that `previewRegion` adds zero border/padding and the iframe fills it exactly. Document the coordinate-space contract (scaled, iframe-viewport-relative) at the overlay site.

### WR-06: `recordRunEvents` status union omits `'suspended'`, forcing a lossy remap

**File:** `apps/platform/src/app/api/experience-lab/run/route.ts:134-140`, `packages/convex/convex/experienceRuns.ts:59-81`
**Issue:** A suspended run is persisted as `status: 'gap'` (`route.ts:135`) because the Convex `recordRunEvents` status union only allows `running|artifact|gap|error`. A genuinely suspended campaign-plan run is then indistinguishable from a true foundation/component gap in the persisted record, so `listRunsByBrand` / any history UI cannot tell "awaiting human direction" apart from "refused." The event log still carries the distinction, but the queryable `status` is wrong.

**Fix:** Add `v.literal('suspended')` to the `recordRunEvents` status union (additive, no migration) and record the real terminal status, mapping only the wire `RunCompletedEvent` to `gap` where that frozen union demands it.

### WR-07: `uploadThumbnail` does not check the upload `Response.ok` before reading `storageId`

**File:** `apps/platform/src/app/api/experience-lab/run/route.ts:83-91`
**Issue:** After `fetch(uploadUrl, { method: 'POST', ... })` the code immediately does `await uploadRes.json()` and casts to `{ storageId }` without checking `uploadRes.ok`. A failed upload (4xx/5xx) may return an empty/HTML body, throwing an opaque JSON parse error or yielding `storageId === undefined`, which then flows into `persistArtifact` as `thumbnail`. The caller's try/catch makes it non-fatal, but the failure mode is silent and the error message is misleading (looks like a parse bug, not an upload failure).

**Fix:**
```ts
const uploadRes = await fetch(uploadUrl, { method: 'POST', headers: { 'Content-Type': 'image/png' }, body: png as unknown as BodyInit });
if (!uploadRes.ok) throw new Error(`thumbnail upload failed: ${uploadRes.status}`);
const { storageId } = (await uploadRes.json()) as { storageId: Id<'_storage'> };
if (!storageId) throw new Error('thumbnail upload returned no storageId');
```

## Info

### IN-01: `previewSameOrigin` boolean is duplicated across three layers without a shared constant

**File:** `runStream.ts:61-62`, `ArtifactCardShape.tsx:98-100`, `route.ts:505`
**Issue:** The trust-scope flag is threaded as a bare boolean through the result frame, the shape props, and the iframe sandbox string. The sandbox string literals `'allow-scripts'` / `'allow-scripts allow-same-origin'` are inline. Given CR-01, the sandbox policy should live in one named place so the trust decision is auditable in a single location.
**Fix:** Extract a `previewSandbox(sameOrigin: boolean): string` helper (one source of truth) and reference it from the card.

### IN-02: `mockGeneration.ts` still exported as a public API alongside the real `generateIR`

**File:** `packages/experience-builder-agents/src/index.ts:34-39`
**Issue:** `mockGenerate` / `MockGenerateInput` / `MockGenerateResult` remain exported as a "legacy P1 path." With the real `generateIR` now driving the workflow, the mock is dead public surface that can drift or be imported by mistake. Not a defect, but it widens the package's API and the maintenance surface.
**Fix:** If the mock is retained only for tests, move it under a test-only entry point or mark it `@deprecated` so consumers don't reach for it.

### IN-03: `validateIRStructure`'s `hasRealStructure` treats any nested layout node (even an empty `spacer`) as "real structure"

**File:** `packages/experience-builder-validation/src/astValidator.ts:728-738`
**Issue:** Called with `depth=1`, the first nested layout child satisfies `depth + 1 >= 2` and returns `true` regardless of whether that child carries real content (e.g. a `stack` whose only child is an empty `spacer`). The `flat-layout` gate can be cleared by structurally-nested-but-empty layouts. The current generator emits a content-bearing inner `row`, so this is not hit today, but the gate is weaker than its docstring ("a genuine product UI, real nesting") implies.
**Fix:** Require the nested structure to ultimately contain at least one registry component instance (a leaf), not just another layout node, before treating it as real structure.
