# Phase 04.2 Plan 03: Advisor Threading into IR Assembly Summary

Made the Planner, Design advisor, and ToV advisor stop being dead weight (root cause #2): the IR Generator now consumes the planner's per-section intent + message hierarchy, the Design advisor's chosen components + surfaceMode, and the ToV advisor's real headline/body/cta — true assembler-last (D-05). The single `{id,name}` discard at `workflow.ts:408-409` is gone. The GAP-01 placeholder backfill is demoted to a flagged last-resort safety net with `{instanceId, propName, isContent}` provenance persisted to run metadata (root cause #3 / D-08), and the redundant `LAYOUT_TYPE_REMAP` bridge is retired because the generator now emits explicit layout primitives that compile to a real Jio `Container` directly.

## What Was Built

### Task 1 — `irGenerator.ts`: consume advisor specs + emit layout primitives with real copy
- `GenerateIRInput` extended: `sections[].intent`, `messageHierarchy: string[]`, `designSpecs: SectionDesignSpecT[]` (imported from `adapters/designAdapter`), `copySpecs: SectionCopySpecT[]` (imported from `adapters/voiceAdapter`).
- `buildSectionPrompt` now injects, per section: the section intent, the **message hierarchy** (numbered most→least prominent — D-07), the Design advisor's chosen `components` + `surfaceMode` with an instruction to **arrange them inside layout primitives** (stack/grid/row, never a flat list), and the ToV advisor's `headline`/`body`/`cta` as the exact content copy (no placeholders). When a spec is absent the prior generic guidance is the fallback.
- Candidate assembly emits an explicit `stack` `JioIRLayoutNode` per section (`direction:'column'`, `gap:'4'`, `surfaceMode` from the design spec when present, `children` = the section instances). The IR Generator remains the single committing step — no mid-generation advisor re-call. All structured-output schemas stay Anthropic-safe; no `ai`/`@ai-sdk` import added; all model calls route through `callModel`.

### Task 2 — backfill provenance (`irGenerator.ts`) + workflow threading (`workflow.ts` / `runContext.ts`)
- `backfillRequiredProps` is now **exported** and **returns** a `BackfillRecord[]` (`{instanceId, propName, isContent}`), one record per fill; `isContent = CONTENT_PROP_NAMES.has(propName)`. A model-supplied prop is never overwritten and produces no record; only REQUIRED structural props are touched. `generateIR` collects these into `GenerateIRResult.meta.backfilled`.
- `workflow.ts` `generateStep` replaces the `{id,name}` discard with the full advisor read: `sections` (with `intent`), `messageHierarchy`, `designSpecs: ctx.designSpec.designSpecs`, `copySpecs: ctx.copySpec.copySpecs` — each presence-guarded, all context READs (no advisor re-call). `ctx.backfilled` (new optional `RunContext` field) persists `meta.backfilled` for Plan 04's quality gate.
- `LAYOUT_TYPE_REMAP` Stack→Container remap + `remapType` retired from `workflow.ts`; `irToArtifactAst` still synthesizes one Jio import per distinct compiled type (Container/Grid + instances) so the validator import allowlist passes.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] mockGenerate + compiler.test.ts fixtures broke when the Stack→Container remap was retired**
- **Found during:** Task 2 (after retiring `LAYOUT_TYPE_REMAP`).
- **Issue:** `mockGenerate` (legacy mock) and `compiler.test.ts`'s `COMPLIANT_IR`/`NONCOMPLIANT_IR` fixtures produce FLAT IRs (no `layout` tree), which `irToAst`'s frozen fallback compiles to a `Stack` wrapper. With the remap gone, `validateAst` (reused by `compile()` via `irToArtifactAst`) rejected 'Stack' as an unregistered type — `mockGeneration.test.ts` (2) + `compiler.test.ts` leg-2a/leg-3 failed.
- **Fix:** `mockGenerate` and both compiler-test fixtures now emit the same explicit `stack` layout primitive the generator emits, so they compile to a real `Container`. The `compiler.test.ts` leg-3 snapshot was re-blessed (now `Container` + `direction="column"` + `gap="4"`, no `Stack`).
- **Files modified:** `mockGeneration.ts`, `compiler.test.ts`, `__snapshots__/compiler.test.ts.snap`.
- **Commit:** 84817847

**2. [Rule 3 - Blocking] Stale RED `@ts-expect-error` directives in irGenerator.test.ts**
- **Found during:** Task 1 (typecheck after extending `GenerateIRInput`).
- **Issue:** The 04.2-01 RED scaffold pinned the to-be-added advisor-spec fields with four `@ts-expect-error` directives; once the fields became valid (GREEN) the directives were unused (TS2578).
- **Fix:** Removed the four now-stale directives. This is the intended RED→GREEN transition, not a scope change.
