# Phase 04.2: Generation output quality - Research

**Researched:** 2026-06-03
**Domain:** Generation-engine modification — compositional IR layout primitives, advisor threading into IR assembly, real-copy grounding, quality gating (structural + multimodal), multi-agent transparency layer.
**Confidence:** HIGH (all findings traced to read source code with line ranges; no new external packages)

## Summary

This is the first phase to **modify** the Jio Experience generation engine (read-only through 04.1). Three engine-side root causes are confirmed by reading the code:

1. **Flat layout** — `JioIRSection.instances` is `z.array(JioIRComponentInstance)` (schema.ts:140); the compiler `irToAst` wraps each section in a hardcoded `Stack` then `Container` (irToAst.ts:95-114). There is no nesting/columns/grid vocabulary in the IR. **Key discovery:** `Container` and `Grid` already exist as real Jio components with exactly the props D-02 needs — `Container` supports `direction`/`wrap`/`justify`/`align`/`columns`/`gap`/`padding` (all resolving to `var(--Spacing-*)` tokens, Container.shared.ts:148-152, 290-307), and `Grid.columns` accepts a responsive `Partial<Record<Breakpoint,number>>` where `Breakpoint = 'S'|'M'|'L'` maps 1:1 to the `data-Breakpoint` web breakpoints (Grid/responsive.ts:4-20). This makes the D-04 responsive-per-profile requirement a thin mapping, not new infrastructure. **But** `Stack`, `Row/Cluster`, and `Spacer` do **not** exist as components, and `Container`/`Grid` "fall through to EMPTY generated-prop metadata" in the registry (queryRegistry.ts:54-55) — so the validator currently cannot enforce or even allow their layout props.

2. **Advisors are dead weight** — `runPlanner` returns `{sections, messageHierarchy, primaryCTA, screenCount}` (plannerAgent.ts:44-70, each section carries an `intent`); `runDesignAdapter` returns per-section `{surfaceMode, components[]}` (designAdapter.ts:53-65); `runVoiceAdapter` returns per-section `{headline, body, cta, toneScore}` (voiceAdapter.ts:59-72). The workflow writes all three to `ctx.plan`/`ctx.designSpec`/`ctx.copySpec` (workflow.ts:289-380, runContext.ts:57-59) — **but `generateStep` passes only `ctx.plan.sections.map(s => ({id, name}))` to `generateIR`, discarding `intent`, `designSpec`, and `copySpec` entirely** (workflow.ts:408-409). `GenerateIRInput` only declares `sections?: Array<{id; name}>` (irGenerator.ts:104-118). The IR Generator then re-asks a generic per-section model call (buildSectionPrompt, irGenerator.ts:638-660), so copy is thin and structure is undirected.

3. **Placeholder backfill surfaces** — `backfillRequiredProps` (irGenerator.ts:549-601) fills missing required props with `"<Section> item"`, `https://placehold.co/600x400`, or `var(--Typography-Font-Primary)` for icon props. Because copy is thin, these placeholders render as visible content.

**Primary recommendation:** Implement layout primitives as a **distinct closed `kind`-discriminated structural node** that is `union`'d into the slot/section vocabulary (NOT a registry instance, NOT reusing the empty-meta `Container`/`Grid` registry entries for validation). Compile primitives to the real `Container`/`Grid` Jio components in `irToAst`. Thread the full advisor specs into `GenerateIRInput`. Demote backfill to structural-required-only with a run-metadata flag. Reuse the existing evaluate→repair→cap loop unchanged, adding deterministic structural quality checks as a new BLOCKING validator category (the primary gate) and keeping the multimodal threshold as a tunable secondary gate. Persist agent-step traces additively on the existing `experienceArtifactVersions`/`experienceRuns` tables; enrich the existing `RunTurn` step ticker labels. **Defer the canvas agent-cursor mapping (D-06c) to its own wave/plan** — it is the only piece needing net-new agent→IR-node→canvas-region infrastructure.

## User Constraints (from CONTEXT.md)

### Locked Decisions

**Compositional layout (root cause #1)**
- **D-01:** Express layout via **first-class IR layout primitives** — explicit layout nodes in the canonical IR schema that compile to Jio layout CSS. NOT layout-via-existing-registry-components-in-slots, and NOT the both/hybrid option.
- **D-02:** Primitive set = **Stack + Grid + Row/Cluster + Spacer**. (Stack = vertical/horizontal flow w/ gap; Grid = columns + gap; Row/Cluster = horizontal wrap/align; Spacer.)
- **D-03:** Primitives are a **distinct, closed structural node kind** in the IR — NOT registry component instances. They may nest component instances AND each other to arbitrary depth. The AST validator must allow them as a known structural vocabulary; **component instances still require exact registry membership** (REG-03 unchanged). The existing per-instance `slots` field is retained for component-internal named slots (e.g. Card content), not general children.
- **D-04:** Primitive spacing/sizing values are **token refs + responsive per-profile** — `gap`/`padding`/`columns` resolve only to Jio dimension tokens (Spacing-*, f-step grid; zero-literal rule enforced) AND carry per-output-profile values (web vs app-screen vs IG-portrait) resolved from the foundation responsive system so layout adapts across profiles.

**Real copy + advisor threading (root causes #2 & #3)**
- **D-05:** **Thread advisor outputs into IR assembly (true assembler-last).** The IR Generator must receive and consume the Planner's intent + message hierarchy, the Design advisor's per-section layout/component spec, and the ToV advisor's per-section copy spec. Copy comes from ToV; structure from Design + layout primitives. The advisors stop being dead weight. (Chosen over collapsing to a single richer generation call.)
- **D-06 (MAJOR — user-explicit scope expansion):** Build a **multi-agent transparency layer**. The pipeline must be visible and trustable, with each agent leaving a persisted trace. Principle: **transparency, not noise**. Full set in scope:
  - **(a) Persist agent steps** as structured run metadata (append-only Convex, consistent with existing `experience*` persistence).
  - **(b) Chat / run-panel activity** — named, human-readable step lines. NOTE: 04.1 D-11 already streams an in-place step ticker — this is largely *enriching labels + surfacing real agent outputs*, low lift.
  - **(c) Canvas agent-activity pointers/cursors** — lightweight per-agent pointers highlighting regions each agent touched. **HIGHEST-RISK piece** — requires net-new agent→IR-node→canvas-region mapping infrastructure.
  - **(d) "How this was built" trace** on the artifact card — expandable: planner output, design recs, ToV recs, component registry matches, validation result, visual evaluation score.
- **D-07:** **Copy grounding = user prompt intent + planner message hierarchy** (most→least prominent). Reuses existing planner output + ToV engine. (Pulling the brand's REAL voice config from Convex was offered and NOT selected — minimal `DEFAULT_LAB_VOICE_CONFIG` is acceptable this phase.)
- **D-08:** **GAP-01 backfill → last-resort safety net only.** Keep strictly as a convergence guarantee for REQUIRED STRUCTURAL props the model still omits; real ToV copy fills content props first. **Backfilled values must be flagged in run metadata.** (Chosen over demote-to-gap-emit.)

**Quality enforcement bar**
- **D-09:** **Enforce "reads as a genuine product UI" via the eval + bounded IR-patch repair loop** (reuse the Phase-3 multimodal evaluator + repair). Quality becomes a gate, not a hope.
- **D-10:** **Blocking set = structural checks + multimodal threshold (strongest bar).** BLOCKING deterministic structural checks: layout has real nesting (not a flat pile), no placeholder / `placehold.co` / backfilled content props in final output, every section carries substantive copy. AND the multimodal evaluator's overall score must clear a threshold. **FLAG for research:** a subjective threshold risks repair churn / false refusals — tune empirically and keep deterministic checks as the primary gate.
- **D-11:** **Terminal behavior on persistent failure = emit a quality GAP.** Keep the existing bounded repair cap; if gates still fail after the cap, emit a quality gap report rather than placing a low-quality artifact.

**Artifact-type focus**
- **D-12:** **Depth-first on web-ui / landing page.** Prove the full pattern on one primary type FIRST, then generalize. Generalization may extend beyond this phase.

### Pattern 1: Layout-primitive node as a distinct, closed `kind` (D-01/D-03)

**What:** Add a third leaf to the IR's recursive composition vocabulary — alongside `JioIRComponentInstance` and `MarkupFreeString` — discriminated on `kind: 'layout'`, with a closed `primitive` enum and recursive `children`. Keep it `.strict()` so no `tag`/`rawHtml`/markup key can ride along.

**When to use:** Whenever the IR expresses arrangement (sections, columns, rows, spacing) — i.e. always, since `irToAst` no longer hardcodes a `Stack` wrapper.

**Example (target schema shape — discriminated union arm):**
```typescript
// schema.ts — NEW. The closed structural vocabulary (D-02/D-03).
// Source: derived from existing JioIRComponentInstance (schema.ts:101-125)

export const LayoutPrimitive = z.enum(['stack', 'grid', 'row', 'cluster', 'spacer']);

/** A responsive, token-only dimension value (D-04). Keys mirror data-Breakpoint. */
const ResponsiveTokenValue = z.union([
  TokenRefString,                                  // e.g. '4' | 'gutter' (Spacing key) — markup-free + token-only
  z.record(z.enum(['S','M','ML','L','XL']), TokenRefString), // per-breakpoint (D-04)
]);

export interface JioIRLayoutNodeT {
  kind: 'layout';
  id: string;
  primitive: 'stack' | 'grid' | 'row' | 'cluster' | 'spacer';
  /** Token-only, optionally responsive. NEVER a literal length. */
  gap?: ResponsiveTokenValueT;
  padding?: ResponsiveTokenValueT;
  columns?: ResponsiveTokenValueT;       // grid only
  direction?: 'row' | 'column';          // stack
  align?: 'start' | 'center' | 'end' | 'stretch';
  justify?: 'start' | 'center' | 'end' | 'between' | 'around';
  wrap?: boolean;                        // row/cluster
  surfaceMode?: string;
  /** Recursive children: nested layout nodes OR registry instances (D-03). */
  children: Array<JioIRLayoutNodeT | JioIRComponentInstanceT>;
}

export const JioIRLayoutNode: z.ZodType<JioIRLayoutNodeT> = z.lazy(() =>
  z.object({
    kind: z.literal('layout'),
    id: z.string().min(1),
    primitive: LayoutPrimitive,
    gap: ResponsiveTokenValue.optional(),
    padding: ResponsiveTokenValue.optional(),
    columns: ResponsiveTokenValue.optional(),
    direction: z.enum(['row','column']).optional(),
    align: z.enum(['start','center','end','stretch']).optional(),
    justify: z.enum(['start','center','end','between','around']).optional(),
    wrap: z.boolean().optional(),
    surfaceMode: z.string().optional(),
    children: z.array(z.union([JioIRLayoutNode, JioIRComponentInstance])),
  }).strict(),  // no tag/rawHtml — preserves IR-02
);
```
Then make `JioIRSection.instances` a `children: Array<JioIRLayoutNode | JioIRComponentInstance>` (rename or add alongside for backward-compat round-trip; STATE.md notes existing rows must round-trip — prefer an **additive optional `layout?` tree** plus keeping `instances` so old persisted IRs parse). **Confirm the round-trip strategy with the planner** — schema.ts is described as a "frozen contract" in STATE.md, but D-01 explicitly authorizes modifying it this phase.

**Critical:** `TokenRefString` must be a markup-free string further refined to a Spacing-key / token grammar so the zero-literal rule (D-04) is enforced at the IR boundary, not only at the AST validator. The validator's `BRAND_ALLOWED_REGEX` / `VISUAL_LITERAL_RE` (astValidator.ts:308-330) is the AST-side backstop.

### Pattern 2: Compile layout primitives to real Jio components (D-01/LAYOUT-05)

**What:** `irToAst` currently hardcodes `type: 'Stack'` then remaps `Stack→Container` in the workflow bridge (irToAst.ts:104, workflow.ts:79-85). Replace this with an explicit primitive→component map.

**When to use:** In `irToAst.ts` (the only JSX-emitting bridge — IR-03/Pitfall 2 guarantee: no `element` node ever).

**Example:**
```typescript
// irToAst.ts — map each layout primitive to the real Jio layout component.
// Container covers stack/row/cluster (direction/wrap/justify/align/gap/padding
// all → var(--Spacing-*)); Grid covers grid (responsive columns + gap).
// Source: Container.shared.ts:148-152,290-307 ; Grid/responsive.ts:4-20
function layoutNodeToComponentNode(node: JioIRLayoutNodeT): ComponentASTNode {
  const type = node.primitive === 'grid' ? 'Grid' : 'Container';
  const props: Record<string, ASTSerializableValue> = {};
  if (node.primitive === 'stack') props.direction = node.direction ?? 'column';
  if (node.primitive === 'row' || node.primitive === 'cluster') {
    props.direction = 'row';
    if (node.wrap ?? node.primitive === 'cluster') props.wrap = true;
  }
  if (node.gap !== undefined) props.gap = node.gap;          // Spacing key, e.g. '4'
  if (node.padding !== undefined) props.padding = node.padding;
  if (node.columns !== undefined) props.columns = node.columns; // responsive ok for Grid
  if (node.align) props.align = node.align;
  if (node.justify) props.justify = node.justify;
  if (node.surfaceMode) props.surfaceMode = node.surfaceMode;
  return { id: node.id, kind: 'component', type, props,
    children: node.children.map(c =>
      c.kind === 'layout' ? layoutNodeToComponentNode(c) : instanceToComponentNode(c)) };
}
```
`spacer` compiles to a `Container` with a fixed `gap`/`minHeight` from a Spacing token (a leaf with no children). **Do NOT invent a `Stack`/`Row`/`Spacer` component** — they don't exist in `@oneui/ui` (verified: only `Container`/`Grid`/`GridOverlay` exist). The primitive vocabulary is an IR concept that *maps onto* Container/Grid.

### Pattern 3: Validator accepts layout nodes as structural vocab; instances still need exact registry membership (D-03/LAYOUT-05)

**What:** The `astValidator` walk (astValidator.ts:388-464) currently treats EVERY `kind: 'component'` node as a registry lookup. After compile, layout nodes become `Container`/`Grid` `ComponentASTNode`s. Two valid approaches:

- **(A) Recognize compiled layout components structurally** — treat `Container`/`Grid` as a known structural set that bypasses the empty-meta prop allowlist but still passes the literal/token hook (so `gap='4'` validates as a Spacing token, a `12px` literal blocks). This is the least invasive: the validator already has a `kind:'component'` branch; add a `STRUCTURAL_LAYOUT_TYPES = new Set(['Container','Grid'])` early-return that skips the prop-name allowlist (since their generated meta is empty, queryRegistry.ts:54-55) but still runs `checkLiteralHook`.
- **(B) Carry the `layout` kind through to the validator AST** — add a `LayoutNode` arm to `ArtifactAstNode`. More faithful to D-03 but touches the validator's node union, the bridge, and the import synthesizer.

**Recommendation:** Start with **(A)** for the proof-point (web-ui), because the compiled output is already `Container`/`Grid` and the bridge synthesizes their Jio imports (workflow.ts:125-139). It satisfies "the validator allows them as a known structural vocabulary while component instances still require exact registry membership" without restructuring the validator AST. Document (B) as the cleaner end-state.

### Pattern 4: Deterministic structural quality checks (D-10/QUAL-04) — the PRIMARY gate

**What:** Three BLOCKING checks, evaluated before the multimodal track:
1. **Real nesting** — the IR/AST must contain at least one layout node with depth ≥ 2 OR a grid/row (not a single flat Stack of leaves). Compute on the IR (before flatten) where the `layout` tree is explicit.
2. **No placeholder/backfilled content** — scan final content props for the documented sentinels: `https://placehold.co`, the `'<X> item'` / `'Untitled'` defaults, and any prop flagged in the backfill-metadata channel (D-08). A backfilled CONTENT prop in final output blocks; a backfilled STRUCTURAL prop (e.g. `alt` presence) is allowed but flagged.
3. **Substantive copy per section** — every section carries a non-empty headline/body (from `copySpec`), not just component chrome.

**Where:** Checks (2) operate on string content visible in the compiled AST (string props) — best placed inside the existing `walkNode` AST pass. Checks (1) and (3) operate on the IR section/layout TREE, whose section boundaries are LOST after the AST flatten — they MUST run as an IR-shaped pre-check BEFORE `walkNode`. Co-locate the pre-check in `astValidator` as a `validateIRStructure(ir, backfilled[])` function that runs first and feeds its `flat-layout`/`empty-section-copy` blocking codes into the same `blocking(...)` accumulator the AST pass uses (so repair suggestions are single-sourced and drive the existing repair loop). Use NEW violation codes (`flat-layout`, `placeholder-content`, `empty-section-copy`) so repair suggestions are actionable.

**Why primary:** D-10 explicitly flags the subjective threshold as repair-churn risk. Deterministic checks never produce false refusals on a genuinely-good artifact and give the repair loop precise, patchable feedback (unlike a low vision score).

### Pattern 5: Thread advisor specs (D-05) — true assembler-last

**What:** Extend `GenerateIRInput` (irGenerator.ts:104-118) to carry `messageHierarchy`, `designSpecs`, `copySpecs`; change `generateStep` (workflow.ts:399-411) to pass `ctx.plan.sections` (with `intent`), `ctx.plan.messageHierarchy`, `ctx.designSpec.designSpecs`, `ctx.copySpec.copySpecs`. Change `buildSectionPrompt` (irGenerator.ts:638-660) to inject, per section: the design spec's chosen components + surfaceMode (so the per-section model call composes the advised components into layout primitives), and the ToV copy spec's headline/body/cta as the content props (so the model places real copy, not placeholders). The `messageHierarchy` orders prominence (D-07).

**When:** The IR Generator stays the single committing step (no mid-generation tool loop — workflow.ts:397-398 already asserts this).

### Anti-Patterns to Avoid
- **Inventing Stack/Row/Spacer components in `@oneui/ui`.** They don't exist; layout primitives are IR concepts compiled to `Container`/`Grid`. Adding components would be net-new DS surface area outside this phase's scope and risks the zero-literal/Storybook gates.
- **Putting layout props as literals.** `gap: '16px'` blocks (VISUAL_LITERAL_RE). Always Spacing keys → `var(--Spacing-*)`.
- **Making the multimodal threshold the primary gate.** D-10 warns against this; deterministic checks lead.
- **A mid-generation advisor tool loop.** ORCH-04 + assembler-last require advisors run as discrete steps; the IR Generator only READS their specs from context.
- **Building the canvas cursor mapping inline with the schema work.** D-06c needs its own agent→node→region contract; coupling it to the schema wave will stall the proof-point.
- **Persisting the trace as a new table.** Existing `experience*` tables are append-only and additive-optional is the established pattern (schema.ts:2050-2096); add an optional field, never a migration.

## Common Pitfalls

### Pitfall 1: Breaking the round-trip of already-persisted flat IRs
**What goes wrong:** Changing `JioIRSection.instances` to a required `children` union makes every persisted v1 IR fail `parseIR`, breaking version history/canvas restore.
**Why:** `experienceArtifactVersions.ir` is `v.any()` with no migration story (append-only).
**How to avoid:** Make the layout tree additive-optional, or bump `version` to 2 and upcast v1 (`instances → children`) at parse. Keep `parseIR` accepting both.
**Warning signs:** `schema.test.ts` / `irToAst.test.ts` fixtures fail; canvas "restore on reopen" shows empty cards.

### Pitfall 2: Anthropic structured-output schema rejections (Zod 4)
**What goes wrong:** Adding the layout node to a structured-output schema with `z.record(z.string(), …)` (emits `propertyNames`) or `.int()`/`.min()` (emits integer min/max) → Anthropic 400.
**Why:** Documented repeatedly in-repo (irGenerator.ts:150-159, plannerAgent.ts:60-69, evaluatorRubric.ts:13-21).
**How to avoid:** Use `z.object({}).catchall(z.unknown())` for arbitrary bags, plain `z.number()` (clamp after), and a closed `z.enum` for `primitive`. Per-breakpoint maps use `z.object({S:…,M:…})` with explicit keys, NOT `z.record`.
**Warning signs:** 400 "property 'propertyNames' is not supported" / "minimum/maximum not supported".

### Pitfall 3: Validator blocks compiled layout components on empty meta
**What goes wrong:** `Container`/`Grid` have EMPTY generated-prop meta (queryRegistry.ts:54-55). The validator's `checkComponentProps` (astValidator.ts:202-244) flags any prop (`direction`, `gap`) as `invalid-prop` because the meta allowlist is empty.
**Why:** The required-prop/allowlist loop iterates `item.props` which is `[]` for these.
**How to avoid:** Add `STRUCTURAL_LAYOUT_TYPES` early-handling (Pattern 3A) so layout components skip the prop-name allowlist but still run `checkLiteralHook` (token enforcement). Verify with an `astValidator.test.ts` case: `Container gap='4'` passes, `gap='16px'` blocks.
**Warning signs:** Every generated artifact blocks with `invalid-prop` on `direction`/`gap`/`columns`.

### Pitfall 4: Multimodal threshold causes repair churn / false refusals (D-10 flagged)
**What goes wrong:** `threshold: 4` (of 5) is high; a genuinely-good landing page that the vision judge scores 3.8 loops to the cap and emits a false quality gap.
**Why:** Subjective scores are noisy; a fixed high bar over-rejects.
**How to avoid:** (a) Make deterministic checks the gate (Pattern 4) — they decide pass/fail. (b) Treat the multimodal composite as **advisory/secondary**: either lower its threshold (empirically — see Open Questions) OR require it only as a tie-breaker / warning, not a hard block, for the web-ui proof-point. (c) Keep `epsilon` convergence so no-improvement stops the loop before the cap. Record threshold experiments in metadata.
**Warning signs:** Frequent `repair-exhausted` / quality-gap on artifacts that look fine in preview.

### Pitfall 5: Backfill metadata not threaded to the quality check
**What goes wrong:** D-08 says backfilled values must be flagged; D-10/QUAL-04 says backfilled CONTENT props must block. If the flag isn't persisted/threaded, the structural check can't distinguish a real ToV string from a backfilled `'Hero item'`.
**Why:** `backfillRequiredProps` mutates props in place (irGenerator.ts:549-566) with no provenance.
**How to avoid:** Have `backfillRequiredProps` return/record which `(instanceId, propName)` pairs it filled (a `backfilled[]` list on the generate result), persist it in run metadata, and feed it to the structural check so a backfilled CONTENT prop blocks while a backfilled STRUCTURAL prop (e.g. `alt`) only flags.
**Warning signs:** Placeholder text ships; or the check over-blocks legitimate short copy.

### Pitfall 6: Adding observability spans expecting Mastra-native tracing
**What goes wrong:** Reaching for `@mastra/observability` for the trace.
**Why:** STATE.md / workflow.ts:246-252 record that `@mastra/observability@1.14.0` is NOT installed at the pinned `@mastra/core@1.37.1`; passing an `observability` instance silently falls back to NoOp.
**How to avoid:** The agent-step trace (D-06a) rides the existing `ExperienceBuilderEvent` stream + a persisted run-metadata field — NOT a Mastra observability span. Observability is a P5 (PROD-02) concern.
