# Phase 04.2: Generation output quality - Context

**Gathered:** 2026-06-03
**Status:** Ready for planning

<domain>
## Phase Boundary

Make generated artifacts read as **genuine product UIs composed from real Jio components and foundations** â€” not the structurally-valid-but-flat, placeholder-heavy stacks they are today. This is the **first phase that MODIFIES the generation engine** (it was a read-only contract through Phase 04.1).

Three engine-side root causes are in scope:
1. **No compositional layout in the Jio Experience IR** â€” `JioIRSection.instances` is a flat array; the assembler emits a vertical pile with no nesting/columns/grids.
2. **Advisors don't reach the assembler** â€” the Planner (`runPlanner`), Design advisor (`designAdapter`), and ToV advisor (`voiceAdapter`) all exist and call the model, but `GenerateIRInput` only receives `sections: {id, name}`. Their copy/layout output is discarded, so the IR Generator re-asks a generic per-section model call and copy stays thin.
3. **GAP-01 placeholder backfill** â€” the deterministic required-prop backfill in `irGenerator.ts` fills missing content props with placeholders (`"X item"`, `placehold.co`, a font token for icons) that now render as the visible placeholder text.

This phase fixes all three AND adds a **multi-agent transparency layer** (user-explicit): the Planner/Design/ToV/Assembler/Compiler/Validator/Evaluator pipeline becomes legible â€” each agent leaves a visible, persisted trace â€” because the user frames the agents as *how the UI is defined, refined, and approved*, not backend plumbing. Final IR remains the single source of truth.

**Primary target this phase:** web-ui / landing page (depth-first proof-point). Generalization to app-screen / dashboard / social / carousel follows after the pattern is proven (may extend beyond this phase).

**Out of scope:**
- Changing the IR's **markup-free invariant**, the **repair-patches-IR-never-JSX** rule, the **Mastra-orchestrates / AI-SDK-is-model-only** boundary, append-only Convex persistence, or Lab isolation. All locked.
- Inventing components, tokens, dimensions, or type scales â€” missing pieces still emit gap reports.
- New artifact types / output profiles beyond what Phases 1â€“4 deliver.
- Real-time collaboration, cost tracking (Phase 5).
- Full generalization of the quality system across every artifact type (web-ui/landing is the in-phase proof-point; broader types may carry over).

</domain>

<decisions>
## Implementation Decisions

### Compositional layout (root cause #1)
- **D-01:** Express layout via **first-class IR layout primitives** â€” explicit layout nodes in the canonical IR schema that compile to Jio layout CSS. NOT layout-via-existing-registry-components-in-slots, and NOT the both/hybrid option.
- **D-02:** Primitive set = **Stack + Grid + Row/Cluster + Spacer**. (Stack = vertical/horizontal flow w/ gap; Grid = columns + gap; Row/Cluster = horizontal wrap/align; Spacer.)
- **D-03:** Primitives are a **distinct, closed structural node kind** in the IR â€” NOT registry component instances. They may nest component instances AND each other to arbitrary depth. The AST validator must allow them as a known structural vocabulary; **component instances still require exact registry membership** (REG-03 unchanged). The existing per-instance `slots` field is retained for component-internal named slots (e.g. Card content), not general children.
- **D-04:** Primitive spacing/sizing values are **token refs + responsive per-profile** â€” `gap`/`padding`/`columns` resolve only to Jio dimension tokens (Spacing-*, f-step grid; zero-literal rule enforced) AND carry per-output-profile values (web vs app-screen vs IG-portrait) resolved from the foundation responsive system so layout adapts across profiles.

### Real copy + advisor threading (root causes #2 & #3)
- **D-05:** **Thread advisor outputs into IR assembly (true assembler-last).** The IR Generator must receive and consume the Planner's intent + message hierarchy, the Design advisor's per-section layout/component spec, and the ToV advisor's per-section copy spec. Copy comes from ToV; structure from Design + layout primitives. The advisors stop being dead weight. (Chosen over collapsing to a single richer generation call.)
- **D-06 (MAJOR â€” user-explicit scope expansion):** Build a **multi-agent transparency layer**. The pipeline must be visible and trustable, with each agent leaving a persisted trace. Principle: **transparency, not noise** â€” understandable without implementation knowledge. Full set is in scope:
  - **(a) Persist agent steps** as structured run metadata (append-only Convex, consistent with existing `experience*` persistence).
  - **(b) Chat / run-panel activity** â€” named, human-readable step lines ("Planner is defining structure", "Design advisor is selecting layout/hierarchy/components", "Tone advisor is refining copy & CTA", "Assembler is producing valid One UI IR", "Compiler is validating React + Jio CSS", "Evaluator is checking screenshot quality"). NOTE: Phase 04.1 D-11 already streams an in-place step ticker â€” this is largely *enriching labels + surfacing real agent outputs*, low lift.
  - **(c) Canvas agent-activity pointers/cursors** â€” lightweight per-agent pointers highlighting the regions each agent touched (Design â†’ layout regions; ToV â†’ copy/CTA; Evaluator â†’ hierarchy/spacing/density/a11y issues). **HIGHEST-RISK piece** â€” requires net-new agentâ†’IR-nodeâ†’canvas-region mapping infrastructure.
  - **(d) "How this was built" trace** on the artifact card â€” expandable: planner output, design recs, ToV recs, component registry matches, validation result, visual evaluation score.
- **D-07:** **Copy grounding = user prompt intent + planner message hierarchy** (mostâ†’least prominent), so headlines/body/CTA are specific to the request and weighted by prominence. Reuses existing planner output + ToV engine. (Pulling the brand's REAL voice config from Convex was offered and NOT selected â€” the minimal `DEFAULT_LAB_VOICE_CONFIG` is acceptable this phase; see Deferred.)
- **D-08:** **GAP-01 backfill â†’ last-resort safety net only.** Keep it strictly as a convergence guarantee for REQUIRED STRUCTURAL props the model still omits; real ToV copy fills content props first, so placeholders rarely surface. **Backfilled values must be flagged in run metadata** so the user/evaluator knows they are fallback (ties into D-06 transparency). (Chosen over demote-to-gap-emit.)

### Quality enforcement bar
- **D-09:** **Enforce "reads as a genuine product UI" via the eval + bounded IR-patch repair loop** (reuse the Phase-3 multimodal evaluator + repair). Quality becomes a gate, not a hope. (Chosen over improve-generation-only.)
- **D-10:** **Blocking set = structural checks + multimodal threshold (strongest bar).** BLOCKING deterministic structural checks: layout has real nesting (not a flat pile), no placeholder / `placehold.co` / backfilled content props in final output, every section carries substantive copy. AND the multimodal evaluator's overall score must clear a threshold to be "done". **FLAG for research:** a subjective threshold risks repair churn / false refusals â€” tune the threshold empirically and keep deterministic checks as the primary gate.
- **D-11:** **Terminal behavior on persistent failure = emit a quality GAP.** Keep the existing bounded repair cap; if the gates still fail after the cap, emit a quality gap report (like a foundation/component gap) rather than placing a low-quality artifact. (Chosen over deliver-best-of-N-and-flag.)

### Artifact-type focus
- **D-12:** **Depth-first on web-ui / landing page.** Prove the full pattern (layout primitives + advisor threading + real copy + quality gating + transparency) reads as a genuine product UI on one primary type FIRST, then generalize to app-screen / dashboard / social / carousel. Generalization may extend beyond this phase.

### Claude's / Researcher's Discretion
- Concrete requirement IDs (ROADMAP says TBD) â€” derive a `QUAL-*` / `LAYOUT-*` / `AGENT-*` family during planning, or confirm with the user.
- Exact IR schema shape for the layout-primitive node kind (discriminated union vs. a `layout` node type), and the compiler (`irToAst.ts`) + AST-validator changes to treat primitives as a known structural vocabulary while preserving the markup-free invariant and exact registry membership for component instances.
- How responsive-per-profile primitive values map onto the existing dimension/responsive cascade (`data-Breakpoint`, f-step scale).
- The canvas agent-cursor interaction model and the agentâ†’IR-nodeâ†’canvas-region mapping (D-06c) â€” the heaviest new infra; research should assess whether it needs its own plan/wave.
- Where agent-step metadata persists (Convex `experience*` table shape) and whether it attaches per artifact version.
- The exact deterministic quality-check definitions (D-10) and the multimodal threshold value (empirical).
- How "depth-first then generalize" (D-12) splits across plans/waves.
- The empirical repair cap value (D-11) vs. latency budget.
