# Phase 03.1 Plan 04: Complete DaytonaExecutor render + live iframe + preview-error wiring Summary

**`DaytonaExecutor.render` now runs the full deferred D-01 path end-to-end (offline-mockable): assemble the Plan 01 self-contained asset, create a network-blocked sandbox from the Plan 02 named snapshot, screenshot per profile via in-box `capture.js`, start `serve.js` and mint a signed live preview URL (`getSignedPreviewUrl`), and KEEP the sandbox alive for the TTL — with the run route selecting Daytona only behind the opt-in gate and mapping a `previewError` run to a preview-error card instead of a gap.**

## Accomplishments

- **render() pipeline (Task 1).** Replaced the raw `input.bundle` upload with `assembleAsset({ bundle, brandId, ast })` -> the self-contained HTML written to `preview/asset.html`. Create now uses the Plan 02 named snapshot (`{ snapshot: PREVIEW_SNAPSHOT_NAME, networkBlockAll: true }`) with a `buildImage` escape hatch (`{ image: buildDaytonaImage(), … }` + `onSnapshotCreateLogs`) to seed/observe the cold build. Per-profile `capture.js` capture loop retained, repointed at `asset.html`, base64 stdout -> PNG Buffer (PREV-04).
- **Live iframe (Task 1).** Before minting the URL, `serve.js` is started backgrounded on the preview port; `getSignedPreviewUrl(previewPort, previewTtlSeconds)` mints the immutable live URL (PREV-02). `previewState = { url, expiresAt, sandboxId }` and OMITS `sameOrigin` (strict allow-scripts, PREV-01).
- **Lifecycle reconciliation (Task 1).** Removed the unconditional `finally { sandbox.delete() }`. The happy path KEEPS the sandbox alive for the TTL; the only per-render delete is the delete-on-ERROR path (no leaked sandbox on a mid-run throw, T-031-09). Added `expirePreview(sandboxId)` for explicit TTL-expiry teardown.
- **Real-shape tests (Task 1).** Rewrote `DaytonaExecutor.test.ts` to the six plan behaviors with a higher-fidelity mock (records create params, the upload Buffer contents, serve-vs-capture command strings, `getSignedPreviewUrl`, `delete`, and a `get(id)` for `expirePreview`). 9/9 pass; full preview suite 27/27.
- **Route wiring (Task 2).** Added the opt-in selection gate (`EXPERIENCE_LAB_PREVIEW_EXECUTOR === 'daytona' && DAYTONA_API_KEY`) — default stays `LabAstExecutor`. Branched the result frame FIRST on `run.previewError` so a preview-infra failure renders a preview-error card (not a gap/"generation refused" card), and `persistRun` now persists the generated IR on a preview-error run (generation succeeded; only the preview step threw). Added additive `RunResultFrame.previewError`.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] SDK `create` overload mismatch (onSnapshotCreateLogs)**
- **Found during:** Task 1 typecheck.
- **Issue:** A single `create(params, { onSnapshotCreateLogs })` call with a union of snapshot-or-image params failed overload resolution — the SDK's from-snapshot create overload accepts only `{ timeout }` in options, while `onSnapshotCreateLogs` is honoured only on the from-image overload.
- **Fix:** Split into two typed `create` calls — `buildImage` path passes `{ image, … }` + `{ onSnapshotCreateLogs }`; the default snapshot path passes `{ snapshot, … }` with no options. Semantics unchanged (onSnapshotCreateLogs is only meaningful when a build actually happens).
- **Files:** `DaytonaExecutor.ts`. **Commit:** `62b2986b`.

**2. [Rule 2 - Missing critical functionality] Persist the IR on a preview-error run**
- **Found during:** Task 2.
- **Issue:** `persistRun` only persisted the artifact when `outcome === 'artifact'`. A preview-error run carries `outcome === 'error'` but a fully-generated `run.ir` — without this branch the generated IR/version would be lost, contradicting the plan's "still persist the IR/version that DID generate."
- **Fix:** Changed the persist condition to `(run.outcome === 'artifact' || run.previewError) && run.ir`. The thumbnail upload naturally no-ops (no screenshots on a preview throw).
- **Files:** `route.ts`. **Commit:** `f609777f`.

### Decision (not a deviation): additive `PreviewState.sandboxId`
The kept-alive lifecycle requires a handle for TTL-expiry teardown. Rather than touch the frozen `render()`/`RenderInput`/`RenderResult` signatures, I added an additive optional `sandboxId?` to `PreviewState` — exactly the additive pattern Plan 03 used for `previewError`. It is a sandbox identifier, never a secret (the no-leak canary still passes).
