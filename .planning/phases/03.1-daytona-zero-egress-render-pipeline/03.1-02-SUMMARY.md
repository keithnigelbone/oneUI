# Phase 03.1 Plan 02: Daytona Render Runtime (image recipe + in-box harness) Summary

A version-pinned `buildDaytonaImage()` recipe that bakes Playwright 1.59.1 + Chromium at BUILD time so the runtime sandbox holds `networkBlockAll` (PREV-01), plus the three in-box harness scripts — `capture.js` (file:// screenshot → base64 PNG, PREV-04), `serve.js` (static preview server), and `probe.js` (zero-egress proof) — none of which import the Daytona SDK.

## What Was Built

### Task 1 — In-box harness scripts (commit 9936062)
Three credential-free CommonJS scripts under `src/daytona/harness/`, baked into the image and run inside the network-blocked sandbox:

- **capture.js** — `require('playwright')` chromium, `newContext({ viewport, deviceScaleFactor: 2 })`, `page.goto('file:///home/pwuser/preview/asset.html', { waitUntil: 'load' })` (NOT `networkidle` — RESEARCH Pitfall 3), double-`requestAnimationFrame` + `document.fonts.ready` settle (mirrors `playwrightRenderer.ts:194-201`), then `page.screenshot({ type: 'png', fullPage: false })` and `process.stdout.write(png.toString('base64'))`. Only image bytes cross the boundary; any error exits 1. Accepts the existing `DaytonaExecutor.buildCaptureCommand` arg shape (`--bundle`/`--width`/`--height`/`--format=png-base64`).
- **serve.js** — minimal `http.createServer` static server bound to the preview port (`--port`/`PORT`, default 3000) that serves ONLY `asset.html` at `/` (and `/asset.html`); every other path 404s (no traversal). For the deferred live iframe (PREV-02).
- **probe.js** — `fetch(url)` then exit 2 + `EGRESS-LEAK` on success, exit 0 + `EGRESS-BLOCKED` on rejection, exit 1 on missing arg (RESEARCH § Zero-egress Proof).

### Task 2 — Image recipe + version-pin test (commit 7133627)
- **image.ts** — exports `PLAYWRIGHT_VERSION = '1.59.1'`, `DAYTONA_IMAGE_BASE = 'mcr.microsoft.com/playwright:v1.59.1-jammy'`, `PREVIEW_SNAPSHOT_NAME = 'oneui-preview-v1'`, `SANDBOX_PREVIEW_DIR`, `HARNESS_DIR`, and `buildDaytonaImage(): Image` = `Image.base(...).runCommands('npm i -g playwright@1.59.1').addLocalDir(HARNESS_DIR, '/home/pwuser/preview').workdir('/home/pwuser')`. Code comments document the once-register/reuse snapshot lifecycle and `onSnapshotCreateLogs` surfacing by the Plan 04 caller.
- **image.test.ts** — 6 tests: pin equals the repo-resolved Playwright version, base tag equals `v<version>-jammy` and ends with the resolved suffix, snapshot constant is stable, the built dockerfile contains the base tag + `playwright@1.59.1` + the in-box preview dir, `contextList` includes the harness dir, and the harness dir physically contains capture/serve/probe.js.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Harness comments contained the literal `@daytonaio/sdk` string, breaking the `grep -L` SDK-clean check**
- **Found during:** Task 1 verification.
- **Issue:** The Task 1 verify command is `grep -L "@daytonaio/sdk" harness/*.js | wc -l | grep -q 3` (files that do NOT contain the string). My doc-comments said "MUST NOT import `@daytonaio/sdk`", so all three files matched the pattern and `grep -L` returned 0 — a false failure even though none of them actually imports the SDK.
- **Fix:** Reworded the three comments to "MUST NOT import the Daytona SDK" (semantics preserved; the literal package token removed from the harness files). `node --check` and the actual no-import invariant are unaffected.
- **Files modified:** capture.js, serve.js, probe.js.
- **Commit:** 9936062 (fixed before the Task 1 commit).

### Decision: Image imported as a value (no fallback descriptor)
The plan offered a conditional: if a CI import-guard forbids `@daytonaio/sdk` outside `DaytonaExecutor.ts`, export a pure recipe descriptor instead. A repo-wide grep for any guard/lint referencing `@daytonaio/sdk` found none, so `image.ts` imports the `Image` builder directly (the plan's preferred path, and the one its `image.test.ts` assertions on `image.dockerfile`/`image.contextList` require).

### Note: vitest binary not linked in the worktree
`pnpm vitest`/`pnpm test` fail in the worktree (`Command "vitest" not found` — the worktree has no linked `.bin`). Modules resolve from the hoisted main-repo `node_modules`, so tests were run via the absolute `node_modules/.bin/vitest` binary. This is a worktree-environment artifact, not a code issue; the orchestrator's post-merge `pnpm --filter @oneui/experience-builder-preview test` will run normally on the merged branch.
