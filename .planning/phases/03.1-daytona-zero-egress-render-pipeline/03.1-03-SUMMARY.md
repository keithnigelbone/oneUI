# Phase 03.1 Plan 03: Preview-error vs gap polarity Summary

**A preview-executor throw (Daytona/sandbox infra failure) is now contained at the preview step and surfaced as a preview-error terminal carrying `previewError`, instead of being demoted to a misleading "gap — generation refused" — while genuine Jio DS foundation gaps still finalize as 'gap'.**

## Accomplishments
- Root-caused and fixed the failure-attribution bug (RESEARCH Pitfall 5 / CONTEXT "Bug to fix alongside" / T-031-06): a thrown `PreviewExecutor.render` propagated to the run-level `catch`, which `finalizeRun` then demoted to `'gap'` because IR existed but `previewState.url` was empty.
- Added an additive `previewError?: { message: string }` signal to both `RunContext` and `RunExperienceResult`. The frozen `outcome` wire union (`'artifact'|'gap'|'error'|'suspended'`) and the `RunCompletedEvent` union are unchanged.
- `previewStep` now wraps `executor.render(...)` in try/catch: on throw it records `ctx.previewError`, leaves `previewState` empty, sets `rendered=false`, emits a `step preview status:'failed'` event, and returns `{ ctx }` (does not rethrow — rethrow was the demotion cause).
- `finalizeRun` skips the ir-but-no-url → `'gap'` demotion when `previewError` is set, surfacing `'error'` + the `previewError` flag for the route's preview-failed card.
- Two regression tests lock the polarity: a throwing executor yields a preview-error terminal (not gap); a real foundation gap still yields `'gap'` with no `previewError`.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Step event status 'failed' instead of 'error'**
- **Found during:** Task 1 (GREEN)
- **Issue:** The plan's `<action>` said to emit `step ... status:'error'`, but the frozen `ExperienceBuilderEvent` StepEvent schema (`packages/experience-builder-core/src/contracts/events.ts`) defines `status: z.enum(['started','completed','failed'])` — there is no `'error'` status. Emitting `'error'` would break the schema and the existing happy-path test that validates every event against the frozen union (which the plan explicitly requires to stay unchanged).
- **Fix:** Emitted `status: 'failed'` (the frozen-union equivalent) and updated the regression test assertion to match. Added a code comment explaining the choice.
- **Files modified:** `workflow.ts`, `workflow.test.ts`
- **Commit:** `9566b524`

**2. [Rule 2 - Missing critical functionality] versionFreezeStep guard**
- **Found during:** Task 1 (GREEN)
- **Issue:** With the throw now contained, the run no longer aborts at preview — it continues to `versionFreezeStep`, which unconditionally set `outcome='artifact'`. That would falsely freeze a failed-preview run as an approved artifact (there is no live preview to ship). The plan's terminal-mapping intent (preview-error, not artifact) required this.
- **Fix:** `versionFreezeStep` early-returns `{ ctx }` when `ctx.previewError` is set, leaving the terminal for `finalizeRun` to label as a preview-error.
- **Files modified:** `workflow.ts`
- **Commit:** `9566b524`
