# Phase 03.1: Daytona zero-egress render pipeline - Research

**Researched:** 2026-06-02
**Domain:** Server-side React bundling (esbuild/vite) + Daytona custom-image sandboxing + in-box Playwright capture
**Confidence:** HIGH (SDK API + Playwright image + existing monorepo bundle pattern all verified from source; the live-iframe-vs-screenshots fork is a scoped recommendation, MEDIUM)

## Summary

The current `DaytonaExecutor` fails for four locked reasons (see CONTEXT). The single most important discovery in this research is that **the monorepo already solves the hardest sub-problem** — turning `@oneui/ui` into one self-contained, no-external-imports ESM asset — via `packages/ui/vite.playground.config.ts` (`pnpm build:playground`), which produces `dist/playground/oneui-playground.mjs` (~18MB raw / ~3.5MB gzipped) + `oneui-playground.css` with React/ReactDOM externalized and everything else (Base UI, icons, embla, `@oneui/shared`, `@oneui/tokens`) inlined. The Daytona bundler step should **reuse this exact artifact** rather than invent a parallel bundler. The only missing piece for Daytona is a thin per-run **app wrapper** (the generated TSX `GeneratedArtifact` rewritten to import from the pre-built bundle + the inlined brand CSS + an HTML host page), assembled into one HTML file with **esbuild** (already a transitive dep, fast, in-process `build({ write:false })` → string).

The second discovery: `@daytonaio/sdk@0.183.0` (installed, and the current latest — published 2026-05-29) exposes everything the executor needs, verified from `.d.ts`: `daytona.create({ image: Image, networkBlockAll, onSnapshotCreateLogs })`, `Image.base(...).runCommands(...).addLocalFile(...)`, `sandbox.fs.uploadFile(Buffer, path)`, `sandbox.process.executeCommand(cmd) → { exitCode, result, artifacts }`, `sandbox.getSignedPreviewUrl(port, ttl) → { url, token }`, `sandbox.delete()`. The image is built into a **snapshot before the sandbox is network-blocked** (build-time has network; runtime does not), so Playwright + Chromium baked at build time satisfy PREV-01 zero-egress at runtime.

Third: the local Playwright resolves to **1.59.1** (`node_modules/playwright/package.json`), which maps exactly to `mcr.microsoft.com/playwright:v1.59.1-jammy`. That image bakes **Node 24 + Chromium/Firefox/WebKit into `/ms-playwright`** but, per the official Dockerfile, does **NOT** include the `playwright` npm package — so the image recipe must `npm install playwright@1.59.1` at build time (matching the baked browser revision) so `capture.js` can `require('playwright')` in-box with no download.

**Primary recommendation:** Reuse `build:playground`'s pre-built `@oneui/ui` ESM bundle; per run, esbuild-assemble a one-file self-contained HTML asset (rewritten artifact + inlined brand CSS + bundle). Bake `mcr.microsoft.com/playwright:v1.59.1-jammy` + `npm i playwright@1.59.1` + the baked `capture.js`/`serve.js` harness into a Daytona snapshot via `Image.base(...)`. **Ship screenshots-only for v1** (immediate teardown); defer the live in-box iframe.

### Pattern 1: Reuse the pre-built `@oneui/ui` playground bundle as a virtual module
**What:** The generated `GeneratedArtifact` imports `@oneui/ui`. Rewrite those imports to resolve against the CI-built `oneui-playground.mjs` (which already inlines all of `@oneui/ui`'s non-React deps), then esbuild the small wrapper.
**When to use:** Every run. Never re-bundle `@oneui/ui` per run.
**Example (esbuild in-process, bundle-to-string):**
```ts
// Source pattern: mirrors packages/ui/src/playground/template.ts virtual-module idiom
// + esbuild stdin build. Verified shapes: compiler.ts emits `import { Button } from '@oneui/ui'`.
import { build } from 'esbuild';

const ENTRY = `
import React from 'react';
import { createRoot } from 'react-dom/client';
import * as OneUI from 'oneui-bundle';            // alias → pre-built playground mjs
${rewrittenArtifactSource /* GeneratedArtifact, import source swapped to 'oneui-bundle' */}
const root = document.getElementById('root');
createRoot(root).render(React.createElement(GeneratedArtifact));
`;

const result = await build({
  stdin: { contents: ENTRY, resolveDir: process.cwd(), loader: 'tsx' },
  bundle: true, write: false, format: 'iife', platform: 'browser', minify: true,
  jsx: 'automatic',
  // React/ReactDOM must be REAL here (Daytona has no Sandpack runtime providing them):
  // resolve 'react'/'react-dom/client' to the monorepo copies, and 'oneui-bundle'
  // to dist/playground/oneui-playground.mjs via an esbuild alias/plugin.
});
const js = result.outputFiles[0].text;
```
Then wrap into one HTML file with brand CSS + token CSS inlined (see Pattern 2).

### Pattern 2: Inline brand CSS server-side (reuse the engine, never the React hook)
**What:** `useBrandCSS` is a React hook (browser/SSR). For the asset, call the **pure engine functions** server-side to produce the `@layer brand` CSS string + token CSS, and inline both into the HTML `<style>` so `networkBlockAll` holds (no font/CSS fetch).
**When to use:** Per run, for the artifact's `brandId`.
**Example:**
```html
<!doctype html>
<html data-theme="light" data-density="default" data-Breakpoint="L">
  <head>
    <style id="oneui-tokens">/* oneui-playground.css (token cascade) */</style>
    <style id="oneui-foundation-tokens">/* generated brand @layer brand CSS */</style>
    <!-- Jio fonts MUST be inlined as @font-face base64 or omitted; no CDN under zero-egress -->
  </head>
  <body><div id="root"></div><script>/* esbuild iife from Pattern 1 */</script></body>
</html>
```
Brand CSS engine entrypoints: `@oneui/shared/engine` (`cssGenNew.ts`), typography via `generateFullTypographyCSSV2`, font CSS via `@oneui/ui` `foundationCSS.ts:generateTypographyFontCSSV2`. `[CITED: MEMORY.md useBrandCSS pipeline]`

### Pattern 3: Daytona custom image (build-time install, runtime zero-egress)
**What:** Build a snapshot once from the Playwright base, install matching playwright npm + bake the harness, then create network-blocked sandboxes from it.
**Example (verified against Image.d.ts + Daytona.d.ts):**
```ts
import { Daytona, Image } from '@daytonaio/sdk';

const image = Image
  .base('mcr.microsoft.com/playwright:v1.59.1-jammy')   // Node 24 + Chromium baked
  .runCommands('npm i -g playwright@1.59.1')             // package NOT in base image (build-time net OK)
  .addLocalDir('src/daytona/harness', '/home/pwuser/preview')  // capture.js (+serve.js)
  .workdir('/home/pwuser');

const daytona = new Daytona();  // reads DAYTONA_API_KEY server-side ONLY
const sandbox = await daytona.create(
  { image, networkBlockAll: true },                     // zero-egress at RUNTIME
  { onSnapshotCreateLogs: (c) => log.debug(c) },        // surface slow first build
);
```
**Build-once/reuse:** `create({ image })` builds + registers a snapshot; subsequent identical images reuse it. For determinism and speed, prefer registering a **named snapshot** (`daytona.snapshot` service) in a one-time setup script/CI step and then `create({ snapshot: 'oneui-preview-v1' })` at runtime, so per-run create skips the build entirely. `[CITED: daytona.io/docs/snapshots]`

### Pattern 4: In-box capture (no external CDP)
**What:** `capture.js` runs Playwright *inside* the sandbox against `file://…/asset.html`, mirroring `playwrightRenderer.ts`'s settle loop, prints base64 PNG to stdout.
**Example (mirrors playwrightRenderer.ts:165-204):**
```js
// preview/capture.js  (CommonJS; runs under baked Node 24 + npm playwright@1.59.1)
const { chromium } = require('playwright');
(async () => {
  const args = Object.fromEntries(process.argv.slice(2).map(a => a.replace(/^--/,'').split('=')));
  const browser = await chromium.launch({ headless: true });
  const ctx = await browser.newContext({
    viewport: { width: +args.width, height: +args.height }, deviceScaleFactor: 2,
  });
  const page = await ctx.newPage();
  await page.goto('file:///home/pwuser/preview/asset.html', { waitUntil: 'load' }); // file://, not networkidle
  await page.evaluate(() => new Promise(r => requestAnimationFrame(() => requestAnimationFrame(r)))); // double-rAF settle
  await page.evaluate(() => document.fonts && document.fonts.ready);                                  // fonts settle
  const png = await page.screenshot({ type: 'png', fullPage: false });
  process.stdout.write(png.toString('base64'));   // ONLY image bytes cross the boundary
  await browser.close();
})().catch(e => { console.error(e); process.exit(1); });
```
Note: change `waitUntil` from `'networkidle'` (used in the HTTP path) to `'load'` for `file://` — there is no network to idle, and `networkidle` can hang under zero-egress.

### Anti-Patterns to Avoid
- **Re-bundling `@oneui/ui` per run.** Reuse the CI `build:playground` artifact.
- **Connecting an external CDP into the sandbox.** Issue #4456 — not supported; under `networkBlockAll` the proxy is unreachable anyway. Run Playwright in-box.
- **`waitUntil: 'networkidle'` on a `file://` asset under zero-egress.** Use `'load'`.
- **Loading fonts/CSS from a CDN in the asset.** Inline them; otherwise `networkBlockAll` either breaks the render or is silently violated.
- **Putting `@daytonaio/sdk` import anywhere but `DaytonaExecutor.ts`.** CI import-guard.
- **`new Daytona()` key leaking into bundle/previewState/screenshot.** Key is server-side only; the signed URL carries `token` only.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| `@oneui/ui` → self-contained ESM | Custom per-run rollup of the whole library | The existing `vite.playground.config.ts` artifact | Already inlines all non-React deps; built once, cached |
| TSX transpile + bundle wrapper to string | Custom Babel/SWC pipeline | `esbuild.build({ stdin, write:false })` | One call, in-process, fast |
| Headless browser + system deps in a sandbox | Hand-rolled apt-get of Chromium libs | `mcr.microsoft.com/playwright:vX-jammy` base | Microsoft maintains exact browser+deps parity |
| Signed preview URL | Custom token signing | `sandbox.getSignedPreviewUrl(port, ttl)` | SDK-native, token-only, TTL'd |
| Brand CSS string generation | Re-implement OkLCH/surface stacking | `@oneui/shared/engine` pure functions | The engine is already pure + server-runnable |

**Key insight:** Nearly every "missing" piece already exists in the monorepo (playground bundle, brand CSS engine, Playwright settle loop in `playwrightRenderer.ts`). Phase 03.1 is mostly *wiring + a thin esbuild assembly step + an image recipe*, not new infrastructure.

## Common Pitfalls

### Pitfall 1: Playwright version skew between npm package and baked browsers
**What goes wrong:** `playwright` npm version ≠ the browser revision baked in `/ms-playwright` → "Executable doesn't exist" at runtime, with no network to download it.
**Why it happens:** Local resolves to 1.59.1 but `package.json` pins `^1.49.0`; image tag and `npm i playwright@X` must match.
**How to avoid:** Pin the image tag to the resolved Playwright version (`v1.59.1-jammy`) and `npm i -g playwright@1.59.1` at build. Add a check that both equal the repo's resolved version.
**Warning signs:** `browserType.launch: Executable doesn't exist at /ms-playwright/...`.

### Pitfall 2: Snapshot build is slow on first run (cold)
**What goes wrong:** First `create({ image })` blocks for minutes building the snapshot (image pull + `npm i`).
**How to avoid:** Pre-register a **named snapshot** once; runtime `create({ snapshot })` skips the build. Surface `onSnapshotCreateLogs` so the slow path is observable, not a silent hang.
**Warning signs:** Long first-run latency; opaque waits with no logs.

### Pitfall 3: `networkidle` hang under zero-egress
**What goes wrong:** `page.goto(url, { waitUntil: 'networkidle' })` never resolves because there is no network to go idle (and `file://` has no requests).
**How to avoid:** Use `waitUntil: 'load'` for the `file://` asset.

### Pitfall 4: Brand CSS/fonts fetched at runtime → render breaks or zero-egress violated
**What goes wrong:** Asset `<link>`s a font/CSS CDN; under `networkBlockAll` it 404s (broken type metrics) — or worse, appears to work because the block isn't actually enforced.
**How to avoid:** Inline ALL CSS (token + brand) into `<style>`; inline fonts as base64 `@font-face` or omit. This is also the PREV-01 proof surface.

### Pitfall 5: Preview-infra failure mislabeled as "gap — generation refused"
**What goes wrong:** When capture/preview throws, `finalizeRun` currently lands on `outcome:'gap'` (workflow.ts:476 sets gap on `!result.passed`; ctx.rendered comes from previewStep). The card reads "generation refused" though generation succeeded.
**How to avoid:** Distinguish a *preview-step error* (executor threw / `rendered:false` due to infra) from a *validation/DS gap*. Surface a `preview-error` outcome/flag separate from `gap`; the card copy must read "preview failed", not "generation refused". (CONTEXT "Bug to fix alongside".)
**Warning signs:** A run with passing validation but a Daytona exception shows "gap" on the card.

### Pitfall 6: Architecture mismatch on image build
**What goes wrong:** Daytona builds for AMD64; an arm64 dev machine building a local Dockerfile produces an incompatible image.
**How to avoid:** Use `Image.base(...)` declarative builder (Daytona builds AMD64 server-side) rather than `fromDockerfile` of a locally-built arm64 image. If `fromDockerfile`, pass `--platform=linux/amd64`. `[CITED: deepwiki daytona creating-sandboxes]`

## Live-iframe vs Screenshots-only — RECOMMENDATION

**Recommendation: SHIP SCREENSHOTS-ONLY FOR v1. Defer the live in-box iframe.**

Rationale:
- **Cost/complexity:** A live iframe requires (a) an in-box static server (`serve.js`) bound to the preview port, (b) keeping the sandbox alive for the TTL (no immediate `delete()`), and (c) the `getSignedPreviewUrl` proxy interplay validated against a live account (A5). Screenshots-only tears down in `finally` immediately — minimal cost, fewer failure modes.
- **Trust model already favors it:** The Daytona/untrusted path's `previewState` omits `sameOrigin`, so the canvas iframe is strict `allow-scripts` (no same-origin). A live untrusted iframe under that strict sandbox already loses same-origin fonts/CSS — so the live experience would be *degraded* relative to the first-party `LabAstExecutor` path anyway.
- **The canvas card lifecycle accommodates it:** PREV-03's thumbnail → lightweight → live progression can use the per-profile screenshot as the thumbnail/lightweight states. The "live" state can continue to be served by the first-party `LabAstExecutor` (which already returns a same-origin live URL) for trusted preview, while Daytona's job is the *zero-egress screenshot proof* (PREV-04) for the judge and the gallery. PREV-04 (per-profile screenshots for the evaluator) is fully satisfied by screenshots-only.
- **PREV-01/PREV-02 still hold:** PREV-01 (zero-egress) is proven by the live `fetch`-refused probe regardless of iframe. PREV-02 (immutable preview state) is satisfied by returning the per-profile screenshots as the immutable artifact; a signed live URL is not required for the screenshot path.

Plan accordingly: `DaytonaExecutor.render()` returns `{ screenshots, previewState: {} (no live url), rendered: screenshots.length>0 }`, deletes the sandbox in `finally`, and does NOT call `getSignedPreviewUrl` or bake `serve.js` in v1. Keep `serve.js` + `getSignedPreviewUrl` as a documented follow-up (already in CONTEXT Deferred Ideas).

## Zero-egress Proof (PREV-01 / UAT item 3) — human-verify

Before claiming PREV-01, run a live probe inside a `networkBlockAll:true` sandbox built from the real image and confirm an outbound `fetch` to the Convex URL is **refused**:
```js
// preview/probe.js — executed via process.executeCommand at the human-verify checkpoint
const url = process.argv[2]; // e.g. the Convex deployment URL
fetch(url).then(
  () => { console.error('EGRESS-LEAK: fetch succeeded — zero-egress NOT enforced'); process.exit(2); },
  (e) => { console.log('EGRESS-BLOCKED: ' + e.message); process.exit(0); },
);
```
Checkpoint passes only when `exitCode === 0` (fetch rejected/timed out). This is UAT item 3 from `03-HUMAN-UAT.md`. Pair with a positive control: the `file://` asset render still succeeds (proving the block doesn't break local rendering).

### Known Threat Patterns for untrusted in-sandbox code
| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Generated code exfiltrates Convex/auth tokens | Information disclosure | `networkBlockAll:true` (PREV-01) — proven by the live fetch-refused probe; no auth context ever uploaded |
| Generated code reaches the Lab origin via same-origin iframe | Elevation of privilege | `previewState` omits `sameOrigin` → strict `allow-scripts` only |
