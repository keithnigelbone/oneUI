<decisions>
## Implementation Decisions (LOCKED ‚Äî confirmed from source this session)

### Root-cause findings (why Daytona render fails today)
- The so-called "compiled bundle" (`ctx.bundle` from `compile()` ‚Üí `astToReactComponent`) is **uncompiled TSX source** importing `@oneui/ui` (e.g. `import { Button } from '@oneui/ui'; <Button>‚Ä¶</Button>`). It is NOT a runnable artifact. Nothing can render it without React + ReactDOM + the `@oneui/ui` library + brand CSS + a transpiler.
- `DaytonaExecutor.buildCaptureCommand` runs `node preview/capture.js ‚Ä¶` but **`capture.js` is never created or uploaded** ‚Äî only the bundle is `fs.uploadFile`d. ‚Üí `exit 1` at `captureInBox`.
- `render()` calls `getSignedPreviewUrl(previewPort=3000)` for the live iframe, but **nothing starts a server** on that port in-box.
- The default Daytona sandbox image has **no Node/Playwright/Chromium**; `networkBlockAll: true` forbids runtime install.
- `DaytonaExecutor.test.ts` passes only because `process.executeCommand` is **mocked** to return a fake base64 PNG.
- No `render-bundle` route exists (only `render-ast`, `render-code`).

### Architectural constraints (LOCKED)
- **Seam unchanged (D-02):** the `PreviewExecutor` interface (`render(RenderInput) ‚Üí RenderResult`) stays frozen and vendor-free. Only `DaytonaExecutor` imports `@daytonaio/sdk`.
- **Daytona does NOT own codegen (D-03):** the IR compiler still owns React + Jio-CSS output. Daytona only executes/previews/screenshots. The bundler is a Lab-server step, NOT in-box compilation of the design system.
- **Zero-egress holds at runtime (PREV-01):** Playwright + Chromium + harness are baked into the image at BUILD time (which has network); the sandbox runs with `networkBlockAll: true`. The SDK supports `create({ image: Image })` building a snapshot before the sandbox is network-blocked.
- **Trust-scoped sandbox (from 03-08):** the Daytona/untrusted path's `previewState` omits `sameOrigin`, so the canvas iframe stays strict `allow-scripts` (no `allow-same-origin`). Do not regress this.
- **Opt-in selection:** `run/route.ts` selects `DaytonaExecutor` only when `EXPERIENCE_LAB_PREVIEW_EXECUTOR=daytona` AND `DAYTONA_API_KEY` is set ‚Äî never key-presence alone (that broke every run). Default remains `LabAstExecutor`.
- **SDK API verified (v0.183.0):** `daytona.create({ image, networkBlockAll })`, `sandbox.fs.uploadFile(buffer, path)`, `sandbox.process.executeCommand(cmd) ‚Üí { exitCode, result, artifacts }`, `sandbox.getSignedPreviewUrl(port, ttl)`, `sandbox.delete()` all exist as the executor uses them. `Image.base(...).runCommands(...).addLocalFile(...)` builds custom images.

### Bug to fix alongside
- A preview-executor failure currently surfaces on the card as **"gap ‚Äî generation refused"** (misleading: generation succeeded; only the screenshot/preview step failed). A preview-infra failure must read as a preview error, not a DS refusal.

### Claude's / Researcher's Discretion (gray areas to resolve in RESEARCH.md)
- Bundler choice for the self-contained asset (esbuild vs vite vs tsup) and whether to bundle all of `@oneui/ui` or tree-shake; how brand CSS is inlined (reuse the engine's CSS generation).
- Daytona image strategy: `Image.base('mcr.microsoft.com/playwright:vX-jammy')` vs a Node base + `npx playwright install --with-deps chromium`; how the harness scripts get baked in (`addLocalFile`/`addLocalDir` vs uploaded at runtime).
- In-box harness: a tiny static server (`serve.js`) on the preview port + a Playwright screenshot script (`capture.js`); base64-PNG-over-stdout vs Daytona native screenshot API.
</decisions>

<resolved_decision>
## Key User Decision (RESOLVED 2026-06-02)

- **Live in-box iframe AND screenshots ‚Äî both in scope for v1.** The user chose the live interactive preview path over the research's screenshots-only recommendation. v1 MUST:
  - render + screenshot per profile (PREV-04), AND
  - run an in-box static server on the preview port and mint a signed live preview URL via `getSignedPreviewUrl` so the canvas card can embed a live iframe from the Daytona sandbox (PREV-02).
  - This means the sandbox is kept alive for the preview TTL (not torn down immediately after capture) and the plan must address sandbox lifecycle/teardown-on-expiry + cost/latency.
  - Trust constraint still holds: the Daytona/untrusted iframe stays strict `allow-scripts` (NO `allow-same-origin`) ‚Äî interactivity is intentionally degraded vs the first-party `LabAstExecutor` path; do not relax it (PREV-01).
  - The plan must reconcile the kept-alive sandbox with the `finally { sandbox.delete() }` teardown currently in `DaytonaExecutor.render` (teardown moves to TTL-expiry / `expireSignedPreviewUrl`, not per-render).
</resolved_decision>
