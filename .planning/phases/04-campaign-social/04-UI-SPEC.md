# Phase 4 ‚Äî UI Design Contract

> Visual and interaction contract for the **Campaign / Social** path of the isolated Jio Experience Builder Lab. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Brownfield / continuation note:** This phase builds INSIDE a mature Jio Design System and INSIDE an already-shipped Lab shell. The design language, spacing scale, typography, color contract, and component allowlist are **inherited verbatim from `.planning/phases/01-isolated-foundation/01-UI-SPEC.md` (approved)** and from CLAUDE.md. This spec does **not** re-invent any of that ‚Äî it prescribes **HOW the new Phase-4 UI surfaces consume the existing Jio DS**. The new surfaces are: (1) artifact-type-gated **brief fields** in the existing RequestPanel (D-04), (2) the **campaign-plan HITL card** (brief summary ¬∑ audience ¬∑ message hierarchy ¬∑ 3 creative directions ¬∑ recommendation ¬∑ frame count) with direction-select + resume (D-05/D-06/D-08), (3) the **ordered carousel frame group** on canvas (D-07), and (4) the **export card-action menu** (`‚ãÆ ‚Üí Export ‚ñ∏ Code / PNG / JPG / PDF`) ‚Üí export card + downloadable file (D-13).
>
> **shadcn gate:** NOT APPLICABLE. `components.json` absent, no Tailwind config; CLAUDE.md explicitly forbids Tailwind / shadcn / external visual kits. Tool = none (Jio DS is the design system). Registry safety gate = not applicable (allowlist is the reverse posture ‚Äî Jio-only).
>
> **Scope reminder:** Phase 4 is ~90% backend/orchestration (resolver extension, planner schema, carousel driver, export emitters). The UI deltas are deliberately small and additive (D-04 "no new input card kind"; the campaign-plan card is "minimal additive"). Most of the work is invisible; the visible Lab chrome must stay consistent with the Phase-1 contract.

## Spacing Scale

Inherited verbatim from Phase 1 ‚Äî the Jio f-step modular scale is the source of truth (f0 = 16px base; adapts per platform/density). New Phase-4 CSS uses these tokens, never raw px. (Mirror `RequestPanel.module.css`: panel `gap: var(--Spacing-6)`, field `gap: var(--Spacing-2)`, width `calc(var(--Spacing-40) * 2)`.)

| Role (template) | Jio token | Mobile value | Usage in Phase-4 UI |
|-----------------|-----------|--------------|---------------------|
| xs (icon gaps) | `var(--Spacing-1)` | 4px | Icon-to-label gap inside export menu items, direction-card badge insets |
| sm (compact) | `var(--Spacing-2)` | 8px | Brief-field internals, message-hierarchy list rows, plan-card meta rows |
| md (default) | `var(--Spacing-4)` | 16px | Direction-card body padding, brief-field stacking, plan-card section gaps |
| lg (section) | `var(--Spacing-6)` | 24px | Plan-card outer padding, brief-fields group padding (mirror panel padding) |
| xl (layout gaps) | `var(--Spacing-8)` | 32px | Gap between plan-card sections (brief ‚Üí hierarchy ‚Üí directions) |
| 2xl (major breaks) | `var(--Spacing-12)` | 48px | Vertical rhythm between stacked carousel frames if a column hint is shown |
| 3xl (page-level) | `var(--Spacing-16)` | 64px | Page-level margins where `--Spacing-Margin` is not used |
| page margins/gutters | `var(--Spacing-Margin)` / `var(--Spacing-Gutter)` | platform-adaptive | Lab page chrome, panel docking offsets |
| carousel safe-area insets | **foundation-resolved** `Spacing-N` token from the resolver (D-03) | per-canvas | **Inside generated frames** ‚Äî the safe-area inset is a resolved `Spacing-N` token, NEVER a raw px margin. A platform-mandated margin with no matching token is a **gap**, not a magic number. |

**Exceptions:**
- **Touch targets:** interactive controls keep the Jio minimum (44px mobile / 24px desktop), provided automatically by `@oneui/ui`. The export `‚ãÆ` IconButton and direction-select buttons must meet this.
- **tldraw canvas internals:** tldraw owns shape geometry in canvas coordinate space (exempt). The HTML inside each shape's `HTMLContainer` (plan-card chrome, carousel-frame chrome, export card) is **NOT exempt** ‚Äî Jio tokens only.
- **Carousel frame ordering geometry** (left-to-right `orderIndex` layout, D-07) is tldraw canvas-coordinate math, not Lab CSS ‚Äî exempt from the token rule, like the Phase-3 variant-group frame.
- **Generated-artifact dimensions** (e.g. 1080√ó1080 IG, A4 print) come from the **foundation resolver** at runtime (D-01) ‚Äî never authored as literals in Lab CSS (Pitfall 1 / Pitfall 6).

## Typography

Inherited verbatim from Phase 1 ‚Äî **4 distinct type sizes (28 / 16 / 14 / 12px)** across 5 active roles + 1 mono role. Every text element pairs a size token with its line-height token and sets `font-family: var(--Typography-Font-Primary)` (CLAUDE.md MANDATORY rule). **No new sizes or weights are introduced in Phase 4.**

**Font-weight policy (unchanged from Phase 1):**
- **Lab-authored content text (Body / Label / Code) uses exactly TWO chosen weights:** Medium (500) default for controls/labels/meta; Low (400) for multi-line body / summaries / gap descriptions.
- **Heading roles (Headline-L, Title-M) carry FIXED Jio DS role-token weights** (900 / 800) ‚Äî system metadata, not a free Lab choice; each used once per surface.

| Role | Where used in Phase-4 UI | Size token | Weight token | Line-height token |
|------|--------------------------|------------|--------------|-------------------|
| Section heading *(fixed DS weight)* | Plan-card title ("Campaign plan"), export card title, brief-fields group title | `--Title-M-FontSize` (f0 ‚âà 16px) | `--Title-M-FontWeight` (800, fixed) | `--Title-M-LineHeight` |
| Body / summary *(Low ‚Äî content weight)* | Brief summary, direction concept/copy-angle text, gap-report body, export status body | `--Body-S-FontSize` (f-1 ‚âà 14px) | `--Body-FontWeight-Low` (400) | `--Body-S-LineHeight` |
| UI label / control text *(Medium ‚Äî content weight)* | Brief-field labels (Audience / Objective / Channel), direction name, button labels, export menu item labels, frame-count stepper | `--Label-S-FontSize` (f-1 ‚âà 14px) | `--Label-FontWeight-Medium` (500) | `--Label-S-LineHeight` |
| Micro / meta *(Low ‚Äî content weight)* | Direction-role/surface badges, frame `orderIndex` labels ("Frame 1 of 5"), recommended-direction marker, export format meta | `--Label-XS-FontSize` (f-2 ‚âà 12px) | `--Label-FontWeight-Low` (400) | `--Label-XS-LineHeight` |
| Code (export preview) *(Medium ‚Äî content weight)* | Code-export TSX/CSS preview block on the export card | `--Code-M-FontSize` (f0 ‚âà 16px) | `--Code` Medium (500) | `--Code-M-LineHeight` |

**Rules (binding):**
- Never use legacy `--Typography-Size-*` / `--Typography-Weight-*` in new Lab code (CLAUDE.md).
- Plan-card direction names, export menu items, frame labels use the **Label** role ‚Äî never Display/Headline.
- **Display role is not used** in Phase 4 Lab chrome (no hero/marketing surface in the tool). Display IS available to the *generated campaign artifacts themselves* via the IR/compiler ‚Äî but that is generated output governed by the resolver/foundations, not Lab chrome.
- Headline-L is **not** used on the plan card; the plan card is a working card (Title-M heading), consistent with the artifact/run-inspector cards.

## Color

Inherited verbatim from Phase 1 ‚Äî **White Canvas Rule + Attention Pyramid**: ‚âà80% `default` surface, brand accent reserved for the single primary action per surface. Color flows through **surface tokens + role tokens** that remap per `[data-surface]` context ‚Äî Lab CSS references generic role tokens, never hard-coded hex.

| Role | Jio token / mechanism | Usage in Phase-4 UI |
|------|----------------------|---------------------|
| Dominant (‚âà80%) | `default` surface (`var(--Surface-Main)`) | Lab page, canvas, RequestPanel (incl. revealed brief fields), card backgrounds |
| Secondary (‚âà15%) | `<Surface mode="subtle">` / `var(--Surface-Fill-Subtle)` | Campaign-plan **card body**, each **creative-direction tile** inside the plan card, the **export card** body, carousel-frame group fill ("Carousel: {name}") |
| Accent (‚âà5%) | `--Primary-*` role via `<Button variant="bold" appearance="primary">` | **Reserved for: (a) the single "Run generation" CTA in RequestPanel; (b) the single "Generate frames" confirm CTA on the plan card after a direction is chosen.** One bold primary per surface ‚Äî no second accent button. |
| Secondary accent | `--Secondary-*` role | Selected direction tile's selected state; the brief-field Selects (Channel) and any Chip/Toggle, per design-composition (Checkbox/Radio/Switch/Chip default to `secondary`) |
| Recommended marker | `--Informative-*` role (Badge) | The planner's **recommended** direction gets an `appearance="informative"` "Recommended" badge ‚Äî a neutral signal, NOT the bold accent (recommendation ‚âÝ the user's choice) |
| Selected-direction marker | `--Primary-*` (subtle/ghost selected state) | The direction the **user** picks gets the primary-role selected treatment (border/fill), distinct from the informative "Recommended" badge |
| Status: warning / gap | `--Warning-*` role (Badge) | **Non-web foundation-coverage gap** badge on the foundation-profile card ("Foundation gap ‚Äî no canvas defined") when the brand's Platforms foundation lacks the requested non-web canvas (D-02 / FND-03) |
| Status: success / valid | `--Positive-*` role (Badge/pill) | Per-frame "Valid IR" pill after a frame passes the quality loop; "Export ready" pill on the export card after bytes are stored |
| Status: in-progress | `--Informative-*` / `--Neutral-*` | "Planning‚Ä¶", "Generating frame N of M‚Ä¶", "Exporting‚Ä¶" run-inspector states; neutral chrome (export `‚ãÆ` IconButton, dividers, frame outlines) |
| Destructive | `--Negative-*` role | **Not triggered by a user delete action in Phase 4.** Used only for a per-frame **validation/repair-failure** status indicator (problem signalling) when a frame can't be made DS-compliant within the shared budget. No delete/clear destructive UI ships in P4. |

**Accent reserved for (explicit list ‚Äî never "all interactive elements"):**
1. The single **"Run generation"** primary button in the RequestPanel (one per panel).
2. The single **"Generate frames"** primary button on the campaign-plan card ‚Äî enabled only after a direction is selected (D-05: frame generation happens only after selection).
3. The **user-selected** creative-direction tile's selected state (the picked direction, not the recommended one).
4. Focus halo on interactive controls (`--Focus-Outline` + `--Surface-Halo-Gap`, per CLAUDE.md Focus Halo pattern).

Everything else ‚Äî brief-field chrome, plan-card body, direction tiles (unselected), export menu/card chrome, carousel frame outlines, badges/pills ‚Äî stays `default` / `subtle` / semantic-status / `neutral`. **No `bg-bold` / `fg-bold` surface in the Lab chrome** (the Lab is a working tool). `bold` surfaces belong only to the *generated campaign artifacts*, driven by the resolver/foundations.

## Interaction Contract

| Interaction | Contract |
|-------------|----------|
| Reveal brief fields | When the prompt card's artifact type becomes `social-post` / `instagram-carousel`, the RequestPanel progressively reveals the "Campaign brief" group (Audience / Objective / Channel). | Resume ‚Üí generate frames | The selection POSTs and **resumes** the workflow; frames generate **only after** selection.
