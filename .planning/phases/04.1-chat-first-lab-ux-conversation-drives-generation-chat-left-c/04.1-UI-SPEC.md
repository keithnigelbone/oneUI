# Phase 04.1 — UI Design Contract

> Visual and interaction contract for the Chat-first Lab UX restructuring. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Brownfield UX restructuring** of the isolated Experience Builder Lab — canvas-centric → chat-first (chat LEFT, tldraw canvas RIGHT, inline interactive campaign-plan card). No generation-logic change; the engine, run/resume routes, IR, validators, repair loop, and Phase-4 suspend/resume contract are reused **verbatim as read-only contracts**.
>
> **REUSE-FIRST:** Every chat surface maps onto the existing Jio DS `ChatSurface` + `ChatComposer` family (`packages/ui/src/components/ChatSurface`, `ChatComposer`) and their slots. Net-new is limited to: the left/right shell (D-13), the composer control strip composition (D-07, lifting RequestPanel selectors), two `renderMessagePart` parts (run turn + campaign-plan card), the inline-card framework dispatch (D-09), and the chat↔canvas selection binding (D-02). **Do not invent chat primitives.**

## Spacing Scale

Token-only — `--Spacing-*` f-scale tokens ONLY (no literal px except `0`, `9999px`, `100%`). The Lab route is scanned by `pnpm check:literals`. The 8-point-equivalent values below are the f-step Mobile defaults for reference; **the executor writes the token, never the px**.

| Token | Mobile default | Usage in this phase |
|-------|----------------|---------------------|
| `--Spacing-5XS` (f-6) | 4px | Icon→text gap inside selectors, badge internal gap, step-ticker dot gap |
| `--Spacing-3XS` (f-4) | 8px | Gap between control-strip selects, suggestion-chip gap, message-part internal rows |
| `--Spacing-2XS` (f-3) | 10px | Compact selector padding |
| `--Spacing-M` (f0) | 16px | Standard padding inside the inline card, run-turn body, composer padding |
| `--Spacing-L`–`--Spacing-2XL` | 18–24px | Gap between message turns, between card sections (brief / audience / directions) |
| `--Spacing-3XL`–`--Spacing-4XL` | 28–32px | Gap between the chat pane and the canvas pane, around the empty-state greeting |
| `--Spacing-Margin` / `--Spacing-Gutter` | platform-adaptive | Chat pane outer margin; gutter between the 3 creative-direction tiles |

**Layout split (D-13):** left chat + right canvas via CSS grid using token-based track sizing. Recommendation: chat pane `minmax(360px, 30%)` / canvas pane `1fr` on L ; widths must derive from f-step / `%` tokens, not literal px. The `ssr:false` dynamic canvas import + `PageLoader` fallback is preserved; the chat pane lives in the same `'use client'` shell.

**Exceptions:** Touch targets follow the platform minimum — `44px` mobile / `24px` desktop minimum (design-system constant, satisfied by `@oneui/ui` interactive components; do not author raw literals — the components own this). The `9999px` pill constant and `100%`/`0` are the only permitted literals (per `check:literals` allowlist).

## Typography

**Typography V2 (MANDATORY).** Every text element uses a role-specific size token PAIRED with its matching line-height token and a weight token, plus `font-family: var(--Typography-Font-Primary)`. NEVER legacy `--Typography-Size-*` / `--Typography-Weight-*`. The reference px values are the f-step Mobile defaults — the executor writes the token.

| Role (this phase) | Size token | Weight token | Line-height token | px (ref) | Used for |
|-------------------|------------|--------------|-------------------|----------|----------|
| Body | `--Body-M-FontSize` | `--Body-FontWeight-Low` (400) | `--Body-M-LineHeight` | 16 | Chat message text, plan-card brief/audience copy, selector option labels |
| Small body / caption | `--Body-S-FontSize` | `--Body-FontWeight-Low` (400) | `--Body-S-LineHeight` | 14 | Run-turn step descriptions, helper text, focused-artifact caption, timestamps |
| UI label | `--Label-S-FontSize` | `--Label-FontWeight-Medium` (500) | `--Label-S-LineHeight` | 14 | Selector field labels, button labels, chip labels, tab/section labels |
| Micro label / meta | `--Label-XS-FontSize` | `--Label-FontWeight-Low` (400) | `--Label-XS-LineHeight` | 12 | Badge text, step-ticker status tags, frame-count meta |
| Card title | `--Title-M-FontSize` | `--Title-M-FontWeight` (800) | `--Title-M-LineHeight` | 16 | Inline campaign-plan card heading, run-turn summary heading, creative-direction tile titles |
| Empty-state greeting | `--Headline-S-FontSize` | `--Headline-S-FontWeight` (850) | `--Headline-S-LineHeight` | 16 | First-run greeting headline in `ChatSurface.greeting` |
| Code (event log) | `--Code-S-FontSize` | `--Code-FontWeight-Medium` (500) | `--Code-S-LineHeight` | 14 | Expandable `RunInspectorPanel` event-log rows (monospace) — `font-family: var(--Typography-Font-Code)` |

**Rules carried in (CLAUDE.md):** never a font-size without its paired line-height; never a literal weight; always include `font-family: var(--Typography-Font-Primary)` (or `-Code` for the event log). Most of this is satisfied for free because `ChatSurface`/`ChatComposer`/`Select`/`Button`/`Badge` already use V2 tokens — the net-new wrappers (`RunTurn`, `CampaignPlanCard`, `ComposerControlStrip`, shell) MUST follow the table above.

## Color

Color is expressed through Jio **surface modes + role tokens**, never hex. The 60/30/10 split maps onto the Jio Surface system and the Attention Pyramid (60% None / 25% Low / 10% Medium / 5% High). **Surface Context is MANDATORY:** any non-default background uses `<Surface mode="…">`, never a raw `<div style={{background}}>`. No decorative borders/strokes on tinted Surfaces.

| Role | Jio expression | Usage in this phase |
|------|----------------|---------------------|
| Dominant (60%) | `<Surface mode="default">` (page surface) | Both panes (chat transcript + canvas), composer field, message text — the white canvas. Headers/transcript stay `default`. |
| Secondary (30%) | `<Surface mode="minimal">` / `mode="subtle">` | Assistant run-turn container, inline campaign-plan card body, composer control-strip background, focused-artifact banner. `minimal` for the run-turn/strip; `subtle` for the grouped plan card. |
| Accent (10%) | `--Primary-*` role (`Button variant="bold" appearance="primary"`) | RESERVED — see list below. |
| Destructive | `--Negative-*` role (`Badge`/`Button appearance="negative"`) | Real run failure (D-12) only — error headline badge; no destructive *buttons* in this phase (no delete/discard flow). |

**Accent (Primary) reserved exclusively for:**
1. The composer **Send / Run** trigger — the single `Button variant="bold" appearance="primary"` per the composer (one focal action). Inherited from `RequestPanel`'s "one accent moment" rule.
2. The inline campaign-plan card's **"Generate frames"** confirm `Button variant="bold" appearance="primary"` (the resume action — one per card).
3. The **recommended** creative-direction tile's selected/recommended marker (a single `Badge appearance="primary"` or selected `RadioGroup` state).

Everything else is None/Low: selectors are `Select` (neutral chrome), secondary actions are `Button variant="subtle"/"ghost" appearance="neutral"`, the focused-artifact indicator is a neutral `Chip`/`Badge`.

**Semantic roles for run status (reuse `describeEvent` mapping verbatim):**

| Run event state | Role token | Component |
|-----------------|------------|-----------|
| started / in-progress step | `informative` | `Badge appearance="informative"` |
| completed step / IR produced / validation passed / artifact outcome | `positive` | `Badge appearance="positive"` |
| validation blocked / real failure outcome | `negative` | `Badge appearance="negative"` |
| foundation/component gap (refusal) | `warning` | `Badge appearance="warning"` |

**Pitfall (locked, D-12 / CHAT-06):** a `gap` result frame **carrying `campaignPlan`** is a *suspend*, not a failure — render the inline `CampaignPlanCard`, NOT the negative/warning headline. Only a `gap` with no plan + a gap event is a real refusal (→ warning headline + canvas gap card).

Optional role-fill override pattern for a role-tinted card (e.g. the plan card leaning secondary) is allowed via inline `style={{ '--Surface-Fill-Subtle': 'var(--Secondary-Subtle)' }}` on the `<Surface>` (per design-composition skill) — still token-only.

## Interaction & Accessibility (WCAG AA — mandatory)

| Concern | Contract |
|---------|----------|
| Focus Halo | All net-new interactive elements (direction tiles, control-strip selects, send/confirm buttons) draw the 2-layer focus indicator: inner gap via `var(--Surface-Halo-Gap, var(--Surface-Main))` + outer `var(--Focus-Outline)` at `var(--Focus-Outline-Width)`. Reuse `@oneui/ui` component focus styles where the component owns it; net-new wrappers must not use `--Surface-Main` directly. |
| Streaming live region (D-11) | The run-turn step ticker and `StreamingText` update inside an `aria-live="polite"` region so screen readers announce step progress (resolve→…→repair) without stealing focus. Terminal summary announced once; full event log behind an expandable is NOT live (avoids verbosity). |
| Run failure / gap (D-12) | The inline error/gap headline is announced via `aria-live="assertive"` (it is an actionable terminal state). |
| Keyboard nav | Composer textarea: Enter submits, Shift+Enter newline (ChatComposer default). Control-strip selectors, suggestion chips, regenerate/stop, and the inline card (direction `RadioGroup`, frame-count `Stepper`, confirm button) are fully keyboard-operable in DOM order. |
| Focus management (inline card, D-08/D-09) | When the campaign-plan card mounts in the stream, move focus to the card's heading (or first interactive control) so keyboard users land on the HITL checkpoint; after "Generate frames" resumes, return focus to the composer. The card is a labelled region (`role="group"` + `aria-labelledby` the card title). |
| Focused-artifact indicator (D-02) | Visible (neutral `Chip`/`Badge` in `aboveComposer`) AND programmatic — selecting a canvas artifact sets chat focus, scrolls to its originating turn, and the focus state is announced. Reverse: a turn's "focus on canvas" action calls `editor.select` + `zoomToSelection`. |
| Contrast | All text/role pairings inherit WCAG-AA-compliant Jio role tokens (`--{Role}-High`/`-TintedA11y`/`-Bold-High`). No custom colors. |
| Reduced motion | `AgentThinking`/`StreamingText` animations respect the system motion tokens; no new bespoke animation. |
| Stop / regenerate | `ChatSurface.onStop` (streaming) + `onRegenerate` (assistant turns) are wired to the NDJSON run controller (abort + re-run last prompt against the focused artifact). |

## Responsive / First-run (planner-surface recommendations)

CONTEXT flagged these as discretion — recommendations locked here:

- **Desktop / landscape (L):** side-by-side — chat pane `minmax(360px, 30%)`, canvas `1fr`. Default working layout.
- **Tablet (M):** side-by-side allowed; chat pane may collapse to ~40% with a drag/toggle (use a `Surface`-based splitter, no literal widths).
- **Mobile (S):** stack — chat is primary (full width); canvas behind a toggle/tab (`Tabs` or a `Button`-driven view switch), since tldraw spatial interaction is poor on small touch targets. Chat-first is the natural mobile mode.
- **First-run / empty state:** `ChatSurface` empty layout with the greeting + suggestion chips above; the composer control strip is visible and pre-focusable so the user sets brand/type before typing. Canvas pane shows an empty tldraw board (no artifacts yet) — no separate empty illustration needed.
- **Reload-safety (D-06):** on reopen, the Mastra-memory thread rehydrates `messages[]` and linked Convex artifacts re-place on the canvas; the transcript scroll position resets to the latest turn.
