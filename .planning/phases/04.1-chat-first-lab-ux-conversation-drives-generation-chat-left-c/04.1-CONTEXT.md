# Phase 04.1: Chat-first Lab UX - Context

**Gathered:** 2026-06-03
**Status:** Ready for planning

<domain>
## Phase Boundary

Reframe the **interaction model** of the isolated Experience Builder Lab from canvas-centric to **chat-first**. Today the Lab is driven by an on-canvas `PromptCardShape` edited through a right-docked `RequestPanel` + `RunInspectorPanel`. After this phase:

- A **persistent chat conversation on the left drives generation** — prompt entry, brand/type/profile selection, run triggering, iterate, and the HITL plan checkpoint all happen in chat.
- The **tldraw canvas on the right displays the generated UIs** (and retains informational shapes: foundation-profile, component-reference, gap-report cards).
- The **campaign-plan HITL renders as an inline interactive card** inside the chat stream.
- Everything is built from **Jio DS components + Jio CSS only** (LAB-02 token-only).
- The existing Mastra workflow / IR / compiler / validators / repair / run route / foundation engine are **reused as read-only contracts** — no changes to generation logic, IR ownership, or the repair-on-IR rule.
- **Lab isolation is preserved** — no modification to or dependency on the `(builder)` / `ExperienceCanvas` feature.

**This is a UX restructuring, not a generation-capability change.** The generation backend (IR, compiler, validators, repair, run route, Convex `experience*` tables) stays. We are clarifying *how the user interacts*.

**Requirements:** The ROADMAP entry lists `Requirements: TBD`. The decisions below + the phase title are the de-facto spec for this phase; the planner should derive concrete requirement IDs (e.g. a `CHAT-*` / `UX-*` family) from these decisions, or confirm with the user during planning.

**Out of scope:**
- Any change to IR/compiler codegen ownership, the validator allowlists, or the bounded-repair contract — chat is a new front-end over the **existing** generate→compile→validate→evaluate→repair loop.
- Real-time multiplayer / collaboration on the chat or canvas (Phase 5).
- Observability / cost tracking surfaced in chat (Phase 5).
- Modifying the legacy `(builder)/create` campaign feature or `ExperienceCanvas` (hard isolation rule).
- New generation modalities or output profiles beyond what Phases 1–4 already deliver.

</domain>

<decisions>
## Implementation Decisions

### Chat ↔ canvas split
- **D-01:** **Chat fully replaces the on-canvas prompt card.** The `PromptCardShape` is retired as the entry surface; all prompt entry + brand/sub-brand/artifact-type/output-profile selection + run triggering move into the chat composer. The canvas becomes **display-only** for generated artifacts (plus the info shapes in D-04).
- **D-02:** **Canvas selection scopes the chat (two-way link).** Selecting an artifact card on the canvas focuses the conversation on that artifact — follow-up messages ("make the CTA bigger", "iterate", "make a variant") target the focused artifact. This is the chat-first realization of the existing Phase-3 iterate-on-artifact flow. Chat must visibly indicate which artifact is in focus.
- **D-03:** **Auto-arrange artifacts, user can still rearrange.** Each run drops its artifact(s) into an auto-computed canvas position, reusing the existing Run #N / variant / carousel frame placement logic. The user retains tldraw spatial freedom to drag/zoom/organize afterward. Chat owns creation; canvas owns arrangement.
- **D-04:** **Informational shapes stay on the canvas.** Foundation-profile, component-reference, and gap-report cards remain canvas shapes; chat triggers them. (The campaign-plan HITL is the deliberate exception — it goes inline, see D-08.) Net effect: canvas = rendered artifacts **+ reference/report shapes**; chat = conversation, run progress, and the interactive plan card.

### Conversation model & state
- **D-05:** **One linear thread per session.** A single chronological conversation; every run / iterate / variant / carousel is a turn in the same thread. Selecting a canvas artifact (D-02) focuses/scrolls to its originating turn.
- **D-06:** **Persist chat via Mastra memory (user decision).** Use Mastra's memory subsystem (threads/resources + storage adapter) as the conversation store rather than a new Convex chat table. Artifacts/versions/exports continue to persist in Convex `experience*` tables (unchanged). **Boundary note — load-bearing:** this consciously pulls a slice of the Phase-5 "durable workflow / Mastra storage adapter" work forward and introduces a Mastra-managed persistence layer alongside Convex. The researcher MUST resolve: (a) Mastra memory ↔ pinned `@mastra/core@1.37.1` / `@mastra/ai-sdk@1.4.3` / `ai@6` compatibility (the same pin-compatibility discipline used since Phase 1); (b) how a Mastra `thread`/`resource` maps to a Lab session and to the focused-artifact lineage; (c) keeping the agents package's backend-free posture vs. wiring a storage adapter; (d) reload-safety (reopen → thread + linked artifacts restore).
- **D-07:** **Generation controls live in a composer control strip.** Brand / sub-brand / artifact-type / output-profile selectors become compact controls attached to the chat input (a strip above/around the text box), set once and reused for subsequent turns until changed. Keeps generation context visible at the point of entry. These reuse the existing selector logic from `RequestPanel` (Convex brand/sub-brand queries, profile/type enums).

### Inline interactive cards
- **D-08:** **Campaign-plan HITL renders inline and interactive in chat** (the named deliverable). The card shows brief summary + audience + message hierarchy + the 3 creative directions + the recommended one; direction selection happens inline and resumes the suspended Mastra workflow (Phase-4 D-05 suspend/resume). Frame generation happens only after inline selection. Frame-count adjustment (Phase-4 D-08) is part of this inline card.
- **D-09:** **Build a reusable inline-card framework.** A small general pattern: Jio-DS components rendered in the chat stream that can post an action back to resume/continue the workflow. The campaign-plan card is the first consumer; future HITL / confirm moments reuse the same pattern. Designed upfront to avoid a refactor when the second inline card arrives.
- **D-10:** **Reuse the suspend/resume contract; new inline view.** Keep the Phase-4 suspend/resume + plan-data contract + resume route **untouched (read-only contract)**. Build a net-new Jio-DS inline chat component that renders the same plan data and posts the same selection back. **Retire/hide the canvas plan-card shape** (its rendering, not its contract). Honors "engine reused as read-only contracts."

### Run feedback & streaming
- **D-11:** **Run progress is a streaming assistant turn.** Each run is an assistant message that streams live step progress as `ExperienceBuilderEvent`s arrive (resolve → retrieve → plan → generate → compile → validate → evaluate → repair), updating a step ticker / status line in place and collapsing to a compact summary when done. The full event detail is **expandable** within the turn (this is where `RunInspectorPanel`'s content goes).
- **D-12:** **Failures: inline headline + canvas gap card.** A run error or a foundation/component gap ("generation refused") flips the streaming turn to a readable inline error/gap state (e.g. "gap — generation refused: missing IG-portrait dimensions for brand X"), AND the detailed gap-report card lands on the canvas (consistent with D-04). Chat = headline, canvas = full report.
- **D-13:** **Both docked panels retired into chat.** `RequestPanel` → chat composer + control strip (D-07); `RunInspectorPanel` → the streaming assistant turn's expandable detail (D-11). The right dock is removed; the Lab layout becomes **left chat + right canvas**.

### Claude's / Researcher's Discretion
- The concrete requirement IDs for this phase (ROADMAP says TBD) — derive a `CHAT-*`/`UX-*` family from these decisions during planning, or confirm with the user.
- Exact Mastra-memory integration shape (storage adapter choice, thread/resource modeling, where it lives relative to the backend-free agents package) — pending the D-06 research resolution; fall back to a Convex chat table only if Mastra memory proves incompatible with the pinned stack.
- Composer layout specifics (strip above vs beside the input; how sub-brand dependency/skip-gating from Phase 02.1 surfaces in the strip).
- How much rendering (if any) is shared between the retired canvas plan-card shape and the new inline card vs. fully net-new (D-10 locks the contract as read-only and inline as the target).
- Responsive/mobile behavior of the chat + canvas split, empty/first-run state, and chat accessibility (WCAG AA applies) — not discussed; standard Jio-DS approaches expected unless the planner surfaces a decision.
- The exact `ExperienceBuilderEvent` → streaming-turn rendering granularity (per-step summary vs full event log in the expandable detail).

</decisions>

- **Info card shapes** (`FoundationProfileCardShape`, `ComponentReferenceCardShape`, `GenericPlaceholderShape`): kept on canvas (D-04); gap card is the failure-detail surface (D-12).

### Established Patterns
- **Mastra owns orchestration; Vercel AI SDK is model-only** via the single `modelAdapter` seam — unchanged. Chat is a front-end over the existing workflow.
- **IR is the only source of truth; repair patches IR, never JSX** — unchanged. Chat introduces no IR fields.
- **Append-only Convex persistence** for `experience*` artifacts/versions/exports — unchanged; D-06 adds Mastra memory for the conversation only.
- **Isolated route, never importing `(builder)`/`ExperienceCanvas`** (LAB-03, eslint boundary) — the chat panel must respect this.
- **Token-only Jio-DS UI** (LAB-02, check-literals scan over the Lab route) — applies to the chat panel, composer, and inline cards.
- **`ssr:false` dynamic canvas import** with a `PageLoader` fallback — the chat panel coexists in this client shell.

### Integration Points
- **Chat composer → run route** (`api/experience-lab/run`) — replaces the RequestPanel "Run" path (D-01/D-07).
- **Inline plan card → resume route** — posts the selected direction to resume the suspended workflow (D-08/D-10).
- **Streaming assistant turn ← `ExperienceBuilderEvent` stream** — replaces the RunInspectorPanel feed (D-11).
- **Canvas focus ↔ chat focus** — a new two-way selection binding between the tldraw editor and the chat thread (D-02).
- **Mastra memory ← conversation turns** — new persistence wiring to resolve in research (D-06).
