## Chat ↔ canvas split

### What happens to the on-canvas prompt card?
| Option | Description | Selected |
|--------|-------------|----------|
| Chat fully replaces it | Prompt card retired; all entry + selectors + run move into chat composer; canvas display-only | ✓ |
| Chat is primary, card optional | Chat main driver, card still droppable for power users | |
| Canvas-only stays for non-chat | Keep card for web; chat only for campaign flows | |

**User's choice:** Chat fully replaces it.

### When the user clicks an artifact on the canvas, how should chat respond?
| Option | Description | Selected |
|--------|-------------|----------|
| Canvas selection scopes chat | Selecting an artifact focuses the conversation on it; follow-ups target it (two-way link) | ✓ |
| Chat stays global, canvas is a viewer | Click just pans/zooms; loose coupling | |
| Canvas card actions feed chat | Action buttons inject chat turns; selection itself doesn't scope | |

**User's choice:** Canvas selection scopes chat.

### How should generated artifacts be laid out?
| Option | Description | Selected |
|--------|-------------|----------|
| Auto-arrange, user can rearrange | Auto position via existing Run/variant/carousel frames; user can still drag/zoom | ✓ |
| Strict auto-gallery | Managed grid/timeline, minimal manual rearrange | |
| Focus-one, history in chat | Active artifact large; prior ones in chat history | |

**User's choice:** Auto-arrange, user can rearrange.

### Where do foundation/component/gap info cards belong?
| Option | Description | Selected |
|--------|-------------|----------|
| Move to chat as inline cards | Info outputs render inline; canvas purely rendered artifacts | |
| Keep on canvas | Remain canvas shapes; chat only triggers them | ✓ |
| Split by type | Gap inline, foundation/component on canvas | |

**User's choice:** Keep on canvas. **Notes:** Phase title's "inline interactive plan card" is a deliberate exception (campaign-plan HITL), confirmed in Inline cards area.

## Conversation model & state

### How should the conversation be structured?
| Option | Description | Selected |
|--------|-------------|----------|
| One linear thread per session | Single chronological thread; all runs/iterates/variants/carousels are turns | ✓ |
| Thread per artifact lineage | Each artifact its own thread | |
| Linear thread + focus filter | One thread, filtered view when artifact focused | |

**User's choice:** One linear thread per session.

### How should chat history be persisted?
| Option | Description | Selected |
|--------|-------------|----------|
| Convex now, Mastra memory deferred | Append-only Convex chat turns; Mastra memory as P5 option (recommended) | |
| Use Mastra memory now | Adopt @mastra/memory + storage adapter this phase | ✓ |
| Let research decide | Leave open for researcher to compare | |

**User's choice:** Use Mastra memory now (chose "Other" first — "Mastra, I believe, has memory as well so probably we could use from there or otherwise we use Convex"; then confirmed Mastra memory after the P5-deferral tradeoff was surfaced). **Notes:** Consciously pulls a slice of P5 durable-storage work forward; introduces Mastra persistence alongside Convex. Convex is the documented fallback if pinned-stack compatibility fails. Researcher must confirm Mastra memory ↔ pinned `@mastra/core@1.37.1`/`ai@6` compatibility + thread/resource→session mapping.

### Where do the generation selectors live now?
| Option | Description | Selected |
|--------|-------------|----------|
| Composer control strip | Compact selectors attached to chat input, set once, reused until changed | ✓ |
| Settings panel / drawer | Selectors in a dedicated panel; composer is just text | |
| Inferred + inline confirm | Infer from message, inline confirm card | |

**User's choice:** Composer control strip.

## Run feedback & streaming

| Persistent run-status strip | Dedicated status area separate from transcript | |
