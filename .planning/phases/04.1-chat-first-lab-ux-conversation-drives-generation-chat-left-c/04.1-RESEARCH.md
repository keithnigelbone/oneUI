# Phase 04.1: Chat-first Lab UX - Research

**Researched:** 2026-06-03
**Domain:** Brownfield UX restructuring of an isolated tldraw + Mastra Lab — chat-first React UI, Mastra memory persistence, inline interactive HITL cards, NDJSON streaming run feedback. NOT a generation-capability change.
**Confidence:** HIGH (existing contracts read directly from source; Mastra memory verified against installed packages + current docs; a purpose-built Jio-DS `ChatSurface`/`ChatComposer` already exists in the repo).

## Summary

This phase reframes the *interaction model* of the existing isolated Experience Lab from canvas-centric (an on-canvas `PromptCardShape` edited via a right dock) to chat-first (a persistent left-hand conversation drives generation; the right-hand tldraw canvas displays generated artifacts). **No generation logic changes** — the IR/compiler/validators/repair/run route/resume route/foundation engine and the Phase-4 suspend/resume HITL contract are all reused verbatim as read-only contracts. The work is almost entirely a new front-end over the *existing* `/api/experience-lab/run` (web) and `/api/experience-lab/resume` (campaign) NDJSON endpoints plus a new Mastra-memory-backed conversation store.

The single highest-leverage discovery is that **the Jio Design System already ships a purpose-built `ChatSurface` + `ChatComposer` component family** (`packages/ui/src/components/ChatSurface`, `ChatComposer`) with a `renderMessagePart` slot, `AgentThinking` step-ticker, `StreamingText`, `onStop`/`onRegenerate`, suggestion chips, and composer adornment slots (`leading`/`leadingInline`/`trailing`/`modelLabel`). Every decision in CONTEXT.md (chat transcript, streaming run turn, inline interactive card framework, composer control strip) maps directly onto this component's existing API. The phase should *consume* `ChatSurface`/`ChatComposer`, not build a chat UI from primitives. This satisfies LAB-02 (token-only Jio-DS) by construction.

The second load-bearing finding (D-06): **`@mastra/memory` is compatible with the pinned stack** — `@mastra/memory@1.20.1` peer-requires `@mastra/core >=1.4.1-0 <2.0.0-0`, satisfied by the pinned `@mastra/core@1.37.1`. Moreover `@mastra/core` itself already ships `InMemoryStore`/`MockStore` and `FilesystemStore` (via `@mastra/core/storage`) and a `MastraMemory` base (via `@mastra/core/memory`) — so a basic durable conversation store needs **no new vendor package** beyond optionally `@mastra/memory` for the high-level `Memory` class. The Lab uses a deterministic workflow runner (not a Mastra `Agent`), so memory is used as a plain thread/message store (`createThread`/`saveMessages`/`query`), not as agent auto-recall.

**Primary recommendation:** Build the chat panel on `@oneui/ui` `ChatSurface` + `ChatComposer`; render the campaign-plan HITL and the streaming run as `renderMessagePart` custom data-parts (the inline-card framework D-09); reuse `useExperienceLabRun`/`runStream.ts` NDJSON consumer untouched as the transport (do NOT adopt `ai`'s `useChat` — that would breach the Lab-scoped single-`ai` gate and the NDJSON contract); persist the conversation with a thin `@mastra/core/storage` `FilesystemStore`-or-`InMemoryStore`-backed memory layer owned in the **app/route layer**, keeping `packages/experience-builder-agents` backend-free.

## User Constraints (from CONTEXT.md)

### Locked Decisions

**Chat ↔ canvas split**
- **D-01:** Chat fully replaces the on-canvas prompt card. `PromptCardShape` retired as the entry surface; all prompt entry + brand/sub-brand/artifact-type/output-profile selection + run triggering move into the chat composer. Canvas becomes display-only for generated artifacts (plus info shapes in D-04).
- **D-02:** Canvas selection scopes the chat (two-way link). Selecting an artifact card on the canvas focuses the conversation on that artifact; follow-up messages target the focused artifact (chat-first realization of the Phase-3 iterate-on-artifact flow). Chat must visibly indicate which artifact is in focus.
- **D-03:** Auto-arrange artifacts, user can still rearrange. Each run drops artifact(s) into auto-computed canvas positions, reusing existing Run #N / variant / carousel frame placement logic. User retains tldraw spatial freedom. Chat owns creation; canvas owns arrangement.
- **D-04:** Informational shapes stay on the canvas. Foundation-profile, component-reference, gap-report cards remain canvas shapes; chat triggers them. (Campaign-plan HITL is the deliberate exception → inline, D-08.) Net: canvas = rendered artifacts + reference/report shapes; chat = conversation, run progress, interactive plan card.

**Conversation model & state**
- **D-05:** One linear thread per session. Single chronological conversation; every run/iterate/variant/carousel is a turn in the same thread. Selecting a canvas artifact (D-02) focuses/scrolls to its originating turn.
- **D-06:** Persist chat via Mastra memory (user decision). Use Mastra's memory subsystem (threads/resources + storage adapter) as the conversation store rather than a new Convex chat table. Artifacts/versions/exports continue to persist in Convex `experience*` tables (unchanged). **Load-bearing boundary note:** consciously pulls a slice of the Phase-5 durable-workflow/Mastra-storage work forward. Researcher MUST resolve (a) Mastra-memory ↔ pinned `@mastra/core@1.37.1`/`@mastra/ai-sdk@1.4.3`/`ai@6` compatibility; (b) thread/resource ↔ Lab session ↔ focused-artifact lineage mapping; (c) keeping the agents package backend-free vs. wiring a storage adapter; (d) reload-safety. Convex chat table is the documented FALLBACK if Mastra memory proves incompatible.
- **D-07:** Generation controls live in a composer control strip. Brand/sub-brand/artifact-type/output-profile selectors become compact controls attached to the chat input, set once and reused until changed. Reuse the existing selector logic from `RequestPanel` (Convex brand/sub-brand queries, profile/type enums).

**Inline interactive cards**
- **D-08:** Campaign-plan HITL renders inline and interactive in chat (the named deliverable). Card shows brief summary + audience + message hierarchy + 3 creative directions + recommended one; direction selection happens inline and resumes the suspended Mastra workflow (Phase-4 D-05). Frame generation happens only after inline selection. Frame-count adjustment (Phase-4 D-08) is part of this inline card.
- **D-09:** Build a reusable inline-card framework. A small general pattern: Jio-DS components rendered in the chat stream that can post an action back to resume/continue the workflow. Campaign-plan card is the first consumer; future HITL/confirm moments reuse the same pattern. Designed upfront to avoid a refactor when the second inline card arrives.
- **D-10:** Reuse the suspend/resume contract; new inline view. Keep the Phase-4 suspend/resume + plan-data contract + resume route untouched (read-only). Build a net-new Jio-DS inline chat component that renders the same plan data and posts the same selection back. Retire/hide the canvas plan-card shape (its rendering, not its contract).

**Run feedback & streaming**
- **D-11:** Run progress is a streaming assistant turn. Each run is an assistant message streaming live step progress as `ExperienceBuilderEvent`s arrive (resolve → retrieve → plan → generate → compile → validate → evaluate → repair), updating a step ticker/status line in place and collapsing to a compact summary when done. Full event detail is expandable within the turn (where `RunInspectorPanel`'s content goes).
- **D-12:** Failures: inline headline + canvas gap card. A run error or foundation/component gap flips the streaming turn to a readable inline error/gap state, AND the detailed gap-report card lands on the canvas (consistent with D-04). Chat = headline, canvas = full report.
- **D-13:** Both docked panels retired into chat. `RequestPanel` → chat composer + control strip (D-07); `RunInspectorPanel` → the streaming assistant turn's expandable detail (D-11). The right dock is removed; the Lab layout becomes left chat + right canvas.

### Pattern 1: Inline-card framework via `renderMessagePart` (D-09)
**What:** ChatSurface already supports custom message parts. A message carries `parts: [{ type, ...data }]`; `renderMessagePart(part, ctx)` returns a React node for known part types and `null` to fall through to text. The campaign-plan card and the run turn become part types.
**When to use:** Every inline interactive moment (campaign-plan now; any future confirm/HITL card).
**Example:**
```tsx
// Source: packages/ui/src/components/ChatSurface/parts/index.tsx (defaultRenderMessagePart pattern)
const PART_CAMPAIGN_PLAN = 'data-campaign-plan';
const PART_RUN_PROGRESS  = 'data-run-progress';

const renderLabMessagePart: RenderMessagePart = (part, ctx) => {
  if (part.type === PART_CAMPAIGN_PLAN) {
```

### Pattern 2: Streaming run turn (D-11) — drive `AgentThinking`/step ticker from the NDJSON stream
**What:** Reuse `useExperienceLabRun`'s per-event accumulation (`setEvents((prev) => [...prev, frame.event])`) but render it as an in-place assistant turn. The step ticker maps the ordered steps (resolve→retrieve→plan→generate→compile→validate→evaluate→repair); on the terminal frame it collapses to a one-line summary; the full event log (the current `RunInspectorPanel.describeEvent` rows) lives behind an expandable (`Collapsible`/`Accordion` from `@oneui/ui`).
**When to use:** Every run and resume.
**Example shape:** Keep `RunInspectorPanel.describeEvent` (it maps `ExperienceBuilderEventT` → label + Badge appearance + statusText) and render the rows inside a `Collapsible` body; use `AgentThinking thinkingMessages={…}` for the live ticker while `isRunning`.

### Pattern 3: Mastra memory as a plain thread/message store (D-06) — NOT agent recall
**What:** The Lab runs a *deterministic workflow runner*, not a Mastra `Agent`, so memory is used directly as a thread store, not as auto-recall context. Instantiate a store + memory in a Node route, create one thread per Lab session keyed by a session id (`threadId`), use `resourceId` for the (future) user/brand scope, and `saveMessages`/`query` to append/load turns.
**When to use:** Conversation persistence + reload-safety (D-05/D-06d).
**Example:**
```ts
// Source: mastra.ai/docs/memory/overview + @mastra/core/memory MastraMemory API (verified .d.ts)
// Lives in apps/platform (Node route) — NOT in packages/experience-builder-agents (D-06c).
import { Memory } from '@mastra/memory';                // optional high-level wrapper
import { FilesystemStore } from '@mastra/core/storage'; // ships with pinned @mastra/core
const memory = new Memory({ storage: new FilesystemStore({ /* dir */ }) });
await memory.createThread({ threadId: sessionId, resourceId: brandId, title: 'Lab session' });
await memory.saveMessages({ messages: [{ role: 'user', content }] });
const { messages } = await memory.query({ threadId: sessionId });
```
Reload-safety: on reopen, read the thread by `sessionId` → rehydrate `messages[]`; for each message that links a Convex `artifactId`/`versionId`, the canvas re-fetches via existing Convex queries and re-places the artifact (D-06d).

### Pattern 4: Chat ↔ canvas two-way selection binding (D-02)
**What:** Observe tldraw selection with the side-effect handler already used in `ExperienceLabCanvas.tsx` (`registerAfterChangeHandler('instance_page_state', sync)` + `getSelectedShapeIds()`); map the selected `ArtifactCardShape` → its originating turn id (stored as a message part or shape prop), set chat focus, scroll to that turn. Reverse direction: clicking a turn's "focus on canvas" action calls `editor.select(shapeId)` + `editor.zoomToSelection()`.
**Anti-pattern avoided:** do NOT poll the editor; use the existing reactive side-effect registration.

### Anti-Patterns to Avoid
- **Adopting `ai`'s `useChat`/`DefaultChatTransport` in the Lab** — breaches the Lab-scoped single-`ai` gate and the NDJSON contract. The Lab's transport is `useExperienceLabRun` + `runStream.ts`. (The Builder's `useAgentChat.ts` uses `useChat`, but the Builder is a *separate, non-isolated* surface — do not mirror it here.)
- **Wiring the Mastra store inside `packages/experience-builder-agents`** — breaks the backend-free posture (D-06c, and the established "agents package never sees `@oneui/convex`/backend" rule). The store lives in the app/route layer.
- **Changing `runStream.ts`, `workflow.ts`, the run/resume route contracts, or `CampaignPlanSchema`** — these are read-only contracts (D-10). The inline card renders the same `CampaignPlanT` and posts the same `{directionIndex, frameCount}`.
- **Setting a manual `background` on a chat container holding Jio components** — use `<Surface mode="…">` (Surface Context mandate). Decorative strokes on tinted Surfaces are forbidden.
- **Hardcoded font sizes/weights/line-heights or any literal color/spacing** — Typography V2 tokens + token-only CSS only (LAB-02 + check:literals over the Lab route).

## Common Pitfalls

### Pitfall 1: Breaching the Lab-scoped single-`ai` gate via the chat UI
**What goes wrong:** Reaching for `ai`'s `useChat`/`DefaultChatTransport` (as the Builder's `useAgentChat.ts` does) pulls `ai`/`@ai-sdk/react` into the Lab subtree, tripping the Lab-scoped single-`ai` gate in `ci:gates`.
**Why it happens:** `ChatSurface`'s `ChatMessage`/`UIMessage`/`ChatStatus` types are *structurally* AI-SDK-shaped, tempting a copy of the Builder's hook.
**How to avoid:** Keep the Lab's transport as the existing NDJSON `useExperienceLabRun`. `ChatSurface` is headless of transport — it only needs `messages[]`, `status`, `onSubmit`. The only module that may touch `ai`/`@ai-sdk` in the Lab subtree is `packages/experience-builder-agents/src/modelAdapter.ts`.
**Warning signs:** Any new `import … from 'ai'` or `@ai-sdk/react` under `(experience-lab)`.

### Pitfall 2: Wiring the Mastra store into the agents package
**What goes wrong:** Importing a storage adapter into `packages/experience-builder-agents` breaks its backend-free posture (the same rule that keeps `@oneui/convex` out of it — STATE.md 02.1 decisions).
**How to avoid:** Own the `Memory`/store in a Node route or app module (D-06c). The agents package never sees the store; if a turn needs persisting, the route does it after the run, exactly as `persistRun`/`persistCampaignPlan` are injected today.
**Warning signs:** `@mastra/memory`/`@mastra/libsql`/`@mastra/core/storage` imported under `packages/experience-builder-agents`.

### Pitfall 3: Mastra `FilesystemStore` and serverless multi-process durability
**What goes wrong:** A filesystem-backed store persists to a local directory; in a serverless/multi-instance deploy a thread written by one process may not be readable by another (the exact class of bug the resume route already documents: "process-memory cache does NOT survive across HTTP requests in serverless/multi-process — Pitfall 4").
**Why it happens:** Reusing the suspend/resume Pitfall-4 lesson: in-process or per-instance stores are not durable across requests.
**How to avoid:** For a single dev/UAT machine `FilesystemStore` is fine. For any multi-instance target, prefer `@mastra/libsql` (a real DB file/URL) OR the Convex fallback (which is already multi-process-durable). The planner should pick based on the deploy target; given Phase-5 defers full durable workflow state, single-session filesystem durability is acceptable for this phase if UAT is single-instance — but DOCUMENT this as the boundary.
**Warning signs:** Conversation appears empty after reload in a multi-instance preview deploy.

### Pitfall 4: Losing the Phase-02.1 sub-brand skip-gate when lifting selectors
**What goes wrong:** Naively lifting the sub-brand `Select` into the strip without the `isResolvedBrandId` guard re-introduces the crash the 02.1 work fixed: a stale/placeholder `brandId` passed to `getByParentBrand` (which validates `v.id('brands')`) throws an `ArgumentValidationError` that crashes the whole canvas.
**How to avoid:** Lift the `isResolvedBrandId` memo + `'skip'` sentinel verbatim (RequestPanel.tsx:147–157). The sub-brand query must stay skipped until the brandId resolves against the live `brands` list.
**Warning signs:** Canvas crash on selecting a brand whose id is not in the loaded list.

### Pitfall 5: Treating a suspended campaign run as a failure in the streaming turn
**What goes wrong:** A campaign run SUSPENDS and the result frame rides `outcome:'gap'` on the wire (the workflow maps every suspend → `gap`). If the run turn renders `gap` as a failure (D-12), the inline plan card never shows.
**Why it happens:** `finalizeRun`/the routes intentionally surface suspend as wire-`gap` while carrying `campaignPlan` + `campaignRunId` on the result frame.
**How to avoid:** In the run turn, branch on the presence of `result.campaignPlan` BEFORE treating `gap` as failure — a `gap` frame carrying `campaignPlan` is a *suspend*, render the inline `CampaignPlanCard`. Only a `gap` with no plan + a gap event is a real refusal (→ D-12 headline + canvas gap card).
**Warning signs:** Campaign prompts produce a "generation refused" headline instead of a plan card.

### Pitfall 6: Anthropic-safe schema rules leaking into any new resume payload
**What goes wrong:** If the inline card builds a new resume schema with `z.number().int()/.min()/.max()`, Anthropic structured-output rejects integer bounds (400).
**How to avoid:** The resume contract already exists (`CampaignResumeSchema` = plain `z.number()` directionIndex/frameCount, clamped at runtime). The inline card just posts `{ runId, brandId, directionIndex, frameCount, subBrandConfigId? }` to `/resume` — do NOT introduce a new schema. Clamp client-side is optional (the route + workflow already clamp).
**Warning signs:** A new Zod schema under the Lab with `.int()`/`.min()`/`.max()` on a number.

## Open Questions (RESOLVED)

> The two load-bearing D-06 questions (Q1 store choice, Q2 sessionId reload-safety) are now **locked as the default plan decision**. The Plan 01 Task 2 spike is a *confirmation/validation* of this locked default, not an open-ended architecture choice — it round-trips the locked store and only branches to the documented Convex fallback if that round-trip fails. Q3 (run-turn granularity, CONTEXT discretion) is settled below for the UI-spec.

1. **Conversation store — RESOLVED: zero-dependency `@mastra/core/storage` store, Convex fallback as contingency.**
   - **Decision (locked default plan):** Use the lower-level `MastraMemory` (via `@mastra/core/memory`) backed by a `@mastra/core/storage` store — `InMemoryStore` for tests, `FilesystemStore` for single-instance UAT durability. This is the **zero-new-dependency path**: `@mastra/core@1.37.1` is already pinned and already ships both the memory base and the store classes, so no new package install and no package-legitimacy gate is required for the default path. The high-level `@mastra/memory` `Memory` wrapper and `@mastra/libsql` remain available as optional upgrades (gated behind the Plan 01 Task 3 checkpoint) but are NOT the planned path.
   - **Why this is the default (already well-argued above):** the Lab runs a deterministic workflow runner (not a Mastra `Agent`), so memory is used purely as a thread/message store (`createThread`/`saveMessages`/`query`) — the lower-level `MastraMemory` covers this without an `Agent`/`Mastra` binding (A2). Filesystem durability is sufficient for the single-instance UAT target this phase ships against (A3); `@mastra/libsql` is the noted multi-instance upgrade, deferred with the rest of the Phase-5 durable-workflow work.
   - **Confirmation, not discovery (Plan 01 Task 2):** the spike instantiates `MastraMemory` + `InMemoryStore` in a Node route and confirms a `createThread`/`saveMessages`/`query` round-trip works with no `Agent` binding. This validates the locked decision. **Contingency branch:** if (and only if) the round-trip does NOT work cleanly, take the documented **Convex `experienceChat` table fallback** (a NEW append-only thread/message table — NOT the campaign-scoped `chatMessages` table) — strictly simpler and already multi-process-durable. Record whichever path was taken in the Plan 01 SUMMARY.

2. **sessionId / threadId — RESOLVED: client-generated, `localStorage`-persisted, `resourceId = brandId`.**
   - **Decision (locked):** Session identity is a stable client-generated `sessionId` (e.g. a UUID) persisted in `localStorage`, used directly as the Mastra `threadId`. There is no auth/user model in the Lab yet, so no server-side session is introduced. `resourceId = brandId` scopes threads per brand for the future multi-user model. On reopen, the Lab reads the `localStorage` `sessionId`, queries the thread, rehydrates `messages[]`, and re-places linked Convex artifacts via existing queries (D-06d reload-safety). The `sessionId` is an opaque, non-sensitive grouping identifier — not an auth token (ASVS V3 partial).
   - This is the locked path for Plan 01 (store keyed by `threadId`/`resourceId`) and Plan 03 (`useLabConversation` reads/writes the `localStorage` `sessionId`).

3. **Run-turn granularity in the expandable detail — RESOLVED (CONTEXT discretion).**
   - **Decision:** Ticker = the 8 named steps (resolve→retrieve→plan→generate→compile→validate→evaluate→repair) with live status; the expandable detail = the full ordered `ExperienceBuilderEvent` rows rendered via the existing `describeEvent` mapping verbatim. Wired in Plan 03 Task 2 RunTurn.

| Pattern | STRIDE | Standard Mitigation |
