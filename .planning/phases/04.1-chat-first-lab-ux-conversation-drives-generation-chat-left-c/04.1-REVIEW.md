# Phase 04.1: Code Review Report

**Reviewed:** 2026-06-03
**Depth:** standard
**Files Reviewed:** 13
**Status:** issues_found

## Summary

The implementation covers the chat-first Lab refactor: a left-pane `ExperienceLabChat` backed by `useLabConversation` (NDJSON transport + D-06 persistence), a `ComposerControlStrip` replacing the retired `RequestPanel`, a `CampaignPlanCard` HITL inline card, and a two-way chat↔canvas selection binding (`useChatCanvasFocus`). The overall isolation constraints (no `(builder)` imports, no `ai`/`@ai-sdk` imports, token-only CSS) are met. However, four correctness/security bugs were found, and six quality issues warrant attention before release.

The most critical issues are: (1) `stop()` cannot actually cancel a mid-stream NDJSON read because the abort signal is not threaded to the `ReadableStream` reader, leaving the UI in a permanently incorrect state; (2) `CampaignPlanCard.handleGenerate` passes non-OK responses to `onResolved` without checking `response.ok`, silently delivering HTTP error payloads as NDJSON; (3) the conversation API route exposes Zod validation detail in 400 responses with no try/catch around store operations, meaning store exceptions propagate as unhandled 500s; and (4) `RunTurn` uses a hard-coded `id="run-turn-detail-rows"` that collides across every turn in the transcript, breaking WCAG 4.1.1 and aria-controls linkage.

## Critical Issues

### CR-01: `stop()` does not cancel the NDJSON body stream — status set prematurely to `'ready'`, or incorrectly to `'error'` on fetch abort

**File:** `apps/platform/src/app/(platform)/(experience-lab)/_chat/useLabConversation.ts:297-398`

**Issue:** The `AbortController` signal is only attached to the initial `fetch()` call (line 306), not to the `ReadableStream` reader. This creates two failure modes:

1. **Abort fires while the fetch is in-flight** (before the response arrives): the fetch `catch` block runs asynchronously and calls `setStatus('error')` _after_ `stop()` has already called `setStatus('ready')`. The last state write wins: final status is `'error'` even though the user intended to stop cleanly. The assistant message turn also gets `outcome: 'error'` instead of being dismissed.

2. **Abort fires while the body is being consumed** (during NDJSON streaming, after the response arrives): the abort signal is ignored by the `ReadableStream` reader (`body.getReader().read()` has no abort awareness). The reader continues until the server closes the stream. `stop()` flips status to `'ready'` and hides the stop button, but the background goroutine is still running and will call `setStatus(...)` again when it finishes, producing a second spurious state transition. The assistant turn stays visually stuck as `status: 'running'` until the stream closes.

**Fix:** Thread a `cancelledRef` into the `runStream` async closure, and check it before each async operation. For the fetch-abort case, check `err.name === 'AbortError'` and handle it distinctly from other errors:

```typescript
const stop = useCallback(() => {
  abortRef.current?.abort();
  abortRef.current = null;
  // Do NOT call setStatus here. The runStream catch will handle the
  // AbortError; patch the run part to a neutral cancelled state instead.
}, []);

// In runStream catch block (line ~308):
} catch (err) {
  const isAbort = err instanceof Error && err.name === 'AbortError';
  patchRunPart(assistantId, (p) => ({
    ...p,
    status: isAbort ? 'done' : 'error',
    outcome: isAbort ? 'gap' : 'error',  // or a new 'cancelled' outcome
  }));
  setStatus('ready');  // always go back to ready on stop/error
  setError(isAbort ? null : (err instanceof Error ? err : new Error('Run request failed')));
  abortRef.current = null;
  return;
}
```

For the body-streaming case, pass the `signal` to the reader or use a `cancelledRef` checked inside the NDJSON loop:

```typescript
for await (const frame of readNdjson(response)) {
  if (controller.signal.aborted) break;   // early exit on stop
  // ... existing frame handling
}
```

### CR-02: `CampaignPlanCard.handleGenerate` passes error `Response` objects to `onResolved` unchecked

**File:** `apps/platform/src/app/(platform)/(experience-lab)/_chat/parts/CampaignPlanCard.tsx:109-117`

**Issue:** `handleGenerate` does not check `response.ok` before calling `onResolved?.(response)`. If the `/resume` endpoint returns a 4xx or 5xx response, the raw error `Response` is handed to the conversation hook's NDJSON consumer. The NDJSON decoder will attempt to read the error body as frames; a JSON error body (e.g., `{"error":"..."}`) will parse as an unrecognised line and be silently dropped, leaving the UI stuck at `submitting=false` with no visible error message. A non-JSON error body causes `parseRunStreamLine` to produce `null` frames throughout — no artifact is ever placed and the user receives no feedback.

Additionally, there is no error state in the component: if `doFetch` itself throws (network failure), the `finally` block resets `submitting=false` but no error is shown to the user, so the generate button silently re-enables without explanation.

**Fix:**

```typescript
const response = await doFetch(RESUME_ENDPOINT, {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify(body),
});

if (!response.ok) {
  setError(`Resume failed (${response.status})`);
  return;
}
onResolved?.(response);
```

Add an `error` state (`const [error, setError] = useState<string | null>(null)`) and render it in the card so the user can retry with context.

### CR-03: Conversation API route — store operations unguarded; Zod issue details leaked in 400 responses

**File:** `apps/platform/src/app/api/experience-lab/conversation/route.ts:107-122`

**Issue (a — unhandled store throws):** The `store.createThread`, `store.saveMessages`, and `store.query` calls at lines 108, 118–121 have NO try/catch. If the `InMemoryStore` throws (e.g., internal Mastra domain error, or a future store replacement that rejects), Next.js catches the unhandled rejection and returns a 500 with a stack trace in development mode. This is an uncontrolled error surface; the route should wrap the store block and return a structured 500.

**Issue (b — Zod issue detail in 400):** Line 101 returns `result.error.issues` directly to the caller. In production, Zod issue objects include path, code, and message details about the schema shape. Since this is an internal API endpoint these are low-severity but constitute unnecessary schema disclosure. An opaque 400 is preferable.

**Fix:**

```typescript
// Wrap store operations:
try {
  if (body.action === 'load') {
    const messages = await store.query({ threadId: body.sessionId });
    return json({ sessionId: body.sessionId, messages });
  }
  const resourceId = body.brandId ?? PLACEHOLDER_BRAND_ID;
  const messages: LabMessage[] = body.messages.map((m) => ({ role: m.role, text: m.text }));
  await store.createThread({ threadId: body.sessionId, resourceId });
  await store.saveMessages({ threadId: body.sessionId, resourceId, messages });
  const restored = await store.query({ threadId: body.sessionId });
  return json({ sessionId: body.sessionId, messages: restored });
} catch {
  return json({ error: 'Conversation store error' }, 500);
}

// Replace Zod issue disclosure (line 101):
return json({ error: 'Invalid conversation request' }, 400);
```

### CR-04: Hard-coded `id="run-turn-detail-rows"` duplicates across transcript — WCAG 4.1.1 violation, broken `aria-controls`

**File:** `apps/platform/src/app/(platform)/(experience-lab)/_chat/parts/RunTurn.tsx:234,239`

**Issue:** The `RunDetail` sub-component renders `<button aria-controls="run-turn-detail-rows">` and `<ol id="run-turn-detail-rows">` with a literal, static `id`. Every assistant turn in the chat transcript renders a `RunTurn`, so the DOM will contain multiple elements with `id="run-turn-detail-rows"`. This violates WCAG 4.1.1 (duplicate id), breaks the `aria-controls` relationship for all instances except whichever the browser picks as canonical, and will fail `pnpm test:a11y`.

**Fix:** Use React's `useId()` to scope the id per instance:

```typescript
// In RunDetail component:
function RunDetail({ rows, expanded, onToggle }: RunDetailProps) {
  const detailId = useId();
  if (rows.length === 0) return null;
  return (
    <div className={styles.detail}>
      <button
        type="button"
        className={styles.detailToggle}
        onClick={onToggle}
        aria-expanded={expanded}
        aria-controls={detailId}
      >
        {expanded ? 'Hide run detail' : 'Show run detail'}
      </button>
      {expanded && (
        <ol id={detailId} className={styles.eventLog} data-testid="run-turn-events">
          ...
        </ol>
      )}
    </div>
  );
}
```

Import `useId` from React at the top of `RunTurn.tsx`.

### WR-01: `htmlFor` attributes in `ComposerControlStrip` reference ids that are never assigned to any DOM element

**File:** `apps/platform/src/app/(platform)/(experience-lab)/_chat/ComposerControlStrip.tsx:202,217,232,247`

The label-to-control association is therefore broken: clicking a label does not focus the associated selector, and screen readers do not associate the label text with the control. <Select

```tsx
  value={brandId}
  onChange={handleBrandChange}
  ...
/>
```

Repeat for `composer-subbrand`, `composer-type`, and `composer-profile`.

### WR-02: `stop()` sets status `'ready'` unconditionally even when called with no run in progress

**File:** `apps/platform/src/app/(platform)/(experience-lab)/_chat/useLabConversation.ts:396-400`

**Issue:** `stop()` calls `setStatus('ready')` regardless of whether a run is currently active. If called when `status === 'error'` (e.g., the user's previous run errored and status is already `'error'`), the call silently resets status to `'ready'` and clears the error signal. Callers relying on the error state to show an error message would see it disappear without user action.

**Fix:** Guard on the current abort ref:

```typescript
const stop = useCallback(() => {
  if (!abortRef.current) return;  // no active run — no-op
  abortRef.current.abort();
  abortRef.current = null;
  setStatus('ready');
}, []);
```

This also prevents the spurious `setStatus('ready')` double-call when invoked from the `onStop` handler after status has already been resolved by the normal run completion path.

### WR-03: Suggestion quick-prompts can fire `submit()` while a run is streaming

**File:** `apps/platform/src/app/(platform)/(experience-lab)/_chat/ExperienceLabChat.tsx:302-308`

**Issue:** The three suggestion chips call `void submit(...)` directly in their `onClick` handlers. The `suggestions` array is memoized on `[submit]`, and the `ChatSurface` component may not disable suggestion chips when `status === 'streaming'`. There is no guard inside `useLabConversation`'s `submit()` (or `runStream()`) against concurrent invocations. If the user clicks a suggestion chip mid-stream, a second `runStream` call starts a new `AbortController`, appends new user/assistant messages to the transcript, and begins a second concurrent fetch — while the first stream is still running. The two streams will both call `setStatus(...)` and `patchRunPart(...)` interleaved on different message ids, producing garbled UI state.

**Fix:** Add a concurrent-run guard at the top of `submit()`:

```typescript
const submit = useCallback(
  async (text: string) => {
    if (status !== 'ready') return;  // guard concurrent runs
    const trimmed = text.trim();
    if (!trimmed) return;
    await runStream(trimmed);
  },
  [runStream, status],  // add status to deps
);
```

Alternatively gate the guard inside `runStream` using the abort ref: if `abortRef.current !== null` a run is already active.

### WR-04: `persistFetch` derived inline (non-memoized) makes the rehydration `useEffect` re-run on every render in test contexts

**File:** `apps/platform/src/app/(platform)/(experience-lab)/_chat/useLabConversation.ts:210-211,245`

**Issue:** `persistFetch` is computed inline on every render:

```typescript
const doFetch = fetchImpl ?? (typeof fetch !== 'undefined' ? fetch : undefined);
const persistFetch = persistFetchImpl ?? doFetch;
```

The rehydration `useEffect` at line 216 lists `[persistFetch, sessionId]` as its deps. In production (no `fetchImpl` prop) the global `fetch` reference is stable, so the effect fires only once per `sessionId` change as intended. However, when tests pass an inline function literal as `fetchImpl` (e.g., `fetchImpl={() => Promise.resolve(...)}`) without wrapping it in `useCallback`, a new function identity is created on every render, causing the effect to fire repeatedly and flood the test mock with repeated `load` requests. This is a latent test-reliability issue that will manifest as flaky or failing tests that depend on `persistFetchImpl`.

**Fix:** Stabilise both references in refs like the existing pattern used for `stripRef`/`canvasRef`:

```typescript
const fetchRef = useRef(fetchImpl);
fetchRef.current = fetchImpl;
const persistFetchRef = useRef(persistFetchImpl);
persistFetchRef.current = persistFetchImpl;

const doFetch = useCallback(
  (...args: Parameters<typeof fetch>) =>
    (fetchRef.current ?? fetch)(...args),
  []
);
const persistFetch = useCallback(
  (...args: Parameters<typeof fetch>) =>
    (persistFetchRef.current ?? doFetch)(...args),
  [doFetch]
);
```

Then the `useEffect` deps remain stable identities regardless of the incoming prop value.

### WR-05: `CampaignPlanCard` leaves user with no feedback on network error

**File:** `apps/platform/src/app/(platform)/(experience-lab)/_chat/parts/CampaignPlanCard.tsx:97-120`

**Issue:** The `handleGenerate` function wraps the `doFetch` call in `try/finally` but has no `catch`. If `doFetch` throws (network failure, DNS error, etc.), `submitting` resets to `false` and the generate button re-enables — but no error message is displayed to the user. Combined with the CR-02 issue (no `response.ok` check), there are two distinct failure paths with zero user-visible feedback.

**Fix:** Add an explicit error state and a catch block:

```typescript
const [submitError, setSubmitError] = useState<string | null>(null);

async function handleGenerate() {
  if (!doFetch || submitting) return;
  setSubmitting(true);
  setSubmitError(null);
  try {
    const response = await doFetch(RESUME_ENDPOINT, { ... });
    if (!response.ok) throw new Error(`Resume failed (${response.status})`);
    onResolved?.(response);
  } catch (err) {
    setSubmitError(err instanceof Error ? err.message : 'Failed to start generation.');
  } finally {
    setSubmitting(false);
  }
}
```

Render `{submitError && <p role="alert" className={styles.submitError}>{submitError}</p>}` above the actions row.

### IN-02: `useSessionId` returns `''` as initial state — callers must handle empty-string `sessionId`

**File:** `apps/platform/src/app/(platform)/(experience-lab)/_chat/ExperienceLabChat.tsx:92-108`

**Issue:** `useSessionId()` initialises `sessionId` to `''` and only populates the real id via a `useEffect` (which runs after first render). The hook return value is passed directly to `useLabConversation` as `sessionId`. The rehydration effect guards `if (!sessionId) return`, so an empty-string session does not fire the load call. However, if the user is fast enough to submit a message before the effect fires (possible when the localStorage read is synchronous but React batching delays the `setSessionId`), `persistTurn` is called with `sessionId=''` — a valid Mastra thread key that creates a real thread under the empty-string id, permanently polluting the store with an orphaned thread.

**Fix:** Initialise with the real id synchronously where possible, using a lazy state initialiser:

```typescript
function readOrCreateSessionId(): string {
  try {
    let id = window.localStorage.getItem(SESSION_STORAGE_KEY);
    if (!id) {
      id = `lab-${Date.now().toString(36)}-${Math.random().toString(36).slice(2, 10)}`;
      window.localStorage.setItem(SESSION_STORAGE_KEY, id);
    }
    return id;
  } catch {
    return `lab-${Date.now().toString(36)}`;
  }
}

// Use lazy initialiser to avoid the '' → real id transition:
const [sessionId] = useState<string>(() =>
  typeof window !== 'undefined' ? readOrCreateSessionId() : ''
);
```

This eliminates the empty-string window entirely for SSR-safe client components.

### IN-03: `--Surface-Main` used as canvas background — deviates from the surface-context rule

**File:** `apps/platform/src/app/(platform)/(experience-lab)/_canvas/ExperienceLabCanvas.module.css:17`

**Issue:** The canvas page container sets `background-color: var(--Surface-Main)`. `CLAUDE.md` states: "Never use `--Surface-Main` directly — it's hardcoded to the page background and creates a visible 'hole' inside `<Surface>` containers." The canvas IS a default surface (not nested inside a coloured Surface), so the White-Canvas effect is correct here in practice. However, the canvasPane is wrapped in `<Surface mode="default">` in `ExperienceLabShell`. Inside that Surface, the CSS engine emits `--Surface-Fill-Default` for the container's own background. The correct token for a raw container background inside a default Surface is `var(--Primary-Default)` (or the fill token) rather than the page-pinned `--Surface-Main`.

Per the CLAUDE.md Focus Halo pattern, `--Surface-Main` is acceptable as a **fallback** before brand CSS injects (first paint), but should not be the primary background reference on a live-surface container.

**Fix:** Align with the surface context rule by removing the explicit `background-color` from `.page` and letting the parent `<Surface mode="default">` own the fill. If the canvas host genuinely needs its own background (to cover tldraw's transparent areas), use `var(--Primary-Default)` with a `var(--Surface-Main)` fallback matching the Focus Halo pattern:

```css
.page {
  position: relative;
  width: 100%;
  height: 100%;
  background-color: var(--Primary-Default, var(--Surface-Main));
}
```
