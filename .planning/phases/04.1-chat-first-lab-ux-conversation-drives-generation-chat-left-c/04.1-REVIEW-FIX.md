## Fixed Issues

### CR-01: `stop()` does not cancel the NDJSON body stream
**Files modified:** `apps/platform/src/app/(platform)/(experience-lab)/_chat/useLabConversation.ts`
**Commit:** f8c14385
**Applied fix:** Threaded the abort signal into `readNdjson` (cancels the `ReadableStream` reader on abort + stops yielding, with an `abort` listener + `finally` cleanup). Both `runStream` catch blocks now detect `AbortError` and treat it as a clean cancel (status → `'ready'`, neutral run part, `error` cleared) so a user stop wins over the async catch instead of leaving the UI on `'error'`. Added a post-loop `controller.signal.aborted` guard that ends the turn neutrally without running terminal disposition/placement. Rewrote `stop()` to abort without calling `setStatus` itself — `runStream` is the single owner of the terminal transition. **Requires human verification** (state-machine logic change).

### CR-02: `CampaignPlanCard.handleGenerate` passes error `Response` objects to `onResolved` unchecked
**Files modified:** `apps/platform/src/app/(platform)/(experience-lab)/_chat/parts/CampaignPlanCard.tsx`, `apps/platform/src/app/(platform)/(experience-lab)/_chat/parts/CampaignPlanCard.module.css`
**Commit:** 6b26b0e8
**Applied fix:** Added `submitError` state and a `response.ok` guard that returns early (with a `Resume failed (status)` message) before the response is handed to `onResolved`, so an HTTP error body is never parsed as a run stream. Added a token-only `.submitError` style (`--Negative-High`, Body-S typography) and a `role="alert"` message above the actions row.

### CR-03: Conversation API route — store operations unguarded; Zod issue details leaked in 400
**Files modified:** `apps/platform/src/app/api/experience-lab/conversation/route.ts`
**Commit:** 7c5f2a91
**Applied fix:** Wrapped all store operations (`query`/`createThread`/`saveMessages`) in a try/catch that returns a controlled `{ error: 'Conversation store error' }` 500. Replaced the 400 response to return an opaque `{ error: 'Invalid conversation request' }` with no `result.error.issues`. Runtime and plain Zod types unchanged.

### CR-04: Hard-coded `id="run-turn-detail-rows"` duplicates across transcript
**Files modified:** `apps/platform/src/app/(platform)/(experience-lab)/_chat/parts/RunTurn.tsx`
**Commit:** 5f513a92
**Applied fix:** Imported `useId` and generated a per-instance `detailId` in `RunDetail`, wiring both `aria-controls` and the `<ol id>` to it — eliminates the duplicate-id WCAG 4.1.1 violation.

### WR-01: `htmlFor` attributes reference ids never assigned to any element

Added an optional `id` prop to `SelectProps`, forwarded it to the `BaseSelect.Trigger`, and passed `composer-brand/subbrand/type/profile` ids on the four selects so each label's `htmlFor` targets a real control.

### WR-02: `stop()` sets status `'ready'` unconditionally even with no run in progress
**Files modified:** `apps/platform/src/app/(platform)/(experience-lab)/_chat/useLabConversation.ts`
**Commit:** f8c14385 (co-committed with CR-01, since both rewrite `stop()`)
**Applied fix:** `stop()` now no-ops when `abortRef.current` is null, so it never clobbers an existing `'error'` state or its error signal when there is nothing to stop. **Requires human verification** (state-machine logic change).

### WR-03: Suggestion quick-prompts can fire `submit()` while a run is streaming
**Files modified:** `apps/platform/src/app/(platform)/(experience-lab)/_chat/useLabConversation.ts`
**Commit:** 4ecb68d9

### WR-04: `persistFetch` derived inline makes the rehydration effect re-run every render

Introduced a `hasFetch` flag to preserve the original "no fetch in environment" guard now that the wrappers are always defined.
