## Gaps

### GAP-01: IR generation refuses with missing-required-prop on Logo/Image/Icon/ListItem
status: resolved
scope: pre-existing generation engine (packages/experience-builder-agents) — NOT introduced by 04.1; surfaced because chat-first now runs real generation
root_cause: `irGenerator.ts` built per-component prop contracts only for the `requested` set; components outside it (Logo/Image/Icon/ListItem) got no guidance, so the model omitted `alt`/`src`/`icon`/`title` and the repair loop couldn't recover.
fix attempt 1 (`9cf867fd`): inject positive required-prop contracts into the retry prompt. INSUFFICIENT — the model still omitted props across all 3 attempts in live UAT.
fix attempt 2 (`ab96a321`, EFFECTIVE): DETERMINISTIC required-prop backfill — before validation, every instance's missing registry-required props are filled with valid defaults (enum→values[0]; font/icon prop→token ref per the validator's font/icon rule; content props→derived/humanized; url→placeholder). Generation now converges on attempt 1 without depending on the model. Verified end-to-end against the REAL compiler+validator (agents suite 115 tests green). Caveat: a backfilled icon uses a token ref (validator-sanctioned), not a specific glyph — the user iterates for a specific icon. Never invents token values or components; artifact stays 100% Jio.

### GAP-02: Generation cannot start without a brand; no plan/ask mode
status: resolved
scope: 04.1 chat UX
root_cause: `/run` route schema requires `brandId: z.string().min(1)`; composer default `brandId` is `''` and `ExperienceLabChat` does not gate submit, so a no-brand submit POSTs `brandId:''` → 400 Bad Request, generation can't start.
fix: gate the chat-first flow on brand selection — when no brand is selected, do not POST; instead surface a graceful in-chat ask ("pick a brand first") + a disabled/affordance state, so the conversation guides the user instead of hard-failing.

### GAP-03: Duplicate messages in the chat feed
status: resolved
scope: 04.1 chat state
root_cause: `useLabConversation` rehydration has no seed-once guard; it can re-seed history (StrictMode double-effect / re-render) or race a live submit, producing duplicate turns. `ChatSurface` keys by `m.id` so the source is duplicate/double-seeded state, not the renderer.
fix: add a seed-once-per-sessionId guard to the rehydration effect; add a jsdom repro covering double-effect + persist-then-remount.
