## Summary

This is pure connective tissue. Every contract on both ends already exists and is frozen:

- The resolver (`foundationResolver.ts`) accepts `BrandFoundations { colorConfig?, presetSelection?, appearanceConfig? }` and runs the real `buildAvailableScales(colorConfig, presetSelection) ‚Üí buildThemeConfig(availableScales, appearanceConfig)` chain. It already handles partial brands (D-08) and gaps (FND-03). **No resolver change is required.**
- `RunExperienceInput` already carries an optional `brandFoundations` field, and the `planStep` and `generateStep` already pass `ctx.request.brandFoundations` into `resolveFoundation`/`generateIR`. **The only missing link is that nothing populates `brandFoundations`.**
- The server-side mapping from a `getBrandOverviewData` record into the resolver's exact field shape is already proven in production by `brandCSSPrecompute.ts` and `brandCSSExport.ts` (`colorConfig: foundationData.color?.config ?? null`). This phase reuses that mapping verbatim.

The work is therefore: (1) add a `foundationsLoader` injection point to `RunExperienceInput`; (2) make `resolveStep` call it and write `ctx.request.brandFoundations`; (3) build the Convex-backed loader in the run route (reusing the `ConvexHttpClient` already imported for `persistRun`); (4) thread an optional `subBrandConfigId` through the UI ‚Üí POST body ‚Üí loader, applying `applySubBrandAccentsToFoundation` before mapping; (5) add a route-through-workflow test with a mock loader.

**Primary recommendation:** Define `foundationsLoader: (input) => Promise<BrandFoundations | null>` (loader maps; resolver stays unchanged). Make `resolveStep` the only step that invokes it, writing `ctx.request.brandFoundations` so the downstream `planStep`/`generateStep` read it exactly as they do today. The Convex-backed loader lives entirely in the route (the Convex boundary), preserving the agents package's Convex-free invariant.

## Exact Mapping (Claude's Discretion #1 ‚Äî RESOLVED)

The canonical, production-proven mapping from a `getBrandOverviewData` record into `PrecomputeInput`/`BrandFoundations` is in **`packages/convex/convex/brandCSSPrecompute.ts:42-49`** and **`brandCSSExport.ts:42`** `[VERIFIED: codebase grep]`:

  presetSelection:    foundationData.presetSelection ?? null,

Field-by-field for **`BrandFoundations`** (the resolver only needs three fields):

| `BrandFoundations` field | Source in overview record | Notes |
|--------------------------|---------------------------|-------|
| `colorConfig` | **`foundationData.color?.config ?? null`** | `overview.color` is the *foundation record* (`foundationsByType['color']`, `foundations.ts:957`); the resolver wants its `.config` sub-object (the `ColorFoundationConfig` with `brandScales` / `savedCustomScales` / `lightnessScale`). **This is the one easy-to-miss step** ‚Äî pass `color.config`, NOT `color`. |
| `presetSelection` | **`foundationData.presetSelection ?? null`** | Already top-level on the overview record (`foundations.ts:967`), shaped `{ collection, selectedScales, selectedScaleNames }`. Pass as-is. |
| `appearanceConfig` | **`foundationData.appearanceConfig ?? null`** | Already top-level (`foundations.ts:968`). ## Loader Signature (Claude's Discretion #2 ‚Äî RESOLVED)

**Recommendation: the injected loader returns pre-mapped `BrandFoundations` (loader maps, resolver unchanged).**

```ts
// In workflow.ts ‚Äî Convex-free; only depends on the resolver's own BrandFoundations type.
export interface FoundationsLoaderInput {
  brandId: string;
  subBrandConfigId?: string;
}
export type FoundationsLoader = (
  input: FoundationsLoaderInput,
) => Promise<BrandFoundations | null>;
```

Add to `RunExperienceInput`:
```ts
export interface RunExperienceInput extends FoundationResolveInputT {
  requestedComponents?: string[];
  brandFoundations?: BrandFoundations;   // existing
  prompt?: string;                        // existing
  /** Injected Convex-backed foundations fetch (D-05). Tests inject a mock. */
  foundationsLoader?: FoundationsLoader;  // NEW
  /** Optional sub-brand selection (D-02). */
  subBrandConfigId?: string;              // NEW
}
```

**Why loader-maps over resolver-maps:**
1. **Keeps the agents package Convex-free (D-05).** The mapping `color?.config` references the Convex overview *shape*. If the resolver mapped, that shape knowledge leaks into the isolated package. Keeping the map in the route loader confines all Convex-shape coupling to the Convex boundary. `[VERIFIED: agents package.json has no @oneui/convex dependency]`
2. **`BrandFoundations` is already the resolver's stable input contract.** The loader producing it means the resolver and `planStep`/`generateStep` need zero changes ‚Äî they already read `ctx.request.brandFoundations`.
3. **Matches the established DI idiom.** `__setCallModelImpl` (`modelAdapter.ts:121`) and `__setCompileImpl` (`irGenerator.ts:65`) inject a *function* with a clean typed signature and a restore handle. The `foundationsLoader` injected through run input is the same idiom ‚Äî except injected per-run (cleaner than a module-level seam, since the loader is request-scoped and credentialed).

**`resolveStep` becomes (D-04):**
```ts
const resolveStep = createStep({
  id: 'resolve-foundation',
  inputSchema: ctxSchema, outputSchema: ctxSchema,
  execute: async ({ inputData }) => {
    const { ctx } = inputData;
    emit(ctx, { type: 'step', runId: ctx.runId, step: 'resolve-foundation', status: 'started', at: now() });
    if (ctx.request.foundationsLoader && !ctx.request.brandFoundations) {
      const loaded = await ctx.request.foundationsLoader({
        brandId: ctx.request.brandId,
        ...(ctx.request.subBrandConfigId ? { subBrandConfigId: ctx.request.subBrandConfigId } : {}),
      });
      if (loaded) ctx.request.brandFoundations = loaded;  // downstream steps already read this
    }
    emit(ctx, { type: 'step', runId: ctx.runId, step: 'resolve-foundation', status: 'completed', at: now() });
    return { ctx };
  },
});
```
Note: `planStep` (`workflow.ts:258`) and `generateStep` (`workflow.ts:393`) already spread `...(ctx.request.brandFoundations ? { brandFoundations: ... } : {})` into their `resolveFoundation`/`generateIR` calls ‚Äî so writing `ctx.request.brandFoundations` in `resolveStep` is automatically consumed. `[VERIFIED: source read]`

**The Convex-backed loader (in the run route ‚Äî the Convex boundary):**
```ts
function makeConvexFoundationsLoader(convex: ConvexHttpClient): FoundationsLoader {
  return async ({ brandId, subBrandConfigId }) => {
    if (brandId === PLACEHOLDER_BRAND_ID) return null; // 'jio' placeholder ‚Üí system defaults
    const overview = await convex.query(api.foundations.getBrandOverviewDataInternal, {
      brandId: brandId as Id<'brands'>,
    });
    if (!overview) return null;
    let base: Record<string, unknown> = overview;
    if (subBrandConfigId) {
      const sub = await convex.query(api.subBrandConfigs.getById, {
        id: subBrandConfigId as Id<'subBrandConfigs'>,
      });
      if (sub) base = applySubBrandAccentsToFoundation(overview, sub) ?? overview;
    }
    return {
      colorConfig:      (base as any).color?.config ?? null,
      presetSelection:  (base as any).presetSelection ?? null,
      appearanceConfig: (base as any).appearanceConfig ?? null,
    };
  };
}
```

**Open question for the planner (LOW risk):** `getBrandOverviewDataInternal` is an `internalQuery` (`foundations.ts:986`). Verify a `ConvexHttpClient` (a *browser/public* client used server-side) can call it, or whether the public `getBrandOverviewData` query (`foundations.ts:977`, same handler) must be used instead. `persistRun` calls public mutations via `ConvexHttpClient` (`route.ts:67`), suggesting the client reaches public functions; an `internalQuery` may be invokable only from Convex functions, not external clients. **Recommendation:** plan a Wave-0 spike or default to the public `getBrandOverviewData` (identical handler, `foundations.ts:979`) if `ConvexHttpClient` cannot invoke internals. CONTEXT.md D-06 names `getBrandOverviewDataInternal`; flag this as the one place the named internal may need to be the public twin. `[ASSUMED ‚Äî needs verification]`

## Theme Threading (Claude's Discretion #3 ‚Äî RESOLVED)

**Recommendation: theme stays presentation-only. Do NOT thread `theme` into resolution.**

Evidence `[VERIFIED: source read]`:
- `buildThemeConfig(availableScales, appearanceConfig)` takes **no theme argument** (`buildThemeConfig.ts`). It builds a theme-independent `ThemeConfig` (palette + scale definitions).
- Theme (`light`/`dark`) is applied *downstream of resolution* ‚Äî in `precomputeBrandCSSNew` it only selects `outerParentStep = darkMode ? 200 : 2500` during CSS generation (`precompute.ts:259-260`), long after `buildThemeConfig`.
- `resolveFoundation` already accepts `theme` (it's part of the frozen `FoundationResolveInputT`, `foundationResolve.ts:36-37`) but **never reads it** ‚Äî it passes through for callers who want it.
- The resolver's result is a `ThemeConfig` with `appearances` keyed by role ‚Äî theme-independent (`foundationResolve.ts:76`).

So the run's `theme` continues to flow as request metadata (it's already on `RunExperienceInput` and persisted via `persistRun`), but resolution does not consume it. The deferred idea "theme deep-threading" is confirmed unnecessary this phase.

### Pattern 1: Per-run function injection (vs module-level seam)
**What:** The loader is injected through `RunExperienceInput` (request-scoped), not via a `__set*` module global.
**When to use:** When the injected dependency is request-scoped and credentialed (the Convex client is built per-request in the route).
**Why this differs from `__setCallModelImpl`:** Model/compile seams are module-level because they're process-wide stateless implementations swapped only in tests. The loader carries a per-request Convex client, so request-scoped injection is correct. Tests still inject a plain mock function on the input object ‚Äî same testability.

### Anti-Patterns to Avoid
- **Mapping `colorConfig: overview.color`** instead of `overview.color?.config` ‚Äî yields empty scales and a spurious gap (see Exact Mapping pitfall).
- **Importing `@oneui/convex` (or `ConvexHttpClient`) into `experience-builder-agents`** ‚Äî breaks the Convex-free isolation (D-05). The loader is constructed only in the route.
- **Re-deriving the sub-brand merge** instead of calling `applySubBrandAccentsToFoundation` ‚Äî drift risk (D-03).
- **Memoizing the loaded foundations** ‚Äî D-07 forbids it; designers re-run expecting edits to take effect.
- **Threading `theme` into `buildThemeConfig`** ‚Äî it takes no theme arg; would be dead code.

## Common Pitfalls

### Pitfall 1: `colorConfig` sourced from the record instead of `.config`
**What goes wrong:** `colorConfig: overview.color` produces empty/undefined `brandScales`, so `buildAvailableScales` returns no usable scales, `buildThemeConfig` returns `null`, and the run emits a foundation gap for a perfectly valid brand.
**Why it happens:** `overview.color` *is* a foundation record; the config is nested at `.config`. Easy to miss because the field name "color" reads like the config.
**How to avoid:** Map exactly `colorConfig: overview.color?.config ?? null` (matches `brandCSSPrecompute.ts:43`).
**Warning signs:** A configured brand returns a gap; `buildThemeConfig` returns null in tests with real overview shapes.

### Pitfall 2: Convex import leaking into the agents package
**What goes wrong:** Adding the loader implementation (or `ConvexHttpClient`) inside `experience-builder-agents` breaks the Convex-free isolation (D-05) and the Phase 1 import guards.
**Why it happens:** It's tempting to co-locate the loader with the resolver.
**How to avoid:** The agents package defines only the `FoundationsLoader` *type*; the implementation is built in the route. Verify `packages/experience-builder-agents/package.json` still has no `@oneui/convex` dependency after the change.
**Warning signs:** CI import guard failure; `pnpm why` shows convex in the agents package.

### Pitfall 3: `internalQuery` not callable from `ConvexHttpClient`
**What goes wrong:** `getBrandOverviewDataInternal` is an `internalQuery`; an external `ConvexHttpClient` may only invoke public functions ‚Üí runtime error.
**Why it happens:** D-06 names the internal twin, but `ConvexHttpClient` is a public-API client.
**How to avoid:** Use the public `getBrandOverviewData` (same handler, `foundations.ts:977-980`) if the internal is not externally callable. Confirm in a Wave-0 spike.
**Warning signs:** Convex auth/visibility error at runtime; works in unit tests (mocked) but fails against a live deployment.

### Pitfall 4: `.strict()` body schema rejects the new field
**What goes wrong:** `RunRequestBody` is `.strict()` (`route.ts:123`); posting `subBrandConfigId` without adding it to the schema returns a 400.
**How to avoid:** Add `subBrandConfigId: z.string().optional()` to `RunRequestBody` and propagate it into `RunExperienceInput` and into `persistRun`'s recorded request if persistence should capture it.
