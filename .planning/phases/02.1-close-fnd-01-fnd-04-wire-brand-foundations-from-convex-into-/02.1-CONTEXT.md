# Phase 02.1: Close FND-01/FND-04 — wire brand foundations from Convex into run route → workflow - Context

**Gathered:** 2026-06-01
**Status:** Ready for planning

<domain>
## Phase Boundary

Close the **end-to-end wiring gap** that Phase 2 left open: the foundation resolver (`foundationResolver.ts`) already accepts `brandFoundations` and runs the real `buildAvailableScales → buildThemeConfig` chain, **but nothing feeds it real data**. The run route receives a real Convex `brandId` (the prompt card's brand Select is populated from `api.brands.list`, D-02) yet never uses it server-side — every real run resolves against empty foundations.

This phase is the connective tissue:

**run route → fetch the selected brand's (and sub-brand's) real foundations from Convex → pass them through the Mastra workflow → resolver produces a real `ThemeConfig`.**

It closes **FND-01** (system resolves brand/sub-brand → token set / type scale / spacing / surface / dimensions / output rules from Jio foundations for a given request) end-to-end, and the end-to-end half of **FND-04** (real Jio foundation source connected without a schema migration) that Phase 2 proved only at the resolver unit level.

**In scope:**
- A real Mastra `resolveStep` that calls an **injected `foundationsLoader`** to fetch brand foundations (the route supplies a Convex-backed loader; tests inject a mock). The agents package stays Convex-free.
- The route's Convex-backed loader: fetch via `getBrandOverviewDataInternal`, map the overview record into the resolver's `BrandFoundations` shape (`colorConfig` / `presetSelection` / `appearanceConfig`), pass into `runExperienceWorkflow`.
- **Sub-brand resolution (newly in scope):** an optional sub-brand Select on the prompt card / RequestPanel, and sub-brand override merge via the existing `applySubBrandAccentsToFoundation` util before the foundations reach the resolver.
- End-to-end verification through the route (not just resolver unit tests) that a real brand yields a real `ThemeConfig`, and a sub-brand yields the merged accents.

**Out of scope (later phases / not this phase):**
- Non-web output-profile foundation coverage (FND-02 — Phase 4); covered web profiles only.
- Sandboxed preview, validation pipeline, evaluator, repair loop, version-history browsing (Phase 3).
- Foundation caching/memoization (explicitly rejected this phase — always fetch fresh).
- Any change to the engine's resolution math or the frozen `FoundationResolveResult` contract.

</domain>

<decisions>
## Implementation Decisions

### Sub-brand scope (newly in scope)
- **D-01:** **Include sub-brand resolution now** (do not defer). FND-01's "brand/sub-brand" wording is satisfied in this phase rather than split into a later slice.
- **D-02:** **Sub-brand selection = optional dependent Select.** Add a second Select below the brand Select in the RequestPanel, populated from `subBrandConfigs.getByParentBrand(brandId)`, with a **"(Parent brand only)"** default. Sub-brand is **optional** — a run with no sub-brand resolves the parent brand alone. The Select is disabled until a brand is chosen. Mirrors the existing brand-Select pattern (D-02 of Phase 1). Requires a new optional `subBrandConfigId` prop on `PromptCardShape` threaded through the run request body.
- **D-03:** **Merge via existing `applySubBrandAccentsToFoundation`** (`@oneui/shared` — `packages/shared/src/utils/subBrandFoundation.ts`). Sub-brand owns **primary / secondary / sparkle / brand-bg**; neutral + semantic roles + dimensions/typography/presetSelection **inherit from the parent** — exactly as platform global injection and ExperienceCanvas already merge. **No new merge logic**; identical to how the live platform renders sub-brands (avoids drift).

### Fetch location & isolation
- **D-04:** **Resolution lives in the workflow, not the route.** Turn the current no-op `resolveStep` in `workflow.ts` into a **real Mastra step** that calls an **injected `foundationsLoader`** function and writes the resolved `brandFoundations`/theme into `RunContext`. This honors **ORCH-04** (resolution is orchestration → belongs in the Mastra workflow), gives the fetch step tracing for free, and keeps the route a thin invoke+stream shell.
- **D-05:** **Dependency injection keeps the agents package Convex-free.** The `experience-builder-agents` package MUST NOT take a hard dependency on Convex. The loader is injected through the run input (the route provides a `ConvexHttpClient` / `getBrandOverviewDataInternal`-backed implementation; unit tests provide a deterministic mock loader). This preserves package isolation and credential-free testability.
- **D-06:** The Convex-backed loader fetches via **`getBrandOverviewDataInternal`** (already returns the exact `presetSelection` + `appearanceConfig` shape the resolver needs) and, when a `subBrandConfigId` is present, fetches the sub-brand via `subBrandConfigs.getById` and applies `applySubBrandAccentsToFoundation` **before** mapping into `BrandFoundations`.

### Caching & determinism
- **D-07:** **Always fetch fresh** — one lightweight Convex query per run. **No foundation memoization.** Rationale: designers edit foundations and re-run expecting the change; staleness from a brand-keyed cache would break that. Determinism (same prompt + brand + type/profile → stable result) is already served by the existing plan/design/copy step memoization keyed on resolved inputs.

### Empty / unresolvable brand semantics (governed by carried-forward D-09)
- **D-08:** A selected real brand with partial or even zero configured foundations **resolves via the engine's system defaults** (`ensureNeutralRole` / `synthesizeBrandBgIfMissing`) and generates — "used a default" is NOT a gap (Phase 2 D-09, matches live platform). A **foundation gap** is reserved for a genuinely unresolvable profile (`buildThemeConfig` returns `null`) or an uncovered (non-web) output profile (FND-03). No new gap semantics are introduced this phase.

### Claude's Discretion
- The exact mapping from the `getBrandOverviewData` overview record fields → the resolver's `BrandFoundations.colorConfig` field (the planner/researcher confirms the precise `colorConfig` source against `precompute.ts`'s `PrecomputeInput` shape).
- Whether the injected `foundationsLoader` signature returns raw overview data (resolver maps) or pre-mapped `BrandFoundations` (loader maps) — pick whichever keeps the agents package contract cleanest and the route thin.
- Whether `theme` (light/dark/dim) needs threading into resolution or stays presentation-only (verify against how the engine consumes theme).
- Test structure for the new end-to-end route + resolve-step coverage.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- **`getBrandOverviewDataInternal`** (`convex/foundations.ts:986`) — server-side internalQuery returning `{ color, presetSelection, appearanceConfig, ... }`; the loader's single fetch. Already used by `brandCSSExport.ts` / `brandCSSPrecompute.ts` server-side, so the pattern is proven.
- **`applySubBrandAccentsToFoundation`** (`shared/src/utils/subBrandFoundation.ts`) — merges sub-brand's 4 accent fields into overview data's `appearanceConfig.accents`; identical to platform/ExperienceCanvas behavior. D-03 reuse target.
- **`resolveFoundation` / `BrandFoundations` / `ResolveFoundationInput`** (`foundationResolver.ts`) — already-built resolver entry; just needs real `brandFoundations` passed in.
- **`subBrandConfigs.getByParentBrand` / `.getById`** — read-only queries for the dependent Select + sub-brand fetch.
- **`ConvexHttpClient`** — already imported in the run route for `persistRun`; reuse for the loader.

### Established Patterns
- **Injected-function seam** — Phase 2 already uses `__setCallModelImpl` / `__setCompileImpl` DI seams for credential-free tests; the `foundationsLoader` injection (D-05) follows the same idiom.
- **Read-only Convex query in panels** — RequestPanel's `useQuery(api.brands.list)` (D-02) is the template for the sub-brand `useQuery(api.subBrandConfigs.getByParentBrand, { parentBrandId })`.
- **Workflow steps own orchestration (ORCH-04)** — `resolveStep` is already a workflow step; making it functional keeps resolution inside Mastra, not the route.
- **Gap-first** — uncovered profile / unresolvable brand → typed gap, never fabricated (FND-03), already wired in `plan`/`generate-ir` steps.

### Integration Points
- `workflow.ts` `resolveStep` (no-op) → real injected-loader fetch writing `ctx.request.brandFoundations`.
- `RunExperienceInput` → add `foundationsLoader` + optional `subBrandConfigId`.
- Run route → construct Convex-backed loader, parse `subBrandConfigId` from body, inject into `runExperienceWorkflow`.
- RequestPanel + PromptCardShape + useExperienceLabRun → sub-brand Select + prop + POST field.

</code_context>

<specifics>
## Specific Ideas

- The sub-brand merge MUST produce the **same** `appearanceConfig` the live platform produces for that sub-brand — verified by reusing `applySubBrandAccentsToFoundation` rather than re-deriving (drift is the explicit risk being avoided).
- End-to-end proof matters: Phase 2 verification noted FND-04 was proven only at the resolver unit level. This phase should add a **route-through-workflow** test (mock loader returning real-shaped overview data) asserting a real `ThemeConfig` reaches the resolver and a sub-brand yields merged accents.
- Keep the agents package **Convex-free** — the loader injection is the isolation boundary; a hard Convex import in `experience-builder-agents` would be a regression.

</specifics>
