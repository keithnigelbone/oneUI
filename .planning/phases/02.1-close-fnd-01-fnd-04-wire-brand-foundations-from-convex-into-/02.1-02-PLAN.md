<objective>
Wire the Convex boundary and the UI for brand/sub-brand resolution. The run route builds a Convex-backed `foundationsLoader` (mapping a real brand overview into `BrandFoundations`, merging an optional sub-brand) and injects it into `runExperienceWorkflow`. The RequestPanel gains an optional dependent sub-brand Select, threaded through `PromptCardShape` → `useExperienceLabRun` POST body → `RunRequestBody` → the loader.

This is the route/UI half of the vertical slice; it consumes the `FoundationsLoader` type and the populated `resolveStep` from Plan 01 (Wave 1). Together they close FND-01 (brand/sub-brand → real ThemeConfig) and the route-level half of FND-04 (real foundation source connected, no schema migration).

Purpose: The route receives a real `brandId` but never uses it server-side, so every real run resolves against empty foundations. This plan supplies the actual Convex fetch + the sub-brand selection UI. All Convex-shape coupling stays in the route (D-05) — the agents package never sees it.
Output: `makeConvexFoundationsLoader` factory + loader injection in `route.ts`, `subBrandConfigId` in `RunRequestBody`, the optional shape prop, the conditional POST field, the dependent sub-brand Select, and extended `runRoute`/`requestPanel` tests.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → run route (POST body) | Untrusted `subBrandConfigId`/`brandId` strings cross here. Validated by `RunRequestBody.strict()` (zod) before use. |
| run route → Convex | The loader queries brand/sub-brand foundations server-side via `ConvexHttpClient` — same trust boundary as existing `persistRun`. `v.id()` validators on the Convex queries reject malformed ids. |
| run route → agents workflow | The route injects the Convex-backed loader; the agents package stays Convex-free (Plan 01 invariant). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02.1-04 | Tampering / Information Disclosure | `subBrandConfigId` / `brandId` in POST body | mitigate | `RunRequestBody.strict()` adds `subBrandConfigId: z.string().optional()` and rejects unknown fields (Pitfall 4). The loader casts to `Id<'brands'>`/`Id<'subBrandConfigs'>` only at the Convex call; Convex `v.id()` validators on `getBrandOverviewData`/`getById`/`getByParentBrand` reject non-id strings server-side. A miss returns `null` → system defaults (D-08), never a crash or fabricated foundation. |
| T-02.1-05 | Information Disclosure | Cross-brand sub-brand leakage | mitigate | `getByParentBrand(brandId)` scopes the Select to the chosen brand's sub-brands; even if a `subBrandConfigId` from another parent is posted, `applySubBrandAccentsToFoundation` applies ONLY the sub-brand's 4 accent fields onto the selected brand's overview — no parent foundation data crosses. Low risk for a design Lab; accepted residual. |
| T-02.1-06 | Information Disclosure | Credential leakage into preview/agents | accept | The route runs Node-only and carries no auth/Convex token into any preview surface (previews are P3); the loader's Convex client lives only in the route closure and is never passed beyond the workflow as data. Same posture as existing `persistRun`. |
| T-02.1-SC | Tampering | npm/pip/cargo installs | mitigate | No packages added this plan (RESEARCH: zero external packages). No install task; nothing to audit. |
</threat_model>
