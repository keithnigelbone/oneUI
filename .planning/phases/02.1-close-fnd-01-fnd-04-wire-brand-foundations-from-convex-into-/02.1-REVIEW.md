# Phase 02.1: Code Review Report

**Reviewed:** 2026-06-01T00:00:00Z
**Depth:** standard
**Files Reviewed:** 10
**Status:** issues_found

## Summary

This slice wires real Convex brand foundations into the Experience Lab run path: a `FoundationsLoader` contract in the agents package (`workflow.ts`), a Convex-backed loader factory + run persistence in the Node route (`route.ts`), a dependent sub-brand `Select` in `RequestPanel.tsx`, a `subBrandConfigId` prop threaded through the prompt card and the canvas run reducer, and accompanying tests.

The Convex-shape coupling is correctly confined to the route (D-05), the overview→`BrandFoundations` mapping reads `color.config` verbatim (Pitfall 1 — verified against `convex/foundations.ts:953-968` and `subBrandConfigs` schema), and the sub-brand doc shape matches `SubBrandAccentFields`. The happy path and the `null`→system-defaults path are correct and well tested.

The main concern is robustness: the injected `foundationsLoader` is awaited with **no error isolation** in either the route loader or `resolveStep`, so a transient Convex failure during foundation loading converts a generatable run into a hard `error` outcome — directly contradicting the D-08 "fall back to system defaults" guarantee that the same code documents for the `null` case. Several secondary correctness/robustness gaps and a few quality nits follow.

## Critical Issues

### CR-01: Foundation-load failure aborts the entire run instead of degrading to system defaults (D-08 violation)

**File:** `packages/experience-builder-agents/src/workflow.ts:264-274` and `apps/platform/src/app/api/experience-lab/run/route.ts:137-174`

**Issue:** The injected loader is awaited bare in `resolveStep`:

```ts
const loaded = await ctx.request.foundationsLoader({ brandId: ..., ... });
if (loaded) { ctx.request.brandFoundations = loaded; }
```

and the route's loader (`makeConvexFoundationsLoader`) issues `convex.query(...)` with no try/catch. Any thrown error from Convex (network blip, auth/transport error, malformed id, transient backend outage) propagates out of the loader → out of `step.execute` → into `runExperienceWorkflow`'s outer `try/catch` (`workflow.ts:611-633`), which sets `outcome = 'error'` and returns a run with NO artifact.

This contradicts the design contract stated in the same module: a missing/`null` foundation must "fall back to engine system defaults (D-08), never a crash or a fabricated foundation" (`route.ts:124-126`, `workflow.ts:262-263`). A `null` result is gracefully handled; a thrown error is not. The two failure modes (no record vs. transient fetch failure) should both degrade to system defaults — a generatable brand should not fail to generate because Convex briefly hiccuped during foundation load. Note `persistRun` IS wrapped as best-effort (`route.ts:107-111`), but the foundation load — which is on the critical path — is not, which is backwards: persistence failure is tolerated, the (recoverable) load failure is fatal.

**Fix:** Treat a loader throw the same as a `null` result. Either wrap the query in the route loader:

```ts
function makeConvexFoundationsLoader(convex: ConvexHttpClient): FoundationsLoader {
  return async ({ brandId, subBrandConfigId }) => {
    if (brandId === PLACEHOLDER_BRAND_ID) return null;
    try {
      const overview = await convex.query(api.foundations.getBrandOverviewData, {
        brandId: brandId as Id<'brands'>,
      });
      if (!overview) return null;
      // ... existing merge + map ...
    } catch (err) {
      console.error('[experience-lab/run] foundation load failed, using system defaults:', err);
      return null; // → D-08 system defaults, never a hard run failure
    }
  };
}
```

and/or guard the call in `resolveStep` so the agents package never lets a loader throw abort the run:

```ts
if (ctx.request.foundationsLoader && !ctx.request.brandFoundations) {
  try {
    const loaded = await ctx.request.foundationsLoader({ brandId: ctx.request.brandId, ... });
    if (loaded) ctx.request.brandFoundations = loaded;
  } catch {
    // loader failure is not a foundation gap — resolveFoundation falls back to system defaults (D-08)
  }
}
```

Add a test: loader rejects → `outcome === 'artifact'` (system defaults), not `'error'`.

## Warnings

### WR-01: Sub-brand fetched but its accent merge result is silently discarded when the doc shape is partial

**File:** `apps/platform/src/app/api/experience-lab/run/route.ts:147-162`

**Issue:** When `subBrandConfigId` is present, the loader fetches the sub-brand and merges via `applySubBrandAccentsToFoundation(overview, sub as any)`. `applySubBrandAccentsToFoundation` (`packages/shared/src/utils/subBrandFoundation.ts:50-79`) calls `buildSubBrandAccentItems(currentSubBrand)`, which dereferences `subBrand.primary.scaleName`, `subBrand.secondary.scaleName`, `subBrand.sparkle.scaleName`, and `subBrand.brandBg.backgroundStep.light`. The `sub as any` cast bypasses all type-checking: if `getById` ever returns a row missing one of those nested fields (e.g. a legacy/partial sub-brand row, or a future schema migration), this throws a `TypeError` inside the loader — which, per CR-01, currently aborts the whole run rather than degrading. Even after CR-01 is fixed (catch → null), a malformed sub-brand would silently fall back to *parent-only* with no signal, which is a surprising outcome for a user who explicitly picked a sub-brand.

**Fix:** Validate the sub-brand shape before merging (the project already has Convex validators; reuse the field shape), or at minimum narrow off `any` and short-circuit to parent-only with a logged warning when required fields are absent:

```ts
if (sub && sub.primary && sub.secondary && sub.sparkle && sub.brandBg) {
  base = (applySubBrandAccentsToFoundation(overview, sub as SubBrandAccentFields) ?? overview) as Record<string, unknown>;
} else if (sub) {
  console.warn('[experience-lab/run] sub-brand row missing accent fields; using parent brand only', subBrandConfigId);
}
```

### WR-02: `RunRequestBody` is `.strict()` but `brandFoundations` / `foundationsLoader` would be rejected — and `subBrandConfigId` is not validated as a real id

**File:** `apps/platform/src/app/api/experience-lab/run/route.ts:177-189`

**Issue:** `subBrandConfigId: z.string().optional()` accepts any non-empty string and is later cast `subBrandConfigId as Id<'subBrandConfigs'>` (`route.ts:149`). A client can post an arbitrary `subBrandConfigId` (the route is reachable from the browser); Convex `getById` with `v.id('subBrandConfigs')` will reject a malformed id by throwing — which again hits the CR-01 abort path. There is no `.min(1)` either, so an empty-string `subBrandConfigId` would pass schema validation (the client guards this, but the route should not rely on the client). The brand id is similarly untrusted: `z.string().min(1)` then `as Id<'brands'>` — a non-placeholder garbage string reaches `getBrandOverviewData` and `createRun` (`v.id('brands')`), throwing on bad ids.

**Fix:** Treat invalid-id throws as non-fatal (covered by CR-01's loader try/catch), and tighten the schema where cheap: `subBrandConfigId: z.string().min(1).optional()`. Consider validating id format defensively rather than relying on Convex to throw on the hot path.

### WR-03: Run is persisted to Convex but the client never learns the persisted run id

**File:** `apps/platform/src/app/api/experience-lab/run/route.ts:232, 257-261` and `apps/platform/src/app/(platform)/(experience-lab)/_canvas/useExperienceLabRun.ts:155-185`

**Issue:** The route exposes the persisted run id via the `X-Experience-Persisted-Run-Id` response header, but `useExperienceLabRun.readNdjson` only consumes `response.body` and never reads response headers. The persisted run id (the durable handle to the run/IR/artifact in Convex) is therefore unreachable from the canvas — there is no way to link the placed artifact card to its persisted `experienceRuns`/`experienceArtifacts` rows. Given VER-03's whole point is durable, reusable runs, dropping the id on the floor undermines the feature (re-open, re-export, version history all need it).

**Fix:** Read `response.headers.get('X-Experience-Persisted-Run-Id')` in `runForPrompt` and either stash it on the artifact card props (`ArtifactCardShape`) or surface it through the returned `RunResultFrame`. At minimum, document why it is intentionally ignored if the lineage is deferred to a later plan.

### WR-04: `persistRun` and `runExperienceWorkflow` are awaited serially before the stream opens — Convex latency blocks first byte

**File:** `apps/platform/src/app/api/experience-lab/run/route.ts:226-249`

**Issue:** The route `await`s `runExperienceWorkflow(...)` AND then `await persistRun(...)` fully before constructing the `ReadableStream` and returning a `Response`. The event stream the client decodes line-by-line is thus not actually streaming — every event is already in `run.events` and is only flushed after persistence completes. A slow `createRun`/`recordRunEvents`/`persistArtifact` round-trip (3 sequential Convex mutations) delays the *entire* response, including the events that exist in memory and could have been sent immediately. The doc comment claims persistence "never blocks ... the event stream" (`route.ts:230-231`), but because it is awaited before the `Response` is returned, it does block time-to-first-byte. (Per the project's stated scope, raw throughput is out of v1 review scope — flagged here as a correctness/contract mismatch between the comment and the code, and a UX-perceptible blocking behavior, not a perf-algorithm issue.)

**Fix:** Either start the stream first and persist after `controller.close()` (best-effort, fire-and-forget with the existing try/catch), or move `persistRun` off the response's critical path. If the persisted-run-id header (WR-03) must be present, accept the trade-off but correct the misleading "never blocks" comment.

### WR-05: Empty-prompt run is allowed through the canvas/route even though the panel disables it

**File:** `apps/platform/src/app/(platform)/(experience-lab)/_canvas/useExperienceLabRun.ts:122-148` and `apps/platform/src/app/api/experience-lab/run/route.ts:184-187`

**Issue:** `RequestPanel` disables the Run CTA when `!brandId || !prompt.trim()` (`RequestPanel.tsx:199`), but `useExperienceLabRun.runForPrompt` performs no such guard — it POSTs whatever the card props hold. The route schema makes `prompt` `.optional()` and never requires `brandId` to be non-placeholder, so a programmatic/keyboard-triggered run on a blank prompt card silently proceeds to a full (wasted) generation. The "run intent" invariant is enforced only at the disabled-button layer, which is bypassable (Enter key, double-fire, future hotkey).

**Fix:** Mirror the panel's guard in `runForPrompt` before fetch (`if (!prompt.props.prompt.trim() || !prompt.props.brandId) { setPromptStatus('error'); return null; }`), so the run-intent check lives with the run, not only the button.

## Info

### IN-01: `ShapeUtil<any>` + `RecordProps<any>` defeat prop type-safety on the new `subBrandConfigId`

**File:** `apps/platform/src/app/(platform)/(experience-lab)/_canvas/shapes/PromptCardShape.tsx:57-69`

**Issue:** The new `subBrandConfigId: T.optional(T.string)` validator is added under `ShapeUtil<any>` / `RecordProps<any>`. The pre-existing `any` (documented as a tldraw generic limitation) means a future mismatch between `PromptCardShapeProps` and the `props` validator (e.g. forgetting to add a runtime validator for a new prop) compiles silently and only fails at runtime inside tldraw's record validation. Not introduced by this diff, but the new optional prop widens the surface.

**Fix:** Where tldraw allows, narrow the `RecordProps` generic to the concrete props type so the validator object is checked against `PromptCardShapeProps` at compile time. Otherwise leave as-is with the existing comment.

### IN-02: `eslint-disable @typescript-eslint/no-explicit-any` repeated four times in the loader mapping

**File:** `apps/platform/src/app/api/experience-lab/run/route.ts:158, 166, 168, 170`

**Issue:** The overview→`BrandFoundations` map uses four separate `as any` casts with inline disables. A single typed accessor (or a `as { color?: { config?: unknown }; presetSelection?: unknown; appearanceConfig?: unknown }` shape) would remove the repetition and document the expected overview contract in one place, reducing the chance a future field rename (the exact Pitfall-1 hazard) goes unnoticed.

**Fix:** Introduce a local `type OverviewShape = { color?: { config?: unknown }; presetSelection?: unknown; appearanceConfig?: unknown }` and cast `base` once.

### IN-03: Dead/misleading header comment in `workflow.ts`

**File:** `packages/experience-builder-agents/src/workflow.ts:122-124`

**Issue:** The banner `// Event sink — collects the emitted ExperienceBuilderEvent stream` sits immediately before the FoundationsLoader contract block, not before any event-sink code (the actual `emit` sink is `workflow.ts:223-225`). It is a stale section header that now mislabels the following 40 lines of loader-contract documentation.

**Fix:** Remove or relocate the stale `Event sink` banner so it precedes the `emit`/`now` helpers it describes.
