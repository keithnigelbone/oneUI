# Convex Schema Rules — Multi-Brand Data Layer

## Core Principle

Recipe selections are stored as **decisions** (intent), not **resolved tokens**
(output). Resolution happens at runtime so that foundation changes propagate
automatically to all components.

---

## Tables

### `componentRecipeSelections`

Stores per-brand, per-component recipe decisions.

```typescript
// Schema
defineTable({
  brandId: v.id("brands"),
  component: v.string(),          // e.g. "Button", "Input", "Card"
  selections: v.object({          // decision key → selected option value
    // Example for Button:
    // boldFillStrategy: "accent",
    // ghostBorder: "visible",
    // cornerRadius: "inherit",
    // textTransform: "none",
    // iconPlacement: "leading",
  }),
  tier3Overrides: v.optional(v.array(v.object({
    property: v.string(),
    value: v.string(),
    variants: v.array(v.string()),
    sizes: v.array(v.string()),
    justification: v.string(),
    reviewDate: v.string(),        // ISO date
    reviewOwner: v.string(),
  }))),
  updatedAt: v.number(),
  updatedBy: v.optional(v.string()),
})
  .index("by_brand", ["brandId"])
  .index("by_brand_component", ["brandId", "component"])
```

### `brandCSSCache`

Server-side CSS cache for pre-computed brand CSS. Auto-invalidated when
foundation data changes.

```typescript
defineTable({
  brandId: v.id("brands"),
  cacheKey: v.string(),          // computeInputHash output
  css: v.string(),               // Pre-computed CSS string
  createdAt: v.number(),
})
  .index("by_brand", ["brandId"])
  .index("by_brand_key", ["brandId", "cacheKey"])
```

**Rules:**
- Cache is invalidated automatically when any CSS-relevant foundation changes.
- The `cacheKey` is generated by `computeInputHash` (from `@oneui/shared/engine`).
- Non-CSS-relevant changes (motion, spacing, shape) do NOT invalidate the cache.

### What goes WHERE

| Data                        | Table                          | Why                                              |
|-----------------------------|--------------------------------|--------------------------------------------------|
| Color palettes              | `brands` (existing)            | Tier 1 foundation                                |
| Typography config           | `brands` (existing)            | Tier 1 foundation                                |
| Appearance/role assignments | `brands` (existing)            | Tier 1 foundation                                |
| Recipe decisions            | `componentRecipeSelections`    | Tier 2, stores intent not output                 |
| Tier 3 raw overrides        | `componentRecipeSelections`    | Attached to the component they override          |
| Resolved CSS                | **Nowhere** — computed at runtime | Derived from foundations + recipe at render time |

## Queries

### `getBrandOverviewData`

The single consolidated query for all foundation data. Used by both platform
and Storybook.

```typescript
// Returns
{
  colorScales: ColorScale[],
  surfacesConfig: SurfacesConfig,
  typography: TypographyConfig,
  presetSelection: PresetSelection,
  appearanceConfig: AppearanceConfig | null,
  availablePlatforms: Platform[],
}
```

**Rules:**
- Do NOT create per-component queries (no `getButtonConfig`, `getCardConfig`).
- Do NOT split this into multiple queries. One subscription, many readers.
- `PlatformLayoutContent` reads from `FoundationDataContext`, never subscribes directly.

### `getComponentRecipeSelections`

Fetches recipe selections for a brand.

```typescript
// Query
getComponentRecipeSelections({
  brandId: Id<"brands">,
  component?: string,  // optional filter
})

// Returns
ComponentRecipeSelection[]
```

**Rules:**
- Returns decision key-value pairs, not resolved tokens.
- The component recipe editor calls this to restore radio group state on reload.
- If no selections exist for a component, use recipe defaults.

### `setComponentRecipeSelections`

Saves recipe selections for a brand + component.

```typescript
// Mutation
setComponentRecipeSelections({
  brandId: Id<"brands">,
  component: string,
  selections: Record<string, string>,  // decision key → option value
  tier3Overrides?: Tier3Override[],
})
```

**Rules:**
- Validates that all decision keys exist in the recipe schema.
- Validates that all option values are valid for their decision.
- Rejects unknown keys (no arbitrary data).
- Sets `updatedAt` automatically.

## Recipe Resolution (Runtime)

Resolution happens in the component recipe editor preview and in `useBrandCSS`.
It is a pure function:

```typescript
function resolveRecipe(
  recipe: ComponentRecipe,
  selections: Record<string, string>,
  tier3Overrides: Tier3Override[]
): Record<string, Record<string, string>> {
  // For each variant in recipe.resolution:
  //   For each token property:
  //     If property has a condition:
  //       Look up selections[condition.decision]
  //       Return condition.mapping[selectedOption]
  //     Else:
  //       Return the static token value
  //   Apply tier3Overrides last (overwrite matching properties)
  // Return: { variant: { property: resolvedTokenValue } }
}
```

**Rules:**
- This function is pure. No side effects, no DOM, no Convex calls.
- Tier 3 overrides apply AFTER recipe resolution (they override recipe output).
- If a selection is missing for a decision, use the recipe's `defaultOption`.
- The output is CSS custom property values (e.g., `var(--Surface-Bold)`),
  never raw values.

## Migration from Granular Editor

### Phase 1: Audit existing `componentTokenOverrides`

For each brand × component, classify overrides:
- **Recipe-expressible:** Maps to a recipe decision option.
- **Foundation-derived:** Should come from Tier 1, not stored per-component.
- **True override:** Needs Tier 3 escape hatch.

### Phase 2: Backfill `componentRecipeSelections`

For each brand, derive the closest recipe selections from existing overrides.
Write a migration script that:
1. Reads all `componentTokenOverrides` for a brand.
2. Matches them against recipe resolution outputs for each option combination.
3. Picks the closest match and writes to `componentRecipeSelections`.
4. Remaining unmatched overrides become Tier 3 entries.

### Phase 3: Deprecate `componentTokenOverrides`

Once all brands have recipe selections:
1. Stop writing to `componentTokenOverrides` for new changes.
2. Recipe editor becomes the default UI.
3. Granular editor moves behind a feature flag.
4. After validation period, remove the feature flag and the old table.
