# Publishing & Versioning

OneUI Studio publishes six packages from this monorepo to the **JioDS** Azure DevOps Artifacts feed under the `@jds4/*` scope. The source tree keeps the `@oneui/*` names — the rename happens at staging time, so monorepo links, Storybook, and `apps/platform` all keep working.

| Source workspace | Published as |
|---|---|
| `@oneui/ui` | `@jds4/oneui-react` |
| `@oneui/icons-jio` | `@jds4/oneui-icons-jio` |
| `@oneui/vite-plugin` | `@jds4/oneui-vite-plugin` |
| `@oneui/webpack-plugin` | `@jds4/oneui-webpack-plugin` |
| `@oneui/next-plugin` | `@jds4/oneui-next-plugin` |
| `@oneui/init` | `@jds4/oneui-init` |

Not published (workspace-only): `@oneui/shared`, `@oneui/tokens`, `@oneui/convex`, `@oneui/platform`, `@oneui/storybook`, `@oneui/ui-native*`. `shared` and `tokens` are inlined into `@oneui/ui` at build time — see `packages/ui/vite.config.ts`.

## Where versions live

Versioning is layered. Each file is the source of truth for a different question:

| File | Question it answers |
|---|---|
| `packages/<name>/package.json` `"version"` field | "What version does the next tarball carry?" — `pack-release.ts` reads this directly. |
| `.changeset/config.json` | "Which packages participate in Changesets? Public or restricted access? Which base branch?" |
| `.changeset/*.md` (one per change) | "What's pending? Which packages bump, by how much, and why?" — written by `pnpm changeset`, consumed by `pnpm version-packages`. |
| `packages/<name>/CHANGELOG.md` | "What was in each previous release?" — generated by `pnpm version-packages`. Committed alongside the version bump. |
| `release.yml` workflow run history | "When was each version actually pushed to Azure?" |

There is no separate "version file" you edit by hand. The source of truth is each package's `package.json`. Changesets just orchestrates how that field gets bumped.

## The day-to-day flow

```
┌────────────────┐  pnpm changeset    ┌─────────────────────┐
│ feature work   │ ─────────────────► │ .changeset/foo.md   │
│ on a branch    │  (describes bump)  │ on the same branch  │
└────────────────┘                    └─────────────────────┘
        │                                       │
        │ PR merges                             │
        ▼                                       ▼
┌────────────────────────────────────────────────────────────┐
│ main branch — has accumulated .changeset/*.md files         │
└────────────────────────────────────────────────────────────┘
                              │
                              │ release.yml runs on push
                              ▼
┌────────────────────────────────────────────────────────────┐
│ Changesets bot opens / updates                              │
│ "chore: release packages" PR                                │
│   • bumps versions in each package.json                     │
│   • writes / updates CHANGELOG.md                           │
│   • deletes the consumed .changeset/*.md files              │
└────────────────────────────────────────────────────────────┘
                              │
                              │ maintainer reviews + merges
                              ▼
┌────────────────────────────────────────────────────────────┐
│ release.yml runs again (no pending changesets now)          │
│   → workflow's `publish:` step (inlined bash):              │
│     1. pnpm pack:release  (stage + rename + npm pack ×6)    │
│     2. npm publish each tarball to JioDS                    │
└────────────────────────────────────────────────────────────┘
```

### Writing a changeset (developer side)

After making a change that should ship:

```bash
pnpm changeset
```

The CLI asks:
1. Which packages changed? (multi-select — pick the publishable ones the change affects)
2. Major / minor / patch for each?
3. Summary line (becomes the CHANGELOG entry)

The result is a markdown file like `.changeset/silly-name-rfx.md`:

```markdown
---
'@oneui/ui': minor
'@oneui/vite-plugin': patch
---

Added `intent` prop to Button; vite-plugin gained an offline mode.
```

Commit that file alongside your code change. Don't bump versions manually.

### Releasing (maintainer side)

Nothing — just merge the Version Packages PR that the bot opened. The next push to main triggers the publish step automatically.

## First publish (one-time, going from 0.1.0-alpha.0 → 1.0.0)

All six publishable packages currently sit at `0.1.0-alpha.0`. To make `1.0.0` the first published version:

```bash
# 1. On a branch off main
git checkout -b chore/first-release

# 2. Write a single changeset bumping every publishable package to 1.0.0.
#    Pick "major" for each. Summary: "Initial public release on JioDS."
pnpm changeset
git add .changeset
git commit -m "chore: bump to 1.0.0 for first JioDS publish"
git push -u origin chore/first-release
```

Open a PR, get it merged. The Changesets bot then opens the **Version Packages** PR. Merge that PR — it'll bump all six `package.json` files to `1.0.0`, write the initial `CHANGELOG.md`, and delete the changeset markdown.

Next push to main → release.yml publishes 1.0.0 to JioDS.

## GitHub Actions secrets

The workflow needs one secret. Add it under **Settings → Secrets and variables → Actions → New repository secret**:

| Secret | Value |
|---|---|
| `AZURE_DEVOPS_PAT` | A Personal Access Token created at `https://dev.azure.com/JioSOI/_usersSettings/tokens` with `Packaging: Read, write, & manage` scope. **Raw token** — the workflow base64-encodes it at runtime. |

PAT lifecycle: Azure DevOps PATs have a max 1-year lifetime. Diary a calendar reminder to rotate it.

The workflow itself uses the auto-provided `GITHUB_TOKEN` for opening the Version PR — no extra setup needed.

## Manual / emergency publish

If you need to publish without going through the Changesets PR (e.g. for a hotfix, or for the very first 1.0.0 before the bot has run), use the manual trigger:

1. Go to **Actions → Release → Run workflow**.
2. Optionally check **Pack only — do not publish to Azure** to do a dry run first.
3. Click **Run workflow**.

This skips the bot entirely, packs whatever versions the package.json files currently carry, and publishes them.

Or do it locally — there's no wrapper script; just the two commands:

```bash
# Set up ~/.npmrc once (see "Local auth setup" below).
pnpm pack:release

# Then publish each tarball. (Loop, or run npm publish per file manually.)
for tgz in release/tarballs/*.tgz; do
  npm publish "$tgz" \
    --registry=https://pkgs.dev.azure.com/JioSOI/LayersAI/_packaging/JioDS/npm/registry/ \
    --access=restricted
done
```

## What the publish actually does

The publish step is two operations, run back to back. Both the workflow and any local one-off do exactly this:

**Step 1 — `pnpm pack:release`** (script: `cdn-release-full-pipeline/build/pack-release.ts`):

> **Warning:** `pack:release` alone does **not** run `bake-jio-fallback`. Use `pnpm pack:all` before publishing so `packages/ui/cdn-bootstrap/jio.ts` exists and `dist/brand-loader.*` / `dist/cdn-bootstrap/*` are emitted. `pack:release` warns when the bake artifact is missing.

- Builds each publishable package (`pnpm --filter @oneui/<pkg> build`).
- Stages each into `release/staging/<slug>/`:
  - Copies `dist/` and any extra files (e.g. `packages/ui/cdn-bootstrap/`).
  - Rewrites `package.json`: maps `@oneui/X` → `@jds4/oneui-X` in `name` and `dependencies` / `peerDependencies`; strips `bundleWorkspaceDeps`; resolves `workspace:*` to concrete semver; drops `private`, `scripts`, `devDependencies`, `publishConfig`.
  - Rewrites `from '@oneui/X'` imports inside the built `.mjs/.cjs/.d.ts` files.
- Runs `npm pack` per package and copies the tarball into `release/tarballs/`.

**Step 2 — `npm publish` per tarball:**

```bash
for tgz in release/tarballs/*.tgz; do
  npm publish "$tgz" --registry=<JioDS URL> --access=restricted --tag=latest
done
```

That's the whole publish flow. The workflow inlines both steps (with auth + a `dry-run` switch); locally you just run them yourself.

Both the rename map (`NAME_MAP`) and the inlined-deps list (`bundleWorkspaceDeps`) live in `cdn-release-full-pipeline/build/publishConfig.ts`. To add a new publishable package, add an entry there.

## Coexistence with the CDN

The npm packages and the CDN are independent. The CDN serves brand CSS + JSON sidecars; the npm packages serve React code + the plugin that fetches CDN data. Bumping `@jds4/oneui-react` to 1.1.0 does not require re-baking the CDN, and updating the Jio brand CSS on the CDN does not require an npm release.

The one connection: `packages/ui/cdn-bootstrap/jio.ts` (baked into `@jds4/oneui-react`) is a *snapshot* of the live Jio brand CSS at the time `pnpm cdn:bake-jio` was last run. It's the fallback used when no plugin / no CDN is configured. Re-baking is a separate decision from cutting an npm release.
